## 1. 委托的概念：为什么要学委托？

### 1.1 没有委托时的问题

想象一个场景：角色血量变化时，需要通知**UI**更新血条、**音效系统**播放警告音、**AI系统**重新评估威胁。如果没有委托，你可能会这样写：

```cpp
// ❌ 紧耦合方案 —— 每个系统都要直接引用
void AMyCharacter::TakeDamage(float Damage)
{
    Health -= Damage;

    // 直接调用UI（紧耦合！改UI类名这里就要改）
    UIManager->UpdateHealthBar(Health, MaxHealth);

    // 直接调用音效系统（紧耦合！）
    AudioSystem->PlayLowHealthSound();

    // 直接调用AI系统（紧耦合！）
    AISystem->ReportDamageToTeam(this, Damage);

    // 问题：每加一个需要知道血量变化的系统，都要回来改这段代码
    // 这叫"紧耦合"——各个系统互相依赖，牵一发而动全身
}
```

### 1.2 用委托的解耦方案

```cpp
// ✅ 委托方案 —— 发送者不知道接收者是谁
void AMyCharacter::TakeDamage(float Damage)
{
    Health -= Damage;

    // 只做一件事：广播"血量变了"
    OnHealthChanged.Broadcast(Health, MaxHealth, Damage);
    //                ───┬───
    //    广播消息："嘿！血量变了！新血量=XX, 最大血量=XX, 变化值=XX"
    //    谁在听？我不知道，也不关心。你们自己处理。
}

// 各个系统各自注册监听（在自己的初始化代码中）
// UI系统：角色->OnHealthChanged.AddDynamic(this, &UMyUI::UpdateHealthBar);
// 音效系统：角色->OnHealthChanged.AddDynamic(this, &UMyAudio::OnHealthChanged);
// AI系统：角色->OnHealthChanged.AddDynamic(this, &UMyAI::OnTeammateDamaged);
```

委托的核心价值是**解耦**：发送者不需要知道接收者是谁，接收者可以随时注册/取消注册。

## 2. 委托类型的全家福

UE5提供了多种委托类型，根据使用场景选择：

| 委托类型         | 宏                                       | 蓝图可用 | 支持多个绑定   | 典型用途               |
| ---------------- | ---------------------------------------- | -------- | -------------- | ---------------------- |
| 单播委托         | `DECLARE_DELEGATE`                       | ❌       | ❌（一个绑定） | C++内部回调            |
| 动态单播委托     | `DECLARE_DYNAMIC_DELEGATE`               | ✅       | ❌（一个绑定） | 蓝图单个回调           |
| 多播委托         | `DECLARE_MULTICAST_DELEGATE`             | ❌       | ✅             | C++内部广播            |
| **动态多播委托** | **`DECLARE_DYNAMIC_MULTICAST_DELEGATE`** | ✅       | ✅             | **蓝图广播（最常用）** |
| 事件             | `DECLARE_EVENT`                          | ❌       | ✅             | 仅特定类可广播         |

**本章重点讲动态多播委托**，因为它是蓝图和C++交互中最常用的类型。

## 3. DECLARE_DYNAMIC_MULTICAST_DELEGATE 宏详解

### 3.1 宏的命名规则

```
DECLARE_DYNAMIC_MULTICAST_DELEGATE[_RetVal][_Param1][_Param2]...[Param8]
───────────────────────┬────────────────── ──┬─── ──┬── ──┬──
      基础前缀              返回值(可选)    参数1   参数2  ...最多8个参数
```

核心规则：**有几个参数就用几个Param后缀**。

### 3.2 无参数的委托

```cpp
// ===== 声明委托类型 =====

// DECLARE_DYNAMIC_MULTICAST_DELEGATE(委托类型名);
// ─────────────────────┬──────────────────── ──┬──
//                   宏名称                   你的委托类型名

// 声明一个无参数、无返回值的动态多播委托
DECLARE_DYNAMIC_MULTICAST_DELEGATE(FOnPlayerDied);
//                                  ──────┬──────
//        委托类型名称，命名规范：F + On + 事件名（首字母大写）
//        这个委托可以：被多个对象绑定、在蓝图中绑定和广播

// ===== 在类中使用 =====
UCLASS()
class MYPROJECT_API AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    // 声明委托变量
    UPROPERTY(BlueprintAssignable, Category = "Events")
    //        ────────┬───────
    //  蓝图可绑定是什么意思？在蓝图中你可以把事件连接到这个委托上
    FOnPlayerDied OnPlayerDied;
    //           ──────┬─────
    //       委托变量名，命名规范：去掉前缀"F"
};
```

### 3.3 带一个参数的委托

```cpp
// 带一个float参数的委托声明
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnHealthChanged, float, NewHealth);
//                                         ────────┬───────  ──┬──  ────┬────
//                                          委托类型名      参数类型  参数名
//  OneParam = 这个委托携带1个参数
//  参数类型 float，参数名 NewHealth（参数名在蓝图中会显示为引脚名）

// 使用方法：
// OnHealthChanged.Broadcast(100.0f);  // 广播：新血量为100
```

### 3.4 带两个参数的委托

```cpp
DECLARE_DYNAMIC_MULTICAST_DELEGATE_TwoParams(FOnDamageReceived, float, Damage, AActor*, Instigator);
//                                          ─────────┬────────  ──┬── ───┬─── ────┬─── ─────┬────
//                                           委托类型名         参数1类型 参数1名  参数2类型  参数2名
//  TwoParams = 这个委托携带2个参数

// 使用方法：
// OnDamageReceived.Broadcast(25.0f, AttackerActor);
```

### 3.5 带三个参数的委托

```cpp
DECLARE_DYNAMIC_MULTICAST_DELEGATE_ThreeParams(FOnItemCollected, FName, ItemName, int32, Count, bool, bIsNewItem);
//                                             ────────┬───────
//                                               委托类型名
//                              后面依次跟着：参数1类型, 参数1名, 参数2类型, 参数2名, 参数3类型, 参数3名
```

### 3.6 完整参数数量对照表

| 参数数量 | 宏名称                               | 示例                                                       |
| -------- | ------------------------------------ | ---------------------------------------------------------- |
| 0个      | `DECLARE_DYNAMIC_MULTICAST_DELEGATE` | `FOnSimpleEvent`                                           |
| 1个      | 加 `_OneParam`                       | `FOnValueChanged, float, NewValue`                         |
| 2个      | 加 `_TwoParams`                      | `FOnDamage, float, Damage, AActor*, Instigator`            |
| 3个      | 加 `_ThreeParams`                    | `FOnItem, FName, ID, int32, Count, bool, bNew`             |
| 4个      | 加 `_FourParams`                     | `FOnHit, AActor*, A, AActor*, B, FVector, Loc, float, Dmg` |
| 5个      | 加 `_FiveParams`                     | ...                                                        |
| 6个      | 加 `_SixParams`                      | ...                                                        |
| 7个      | 加 `_SevenParams`                    | ...                                                        |
| 8个      | 加 `_EightParams`                    | ...                                                        |

> **⚠️ 如果超过8个参数**：把多个参数封装成一个 `USTRUCT`，委托只传这个结构体。

## 4. DECLARE_DYNAMIC_DELEGATE 对比

### 4.1 单播 vs 多播

```cpp
// ===== 动态单播委托（只有一个绑定者）=====
DECLARE_DYNAMIC_DELEGATE(FOnSingleCallback);
//       ──────┬──────
//  注意：没有MULTICAST！
//  只能绑定一个函数，新绑定会覆盖旧绑定

DECLARE_DYNAMIC_DELEGATE_OneParam(FOnSingleCallbackParam, float, Value);

UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()

public:
    // 单播委托的UPROPERTY说明符不同
    // 单播：BlueprintAssignable（没有BlueprintCallable，因为只能绑定一个）
    UPROPERTY(BlueprintAssignable)
    FOnSingleCallback OnTaskCompleted;

    void SomeFunction()
    {
        // 单播委托的执行方法不同：
        // 多播用 Broadcast()
        // 单播用 Execute() —— 因为只有一个接收者
        if (OnTaskCompleted.IsBound())  // 先检查是否已绑定
        {
            OnTaskCompleted.Execute();  // 执行（单播用Execute，不是Broadcast）
        }
    }
};
```

### 4.2 使用场景选择

```cpp
// 用单播委托：明确的"一对一"关系
// 例如：异步加载完成后的回调
DECLARE_DYNAMIC_DELEGATE_OneParam(FOnAssetLoaded, UObject*, LoadedAsset);

// 用多播委托：明确的"一对多"关系
// 例如：血量变化通知多个系统
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnHealthChanged, float, NewHealth);
```

## 5. DECLARE_EVENT 对比

### 5.1 事件 vs 多播委托

`DECLARE_EVENT` 和 `DECLARE_MULTICAST_DELEGATE` 功能相似，但有一个关键区别：**只有声明它的类可以调用 Broadcast()**。

```cpp
// ===== 事件：只有友元类可以广播 =====
class AMyActor : public AActor
{
public:
    // 声明事件：第一个参数是"拥有者"类型，只有它和它的友元可以Broadcast
    DECLARE_EVENT(AMyActor, FOnActorReady);
    //            ────┬────
    //   只有AMyActor（及其友元）可以调用 Broadcast()
    //   外部类不能随便广播这个事件

    FOnActorReady OnActorReady;  // 事件变量

    void Initialize()
    {
        // ✅ AMyActor内部可以Broadcast
        OnActorReady.Broadcast();
    }
};

// 外部类试图广播：
void SomeOtherClass::TryBroadcast(AMyActor* Actor)
{
    // ❌ 编译错误！外部类不能Broadcast Event
    // Actor->OnActorReady.Broadcast();

    // 外部类只能绑定：
    Actor->OnActorReady.AddUObject(this, &USomeOtherClass::OnReady);
}
```

### 5.2 三种对比

| 特性              | 动态多播委托       | 事件                 |
| ----------------- | ------------------ | -------------------- |
| 蓝图可用          | ✅                 | ❌（仅C++）          |
| 多个绑定          | ✅                 | ✅                   |
| 外部可以Broadcast | ✅                 | ❌（只有声明类可以） |
| 性能              | 较低（有蓝图开销） | 较高（纯C++）        |
| UPROPERTY需要     | ✅                 | ❌                   |
| 典型用途          | 蓝图C++混合通信    | C++内部通信          |

> **简单记忆**：需要蓝图参与 → 动态多播委托。纯C++内部 → 事件。

## 6. 委托的完整生命周期：声明 → 绑定 → 广播 → 解绑

### 6.1 第一步：声明委托类型和变量

```cpp
// ===== 在类声明之前（全局作用域）声明委托类型 =====

// 声明一个带两个参数的动态多播委托类型
DECLARE_DYNAMIC_MULTICAST_DELEGATE_TwoParams(
    FOnHealthChanged,       // 委托类型名（命名规范：F + On + 事件描述）
    float, NewHealth,       // 参数1：类型 float，参数名 NewHealth
    float, OldHealth        // 参数2：类型 float，参数名 OldHealth
);

// ===== 在类中使用 =====
UCLASS(Blueprintable, BlueprintType)
class MYPROJECT_API AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    // 声明委托变量——必须在UPROPERTY中标记 BlueprintAssignable
    UPROPERTY(BlueprintAssignable, Category = "Events",
              meta = (DisplayName = "当血量变化"))
    //        ────────────────
    // BlueprintAssignable：允许蓝图绑定到这个委托
    // 不加这个，委托可以在C++中用，但蓝图看不到
    FOnHealthChanged OnHealthChanged;  // 委托变量（去掉类型名的前缀F）
    //              ────────┬───────
    //            变量名：去掉开头F的委托类型名
};
```

> **⚠️ 重要**：委托变量**必须**放在 `UPROPERTY` 宏中，并且标记 `BlueprintAssignable`，否则蓝图无法绑定它。

### 6.2 第二步：广播委托

```cpp
void AMyCharacter::SetHealth(float NewValue)
{
    // 记录旧血量（给委托的监听者提供上下文）
    float OldHealth = Health;

    // 更新血量
    Health = FMath::Clamp(NewValue, 0.0f, MaxHealth);
    //       ──┬──
    // FMath::Clamp(value, min, max)：把value限制在[min, max]范围内
    // 确保血量不会小于0或大于最大值

    // ===== 广播委托 =====
    OnHealthChanged.Broadcast(Health, OldHealth);
    //           ───┬───
    // Broadcast：把消息发送给所有绑定的监听者
    // 参数顺序必须和声明时的顺序一致
    // 这里：第一个参数是新血量，第二个参数是旧血量

    // 日志输出（调试用）
    UE_LOG(LogTemp, Log, TEXT("血量变化: %.0f -> %.0f"), OldHealth, Health);

    // 血量归零时广播死亡事件
    if (Health <= 0.0f)
    {
        OnPlayerDied.Broadcast();
    }
}
```

### 6.3 第三步：绑定接收（C++端）

在C++中绑定委托有多种方式，根据绑定对象类型选择：

```cpp
// ===== 场景A：绑定到UObject的成员函数（最常用）=====
// 使用 AddDynamic

// 在某个系统的初始化代码中：
void UMyHealthUI::Initialize(AMyCharacter* InCharacter)
{
    if (!InCharacter)
    {
        return;
    }

    // 绑定到委托：当血量变化时，自动调用我的 OnHealthChangedHandler
    InCharacter->OnHealthChanged.AddDynamic(this, &UMyHealthUI::OnHealthChangedHandler);
    //                              ───┬───  ──┬─  ──────────┬──────────
    //                        AddDynamic   this     要绑定的函数指针
    //                        (UObject*,   成员函数指针)
    //
    // AddDynamic 要求：
    // 1. this必须是UObject或其子类
    // 2. 被绑定的函数必须是UFUNCTION()
    // 3. 函数签名必须和委托声明匹配

    // 也可以绑定到死亡委托
    InCharacter->OnPlayerDied.AddDynamic(this, &UMyHealthUI::OnPlayerDiedHandler);
}

// 被绑定的函数必须是 UFUNCTION
UFUNCTION()
void UMyHealthUI::OnHealthChangedHandler(float NewHealth, float OldHealth)
{
    // 更新UI血条
    float HealthPercent = NewHealth / MaxHealth;
    UpdateHealthBarDisplay(HealthPercent);

    // 如果血量很低，播放红色闪烁效果
    if (HealthPercent < 0.3f)
    {
        PlayLowHealthWarning();
    }
}

UFUNCTION()
void UMyHealthUI::OnPlayerDiedHandler()
{
    // 显示"你死了"UI
    ShowDeathScreen();
}

// ===== 场景B：绑定到普通C++对象（非UObject）=====
// 使用 AddRaw 或 AddSP（智能指针）

class FMyPlainClass  // 不继承UObject的普通C++类
{
public:
    void OnHealthChanged(float NewHealth, float OldHealth)
    {
        // 处理逻辑...
    }
};

// 在某个地方绑定：
FMyPlainClass* PlainObj = new FMyPlainClass();
// 注意：非UObject对象不能用动态多播委托！需要用普通多播委托：
// DECLARE_MULTICAST_DELEGATE_TwoParams(FOnHealthChanged_NonDynamic, float, float);
// OnHealthChanged_NonDynamic.AddRaw(PlainObj, &FMyPlainClass::OnHealthChanged);

// ===== 场景C：绑定Lambda表达式 =====
// 使用 AddLambda

Character->OnHealthChanged.AddLambda([this](float NewHealth, float OldHealth)
//                                   ──┬──
//                          捕获this指针（注意生命周期）
{
    UE_LOG(LogTemp, Log, TEXT("Lambda收到：血量从%f变到%f"), OldHealth, NewHealth);
    // Lambda中可以做一些简单的处理
});

// ===== 场景D：绑定到UObject的Lambda =====
// 使用 AddWeakLambda（更安全，自动处理对象销毁）
Character->OnHealthChanged.AddWeakLambda(this, [](float NewHealth, float OldHealth)
{
    // 当this被销毁时，这个Lambda会自动被移除
    // 比AddLambda更安全
});

// ===== 场景E：绑定UFUNCTION但不用AddDynamic =====
// 使用 AddUObject（功能同AddDynamic，旧版本API）
Character->OnHealthChanged.AddUObject(this, &UMyHealthUI::OnHealthChangedHandler);
```

### 6.4 第四步：解绑（RemoveDynamic）

```cpp
void UMyHealthUI::StopListening(AMyCharacter* InCharacter)
{
    if (!InCharacter)
    {
        return;
    }

    // 从委托中移除绑定——以后血量变化不会再通知我
    InCharacter->OnHealthChanged.RemoveDynamic(this, &UMyHealthUI::OnHealthChangedHandler);
    //                              ──────┬─────
    //                           RemoveDynamic 是 AddDynamic 的逆操作
    //                           参数完全一样：this和函数指针

    InCharacter->OnPlayerDied.RemoveDynamic(this, &UMyHealthUI::OnPlayerDiedHandler);
}

// 通常在以下时机解绑：
// 1. UI被销毁时（UObject析构时会自动解绑，但手动解绑是好习惯）
// 2. 角色切换时（玩家换了角色，不再监听旧角色的血量）
// 3. 模块卸载时

// ⚠️ 重要：绑定的对象被销毁后，委托中会留下悬空指针，导致崩溃
// 解决方法：
// 1. 在EndPlay或BeginDestroy中主动RemoveDynamic
void UMyHealthUI::BeginDestroy()
{
    // UE自动生成的绑定在UObject销毁时会自动清理
    // 但手动清理更安全
    if (TrackedCharacter)
    {
        TrackedCharacter->OnHealthChanged.RemoveDynamic(this, &UMyHealthUI::OnHealthChangedHandler);
    }
    Super::BeginDestroy();
}

// 2. 使用 IsAlreadyBound 检查是否已绑定
void UMyHealthUI::SafeBind(AMyCharacter* InCharacter)
{
    if (!InCharacter)
    {
        return;
    }

    // 先检查是否已经绑定，避免重复绑定
    if (!InCharacter->OnHealthChanged.IsAlreadyBound(this, &UMyHealthUI::OnHealthChangedHandler))
    //                                   ───────┬───────
    //                              检查是否已绑定到指定的函数
    {
        InCharacter->OnHealthChanged.AddDynamic(this, &UMyHealthUI::OnHealthChangedHandler);
    }
}
```

## 7. 蓝图中绑定和使用委托

### 7.1 蓝图端的绑定操作

**在蓝图中绑定C++声明的委托：**

1. 打开蓝图 → 事件图表
2. 右键搜索你要绑定的委托变量名（如 `OnHealthChanged`）
3. 你会发现两种方式：

**方式A：使用 "Bind Event to OnHealthChanged" 节点**

```
[执行输入] → Bind Event to OnHealthChanged
              ├─ Target [self]
              └─ Event [Delegate引脚，连线到自定义事件]
                        ↓
                   [自定义事件]
                   ├─ NewHealth
                   └─ OldHealth
                        ↓
                   [写你的蓝图逻辑]
```

**方式B：在细节面板中直接绑定**

- 选中Actor → 细节面板 → Events分类
- 直接在 `OnHealthChanged` 旁边点击"+"创建绑定

**方式C：蓝图中的事件分发器（Event Dispatcher）**

- 蓝图中也可以创建自己的委托（叫Event Dispatcher）
- 在蓝图"My Blueprint"面板 → Event Dispatchers → 添加新的Event Dispatcher
- 用法和C++委托类似：Call（广播）、Bind（绑定）、Unbind（解绑）

### 7.2 蓝图中广播委托

如果你有一个声明了 `BlueprintCallable` 广播方法的委托：

```cpp
// 在C++中可以添加一个专门用于蓝图广播的包装函数
UFUNCTION(BlueprintCallable, Category = "Events")
void BroadcastHealthChanged()
{
    OnHealthChanged.Broadcast(Health, Health);  // 蓝图可以调这个函数来广播
}
```

或者直接在C++逻辑中广播，蓝图不需要管广播时机。

## 8. 实际案例：血量变化委托系统

### 8.1 完整代码：C++ 角色类

```cpp
// ===== HealthSystemCharacter.h =====
#pragma once

#include "CoreMinimal.h"
#include "GameFramework/Character.h"
#include "HealthSystemCharacter.generated.h"

// ============================================================
// 第一步：声明委托类型（在类声明之前）
// ============================================================

float AHealthSystemCharacter::GetHealthPercent() const
{
    if (MaxHealth <= 0.0f)  // 防止除以零
    {
        return 0.0f;
    }
    return CurrentHealth / MaxHealth;
}

void AHealthSystemCharacter::ModifyHealth(float Amount, AActor* Causer)
{
    // ===== 防止在已死亡状态下继续伤害 =====
    if (bIsDead && Amount < 0.0f)  // Amount为负 = 伤害
    {
        return;  // 已经死了，不处理额外伤害
    }

    // ===== 记录旧血量 =====
    float OldHealth = CurrentHealth;

    // ===== 计算并限制新血量 =====
    CurrentHealth = FMath::Clamp(CurrentHealth + Amount, 0.0f, MaxHealth);
    //                            ───────┬───────
    //              当前血量加上变化量，然后限制在[0, MaxHealth]范围

    // ===== 广播血量变化委托 =====
    // 不管增加还是减少，都通知监听者
    OnHealthChanged.Broadcast(CurrentHealth, OldHealth);
    //           ───┬───
    // 所有绑定了这个委托的系统（UI、音效、AI）都会同时收到通知

    // ===== 根据血量变化的正负，细分广播 =====
    if (Amount < 0.0f)  // 伤害（Amount为负）
    {
        float DamageDealt = -Amount;  // 转换成正的伤害值
        FVector HitLoc = Causer ? Causer->GetActorLocation() : GetActorLocation();

        // 广播受伤害委托
        OnDamaged.Broadcast(DamageDealt, Causer, HitLoc);
    }
    else if (Amount > 0.0f)  // 治愈（Amount为正）
    {
        // 广播被治愈委托
        OnHealed.Broadcast(Amount, Causer);
    }

    // ===== 检查低血量警告 =====
    float HealthPct = GetHealthPercent();
    if (HealthPct > 0.0f && HealthPct <= LowHealthThreshold)
    {
        // 触发NativeEvent：C++默认无行为，蓝图可覆写做警告
        HandleLowHealth(HealthPct);
    }

    // ===== 检查死亡 =====
    if (CurrentHealth <= 0.0f && !bIsDead)
    {
        bIsDead = true;

        // 广播死亡委托
        OnDeath.Broadcast();
        //         ───┬───
        // 通知所有监听者：这个角色死了

        // 执行死亡处理（NativeEvent）
        HandleDeath();
    }
}

void AHealthSystemCharacter::SetMaxHealth(float NewMaxHealth)
{
    float OldMaxHealth = MaxHealth;

    // 限制最大血量至少为1，防止除零和边界问题
    MaxHealth = FMath::Max(NewMaxHealth, 1.0f);

    // 如果当前血量超过新最大值，调整当前血量
    if (CurrentHealth > MaxHealth)
    {
        float OldHealth = CurrentHealth;
        CurrentHealth = MaxHealth;
        OnHealthChanged.Broadcast(CurrentHealth, OldHealth);
    }

    // 广播最大血量变化委托
    OnMaxHealthChanged.Broadcast(MaxHealth, OldMaxHealth);
}
```

```cpp
void UHealthBarWidget::InitializeForCharacter(AHealthSystemCharacter* TargetCharacter)
{

    if (!TargetCharacter)
    {
        UE_LOG(LogTemp, Error, TEXT("HealthBarWidget::InitializeForCharacter - TargetCharacter为空!"));
        return;
    }

    // 保存引用
    TrackedCharacter = TargetCharacter;

    // ===== 绑定所有委托 =====
    // 每个委托都绑定到对应的回调函数

    TargetCharacter->OnHealthChanged.AddDynamic(this, &UHealthBarWidget::OnTargetHealthChanged);
    //                                     ───┬─── ──┬─ ──────────┬──────────────
    //                              AddDynamic   this       回调函数指针
    //              绑定规则：AddDynamic(UObject*, 成员函数指针)
    //              this必须是UObject子类（UUserWidget是UObject子类）

    TargetCharacter->OnMaxHealthChanged.AddDynamic(this, &UHealthBarWidget::OnTargetMaxHealthChanged);

    TargetCharacter->OnDamaged.AddDynamic(this, &UHealthBarWidget::OnTargetDamaged);

    TargetCharacter->OnHealed.AddDynamic(this, &UHealthBarWidget::OnTargetHealed);

    TargetCharacter->OnDeath.AddDynamic(this, &UHealthBarWidget::OnTargetDeath);

    // 初始化UI显示（用当前血量立即更新一次）
    OnTargetHealthChanged(TargetCharacter->GetHealth(), TargetCharacter->GetHealth());

    UE_LOG(LogTemp, Log, TEXT("血条UI已绑定到角色 [%s]"), *TargetCharacter->GetName());
}

void UHealthBarWidget::Uninitialize()
{
    // 检查目标角色是否还存在
    if (AHealthSystemCharacter* Character = TrackedCharacter.Get())
    //                                      ──────┬──────
    //                          TWeakObjectPtr::Get() 返回原始指针
    //                          如果Actor被销毁，返回nullptr
    {
        // ===== 解绑所有委托 =====
        Character->OnHealthChanged.RemoveDynamic(this, &UHealthBarWidget::OnTargetHealthChanged);

        Character->OnMaxHealthChanged.RemoveDynamic(this, &UHealthBarWidget::OnTargetMaxHealthChanged);

        Character->OnDamaged.RemoveDynamic(this, &UHealthBarWidget::OnTargetDamaged);

        Character->OnHealed.RemoveDynamic(this, &UHealthBarWidget::OnTargetHealed);

        Character->OnDeath.RemoveDynamic(this, &UHealthBarWidget::OnTargetDeath);
    }

    // 清除引用
    TrackedCharacter.Reset();
    //               ──┬──
    // Reset()：将弱指针置空，相当于设为nullptr
}
```

## 9. 委托的高级用法

### 9.1 带返回值的委托（罕见但有用）

```cpp
// 动态多播委托不支持返回值（因为多个监听者，返回哪个？）
// 但单播委托可以：

DECLARE_DYNAMIC_DELEGATE_RetVal_OneParam(bool, FOnValidateAction, AActor*, Target);
//           ───┬───                     ──┬─
//       RetVal = 这个委托有返回值        返回bool
// 注意：RetVal的声明位置在 _Param 之前

UCLASS()
class AMyActor : public AActor
{
    GENERATED_BODY()

public:
    UPROPERTY(BlueprintAssignable)
    FOnValidateAction OnValidateAction;

    bool TryAction(AActor* Target)
    {
        if (OnValidateAction.IsBound())
        {
            // Execute返回委托的返回值
            return OnValidateAction.Execute(Target);
        }
        return false;  // 默认：不允许
    }
};
```

### 9.2 委托广播的安全检查

```cpp
void AMyCharacter::SafeBroadcast()
{
    // 方法1：使用 IsBound() 检查（多播委托也适用）
    if (OnHealthChanged.IsBound())  // 是否有任何绑定？
    //              ───┬───
    // IsBound()对多播委托：只要有至少一个绑定就返回true
    {
        OnHealthChanged.Broadcast(Health, 0.0f);
    }

    // 方法2：直接广播（推荐）—— 如果没有绑定，Broadcast()本身就是安全的
    // 即使没有绑定者，Broadcast也不会崩溃，只是什么都不发生
    OnHealthChanged.Broadcast(Health, 0.0f);  // 安全，不需要先检查

    // 方法3：检查具体绑定
    if (OnHealthChanged.IsAlreadyBound(this, &UMyClass::MyFunction))
    {
        UE_LOG(LogTemp, Log, TEXT("已绑定MyFunction"));
    }
}
```

## 10. 易错点与最佳实践

### 10.1 常见编译错误

```cpp
// ❌ 错误1：委托变量不在UPROPERTY中
class AMyActor : public AActor
{
    FOnHealthChanged OnHealthChanged;  // ❌ 编译错误或蓝图不可见
};

// ✅ 正确：必须用UPROPERTY(BlueprintAssignable)包裹
UPROPERTY(BlueprintAssignable)
FOnHealthChanged OnHealthChanged;

// ❌ 错误2：绑定函数不是UFUNCTION
void AMyClass::BindDelegates(AMyActor* Actor)
{
    Actor->OnHealthChanged.AddDynamic(this, &AMyClass::MyHandler);
    //                                        ❌ MyHandler不是UFUNCTION
}

// ✅ 正确：回调函数必须是UFUNCTION
UFUNCTION()
void MyHandler(float NewHealth, float OldHealth);

// ❌ 错误3：委托参数不匹配
DECLARE_DYNAMIC_MULTICAST_DELEGATE_TwoParams(FMyDelegate, float, A, float, B);

// ❌ 绑定到的函数参数不匹配
UFUNCTION()
void MyHandler(int32 A, float B);  // 第一个参数类型不对！int32 vs float

// ❌ 错误4：忘记标记BlueprintAssignable
UPROPERTY()  // ❌ 蓝图不可见
FOnHealthChanged OnHealthChanged;

// ✅ 正确：
UPROPERTY(BlueprintAssignable)  // ✅ 蓝图可绑定
FOnHealthChanged OnHealthChanged;

// ❌ 错误5：在UObject销毁后继续持有绑定
// 导致：当被销毁的UObject的委托被广播时 → 崩溃
// ✅ 解决：在EndPlay/BeginDestroy中RemoveDynamic
```

### 10.2 最佳实践速查

| 实践              | 说明                                                  |
| ----------------- | ----------------------------------------------------- |
| 委托类型命名      | `F` + `On` + 事件描述，如 `FOnHealthChanged`          |
| 委托变量命名      | 去掉类型名的前缀F，如 `OnHealthChanged`               |
| 参数命名规范      | 清晰描述含义，如 `NewHealth` 不是 `A`                 |
| 总是做UPROPERTY   | 即使是纯C++委托，也建议加UPROPERTY（利于GC追踪）      |
| 析构时解绑        | 在EndPlay/NativeDestruct中RemoveDynamic，防止悬空引用 |
| 使用WeakObjectPtr | 引用其他Actor时用弱指针，防止悬空引用导致的崩溃       |
| 不超过8个参数     | 超过就用USTRUCT封装                                   |
