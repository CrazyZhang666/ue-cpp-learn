# 5.1 UE垃圾回收详解

> **目标**：理解UE的垃圾回收（GC）机制：它是怎么工作的、UPROPERTY为什么重要、如何避免野指针崩溃。这是UE开发中容易踩坑的知识点。

---

## 为什么需要垃圾回收？

在开始讲UE的GC之前，先快速回忆一下C++原生内存管理的痛点：

```cpp
// ===== C++原生方式的内存管理（只适合普通C++类型）=====
struct FEnemyData { FString Name; };  // 普通C++结构体，不继承UObject/AActor
FEnemyData* EnemyData = new FEnemyData();  // 在"堆"上创建普通C++对象
// ... 使用EnemyData ...
delete EnemyData;  // 用完必须手动删除，否则内存泄漏
EnemyData = nullptr;  // 避免野指针

// 问题1：忘记delete → 内存泄漏（程序占用内存越来越多）
// 问题2：delete后继续用 → 野指针崩溃（最难排查的Bug之一）
// 问题3：delete两次 → 重复释放崩溃
// 问题4：多个地方持有同一个指针，谁来负责delete？→ 所有权混乱
```

UE的GC解决了这些痛点：**你创建对象，引擎帮你决定何时安全地删除它。**

```
手动管理（C++原始方式）:              自动管理（UE的GC）:
┌──────────────────┐                  ┌──────────────────┐
│ 你创建对象        │                  │ 你创建对象        │
│ 你使用对象        │                  │ 你使用对象        │
│ 你判断"谁在用"    │                  │ GC判断"谁在用"    │
│ 你决定何时删除    │                  │ GC决定何时删除    │
│ 你负责delete     │                  │ GC负责销毁        │
│ 你防止野指针      │                  │ UPROPERTY让GC能追踪引用│
│                  │                  │                  │
│ ❌ 全手动，易出错  │                  │ ✅ 少手动delete，但要正确引用 │
└──────────────────┘                  └──────────────────┘
```

> **关键记忆点**：GC管理的是**UObject及其子类**的对象。普通的C++对象（非UObject）不受GC管理，你需要用智能指针管理它们（下一节会讲）。

## GC的核心算法：标记-清除（Mark and Sweep）

UE的GC使用经典的**标记-清除**算法。理解这个算法是理解整个GC机制的关键。

### 第一步：找到"根集"（Root Set）

根集是GC的**起点**，GC从这些对象出发，开始它的"寻路之旅"。

```
什么在"根集"里？
├── UWorld、GameInstance、已加载关卡等引擎持有的对象
│    └── 这些对象会继续引用关卡中的AActor实例
├── 所有用AddToRoot()手动标记为根的对象
├── 引擎核心对象（UObject的"系统保留"集合）
└── 所有加载中的UPackage等资源入口

注意：`UPROPERTY`指向的对象通常**不是根对象本身**。更准确地说，GC从根对象出发，沿着`UPROPERTY`、引擎内部引用和`AddReferencedObjects`登记的引用继续向下找。只要你的对象能从根对象一路被找到，它就是"可达"的。
```

用图理解：

```
         ┌─────────────────────────────────────────────┐
         │               GC的"根集"                     │
         │                                             │
         │  ┌──────────┐  ┌──────────┐  ┌──────────┐ │
         │  │AActor #1 │  │AActor #2 │  │ UObject  │ │
         │  │ (关卡中)  │  │ (关卡中)  │  │ (Root)   │ │
         │  └─────┬────┘  └─────┬────┘  └─────┬────┘ │
         │        │             │             │        │
         └────────┼─────────────┼─────────────┼────────┘
                  │             │             │
                  ▼             ▼             ▼
          开始标记可达对象...
```

### 第二步：标记（Mark），沿着引用链找到所有"还活着"的对象

GC从根集出发，沿着`UPROPERTY`、引擎内部引用和自定义引用收集接口向下追溯，把能找到的对象都标记为"可达"（还活着的）。

```
标记过程示例：

游戏中的角色：
AActor（在关卡中 → 根集）
  │
  ├── UPROPERTY → USkeletalMeshComponent（可达 ✓）
  │     │
  │     └── UPROPERTY → UMaterialInstance（可达 ✓）
  │
  ├── UPROPERTY → UInventoryComponent（可达 ✓）
  │     │
  │     ├── UPROPERTY → UWeaponItem（可达 ✓）
  │     │
  │     └── 没有UPROPERTY → USomeItem（不可达 ✗）
  │           ↑
  │           └── 虽然是UInventoryComponent的一个成员
  │               但是没有UPROPERTY，GC不知道它在用！
  │               这个对象会被回收！访问它 → 野指针崩溃！
  │
  └── 没有UPROPERTY → UOrphanObject（不可达 ✗）
        ↑
        └── 同样是角色持有，没UPROPERTY就是死！
```

### 第三步：清除（Sweep），删除所有"不可达"的对象

标记完成后，所有没被标记到的UObject都会被销毁。

```
标记完毕后的状态：
┌──────────────────────────────────────┐
│                                      │
│  ● 可达对象（被标记）—— 保留，不碰    │
│  ○ 不可达对象（未标记）—— 销毁！     │
│                                      │
│  ● AActor                           │
│  │ ● USkeletalMeshComponent         │
│  │   ● UMaterialInstance            │
│  │ ● UInventoryComponent            │
│  │   ● UWeaponItem                  │
│  │                                  │
│  ○ UOrphanObject      ← 没有UPROPERTY保护，被清除了！│
│  ○ USomeItem          ← 同上                    │
│                                      │
└──────────────────────────────────────┘
```

### 完整的标记-清除过程（伪代码理解）

```cpp
// 这是UE GC内部工作原理的简化伪代码，帮助你理解它的逻辑
// 你不用写这些代码，这是引擎内部的实现

// 第一步：收集所有根对象
TArray<UObject*> RootObjects;
// 收集引擎持有的入口对象，例如World、GameInstance、已加载Package
RootObjects.Add(GetWorld());
RootObjects.Add(GetGameInstance());
// 收集AddToRoot注册的对象
for (UObject* RootObj : GCRootSet)
{
    RootObjects.Add(RootObj);
}

// 第二步：从根开始标记所有可达对象
TArray<UObject*> MarkedObjects;
TArray<UObject*> ToProcess = RootObjects;  // 待处理队列

while (ToProcess.Num() > 0)
{
    UObject* Current = ToProcess.Pop();  // 取出一个对象

    if (MarkedObjects.Contains(Current))
    {
        continue;  // 已经标记过了，跳过
    }

    MarkedObjects.Add(Current);  // 标记为"可达"

    // 遍历这个对象的所有可被GC识别的UObject引用
    for (FProperty* Property : Current->GetClass()->GetProperties())
    {
        if (Property->IsObjectPointer())  // 如果是UObject指针
        {
            UObject* ReferencedObj = Property->GetValue(Current);
            if (ReferencedObj != nullptr)
            {
                ToProcess.Add(ReferencedObj);  // 加入待处理队列
                // 从它继续往下找
            }
        }
    }
}

// 第三步：清除未标记的对象
for (UObject* Obj : AllUObjects)
{
    if (!MarkedObjects.Contains(Obj))
    {
        Obj->BeginDestroy();  // 销毁不可达对象
    }
}
```

> **核心思想**：GC从"根集"出发，沿着UE能识别的引用链一路标记过去。标记到的留下，没标记到的销毁；`UPROPERTY`是常见、关键的引用入口。

## UPROPERTY如何保护指针不被回收

这是UE开发中的**核心知识点**。在多处讲过，但这里要理解其原理。

### UPROPERTY = "GC保护锁"

```cpp
UCLASS()
class MYGAME_API AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    // ===== 场景1：有UPROPERTY保护 =====
    // GC标记时会发现："哦，AMyCharacter持有这个指针
    //                    而且是UPROPERTY的，说明很重要
    //                    标记它指向的对象为'可达'！"
    UPROPERTY()  // ← 这就是GC保护锁
    class UInventoryComponent* InventoryComp;  // ✅ GC绝不会回收它

    // ===== 场景2：没有UPROPERTY = 裸奔 =====
    // GC标记时根本不知道这个指针的存在
    // 它指向的对象如果没有其他UPROPERTY引用，就会被回收
    class UWeaponItem* WeaponItem;  // ❌ 随时可能被GC回收！野指针即将诞生！
};
```

### 详细的保护原理

```cpp
// 让我们用一个完整的例子理解UPROPERTY的保护机制
void Example_UPROPERTYProtection()
{
    // 创建两个UObject
    UWeaponItem* Sword = NewObject<UWeaponItem>();  // 创建一把剑
    UWeaponItem* Shield = NewObject<UWeaponItem>(); // 创建一个盾

    // ===== 正确做法：把指针存入UPROPERTY成员 =====
    AMyCharacter* Player = GetPlayerCharacter();

    Player->InventoryComp->PrimaryWeapon = Sword;  // ✅ PrimaryWeapon有UPROPERTY
    // Sword现在被AMyCharacter的UPROPERTY引用
    // GC标记时会找到它 → 不会被回收

    // ===== 错误做法：用临时变量持有 =====
    Player->TempCacheWeapon = Shield;  // ❌ TempCacheWeapon没有UPROPERTY！
    // 虽然Player持有Shield的指针
    // 但GC不知道这个指针的存在（没有UPROPERTY）
    // 下一次GC时，Shield可能被回收！

    // 过了一会儿...
    GEngine->ForceGarbageCollection();  // 强制执行GC

    // Sword的状态：UPROPERTY保护 → 存活 ✅
    Sword->GetName();  // ✅ 安全！Sword还活着

    // Shield的状态：没有UPROPERTY保护 → 已被回收 ❌
    Shield->GetName();  // ❌ CRASH! 野指针！Shield的内存已被释放
    // 这个崩溃非常难查，因为代码写得看起来没问题
    // 从逻辑上看，Player明明持有Shield，为什么会被GC回收？
    // 答案：GC不知道Player持有Shield，因为没加UPROPERTY！
}
```

### UPROPERTY保护的传递链

```cpp
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    UPROPERTY()  // UPROPERTY保护 → InventoryComp不会回收
    class UInventoryComponent* InventoryComp;
};

UCLASS()
class UInventoryComponent : public UActorComponent
{
    GENERATED_BODY()

public:
    UPROPERTY()  // UPROPERTY保护 → Weapons数组不会回收
    TArray<class UWeaponItem*> Weapons;  // 数组内的每个元素也受保护！
    // 不需要对每个元素单独加UPROPERTY
    // TArray类型的UPROPERTY会自动保护所有元素

    UPROPERTY()  // UPROPERTY保护 → ArmorItem不会回收
    class UArmorItem* ArmorItem;
};

// 保护链：
// AMyCharacter (关卡中的Actor → 在根集中)
//   → UPROPERTY → InventoryComp (被标记)
//     → UPROPERTY → Weapons[0], Weapons[1], ... (被标记)
//     → UPROPERTY → ArmorItem (被标记)
//
// 所有这些对象沿着UPROPERTY链被GC"发现"
// 都不会被回收
```

## GC的触发时机

很多人以为GC是"定时"触发的，实际上UE的GC触发有多种时机：

### 触发方式1：定时触发（最常见）

```cpp
// UE会每隔一定时间自动触发一次GC
// 默认间隔大约在 30-60 秒之间（取决于项目设置）

// 在项目设置中可以调整：
// Project Settings → Engine → Garbage Collection
// ├── Time Between Purging Pending Kill Objects: 60.0s
// │   （多久做一次GC，默认60秒）
// └── 其他高级参数...
```

### 触发方式2：内存压力触发

```cpp
// 当UE检测到内存不足时，会提前触发GC
// 例如：
// - 加载大量资源时
// - 创建大量UObject时
// - 可用内存低于阈值时

// 这个过程是自动的，你通常不需要关心
// 但要知道：在资源密集型操作期间，GC可能在任何时候运行
```

### 触发方式3：关卡切换

```cpp
// 当切换关卡（OpenLevel）时，UE会执行一次完整的GC
// 清除旧关卡中的所有UObject

// 关卡切换时的GC流程：
// 1. 标记新关卡中的对象
// 2. 旧关卡中没有被迁移到新关卡的对象 → 不可达 → 被清除
// 3. 确保新关卡从一个干净的内存状态开始

// ⚠️ 注意：如果你的对象需要在关卡切换时保留
// 需要给它设置一个跨关卡的Outer（如GameInstance）
UGameInstance* GameInstance = GetGameInstance();
// 以GameInstance为Outer创建的对象在关卡切换时不会被清除
UMyPersistentObject* PersistentData = NewObject<UMyPersistentObject>(GameInstance);
```

### 触发方式4：手动强制触发

```cpp
// 在一些特殊情况下，你可能需要手动触发GC
// 例如：卸载了大量资源后，想立即回收内存

// 方法1：通过GEngine强制GC
if (GEngine)
{
    GEngine->ForceGarbageCollection(true);  // true = 完全GC（清空所有）
    // GEngine->ForceGarbageCollection(false); // false = 只清理PendingKill的对象
}

// 方法2：通过UWorld
if (GetWorld())
{
    GetWorld()->ForceGarbageCollection(true);
}

// ⚠️ 注意：不要频繁手动触发GC！会影响性能
// 只在确实需要立即释放大量内存时才用
```

### 各种触发时机总结

| 触发方式   | 何时触发       | 你需要注意什么                   |
| ---------- | -------------- | -------------------------------- |
| 定时触发   | 默认约60秒一次 | 不要在Tick中创建大量临时UObject  |
| 内存压力   | 内存不足时     | 大量资源操作时GC可能随时运行     |
| 关卡切换   | 切换地图时     | 跨关卡数据需要挂在GameInstance下 |
| 手动触发   | 你主动调用时   | 不要频繁调用，影响性能           |
| 编辑器操作 | PIE开始/结束   | 编辑器中GC触发更频繁             |

## IsValid() 的重要性（不仅仅是判空）

很多新手以为 `IsValid()` 就是 `!= nullptr` 的UE版本，但实际上它的功能远不止判空。

### IsValid vs nullptr 的区别

```cpp
// ===== nullptr 检查：只检查指针是不是空 =====
UObject* Ptr = GetSomeObject();

if (Ptr != nullptr)  // 只检查"指针是否为0x00000000"
{
    Ptr->DoSomething();  // 指针不是0，但对象可能已经被GC标记为"待回收"了！
}

// ===== IsValid 检查：检查指针非空 + 对象未被标记为待回收 =====
if (IsValid(Ptr))  // 两步检查：1.指针非空 2.对象还没被标记为垃圾
{
    Ptr->DoSomething();  // ✅ 安全！对象确实还活着
}
```

### 一个致命的场景：PendingKill（待回收）

```cpp
// 这个场景是新手最常见的GC崩溃原因

void AMonster::Die()
{
    // 怪物死了，从世界中移除
    Destroy();  // Actor被标记为"PendingKill"（待回收）
    // 注意：此时对象还没有被真正删除！
    // 只是被标记为"下次GC时清理"
    // 这个状态的指针不是nullptr！但对象已经"半死了"
}

void AMyPlayer::AttackMonster(AMonster* Monster)
{
    Monster->Die();  // 怪物被标记为PendingKill

    // ===== 错误检查 =====
    if (Monster != nullptr)  // ❌ Monster不是nullptr！检查通过！
    {
        Monster->TakeDamage(50);  // ❌ CRASH! 对象已经是PendingKill状态！
        // 内部数据可能已经部分清理了
    }

    // ===== 正确检查 =====
    if (IsValid(Monster))  // ✅ IsValid发现Monster是PendingKill → 返回false
    {
        // 不会执行到这里
    }
    else
    {
        UE_LOG(LogTemp, Warning, TEXT("怪物已经被销毁了"));
    }
}
```

### IsValid的内部逻辑（简化理解）

```cpp
// IsValid的简化实现（不用你写，引擎内部是这样工作的）

template<typename T>
bool IsValid(T* Object)
{
    // 检查1：指针是nullptr吗？
    if (Object == nullptr)
    {
        return false;  // 空指针，无效
    }

    // 检查2：对象被标记为PendingKill了吗？
    // UE4的旧方式：检查内部标记
    if (Object->IsPendingKill())
    {
        return false;  // 对象虽然指针还在，但已经被标记为垃圾了
    }

    // 检查3（UE5新增）：对象是否正常
    // UE5使用更现代的方式检查对象的内部状态
    if (!Object->IsValidLowLevel())
    {
        return false;
    }

    return true;  // 所有检查通过，对象确实有效
}
```

### 何时必须用IsValid

```cpp
// ===== 必须用IsValid的场景 =====

// 场景1：Tick中每帧访问对象
void AMyActor::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    // ✅ 正确：Tick中访问外部对象前必须IsValid
    if (IsValid(TargetEnemy))
    {
        TargetEnemy->GetHealth();  // 安全
    }
    // ❌ 错误：只判空
    // if (TargetEnemy != nullptr) → 可能TargetEnemy刚被Destroy()
}

// 场景2：异步回调中
void AMyActor::OnAsyncLoadComplete(UObject* LoadedAsset)
{
    // ✅ 正确：异步回调中必须IsValid
    if (IsValid(LoadedAsset) && IsValid(this))
    {
        ApplyLoadedAsset(LoadedAsset);
    }
}

// 场景3：定时器回调中
void AMyActor::OnTimerFired()
{
    // ✅ 正确：定时器触发时对象可能已经被销毁了
    if (IsValid(this))
    {
        DoSomething();
    }
}

// 场景4：从容器中获取的对象
void AMyActor::ProcessAllEnemies()
{
    for (AActor* Enemy : EnemyList)
    {
        // ✅ 正确：遍历时每个元素都要检查
        if (IsValid(Enemy))
        {
            ProcessEnemy(Enemy);
        }
    }
}
```

### IsValid不能代替nullptr的场景

```cpp
// ===== 场景：UPROPERTY的TArray，需要判空 =====
UPROPERTY()
TArray<AActor*> MyActors;

void CheckNullExample()
{
    // 获取数组中的元素
    AActor* FirstActor = MyActors[0];

    // IsValid可以，但不适用于下面这种情况：
    if (IsValid(FirstActor))
    {
        // OK
    }

    // 但TArray本身不需要IsValid：
    // TArray不是UObject，不能用IsValid检查
    // 你只需要：
    if (MyActors.Num() > 0)  // 检查数组是否有元素
    {
        // OK
    }
}
```

## TWeakObjectPtr，安全引用UObject的弱指针

`TWeakObjectPtr` 是UE提供的一种特殊指针，用于**安全检查UObject但不阻止其被GC回收**。

### 弱指针 vs 强引用的区别

```cpp
// ===== 强引用（UPROPERTY普通指针）=====
UPROPERTY()
AActor* StrongReference;  // "我持有这个对象，GC别碰它！"
// 后果：只要StrongReference存在，指向的对象就不会被GC回收
// 优点：安全，对象不会被意外回收
// 缺点：可能导致对象一直不释放（内存泄漏）

// ===== 弱引用（TWeakObjectPtr）=====
TWeakObjectPtr<AActor> WeakReference;  // "我关注这个对象，但GC可以回收它"
// 后果：WeakReference指向的对象可以被GC回收
// 优点：不会阻止GC回收对象，用于"观察"而非"持有"
// 缺点：使用前必须检查IsValid，对象可能已经不存在了
```

### TWeakObjectPtr 的使用方式

```cpp
// ===== 创建弱指针 =====
AActor* SomeActor = GetWorld()->SpawnActor<AActor>();

// 方式1：从原始指针构造
TWeakObjectPtr<AActor> WeakActor(SomeActor);

// 方式2：直接赋值
TWeakObjectPtr<AActor> WeakActor2 = SomeActor;

// ===== 使用弱指针 =====
// 使用前必须检查IsValid！（对象可能已经被GC回收了）
if (WeakActor.IsValid())  // 检查对象还活着吗
{
    // 方式1：通过 -> 操作符（像普通指针一样使用）
    WeakActor->SetActorLocation(FVector(0, 0, 100));

    // 方式2：通过 Get() 获取原始指针
    AActor* RawPtr = WeakActor.Get();
    if (RawPtr)
    {
        RawPtr->Destroy();
    }
}

// ===== 重置弱指针 =====
WeakActor.Reset();  // 清空弱指针
// 或者
WeakActor = nullptr;
```

### 什么时候用TWeakObjectPtr

```cpp
// ===== 经典使用场景 =====

// 场景1：观察者模式（监听但不控制）
class UMonsterObserver
{
    // 我的工作是"监控"一个Actor的状态
    // 但我不应该阻止它被销毁
    // 如果Actor被销毁了，我就不再监控它
    TWeakObjectPtr<AActor> ObservedActor;

public:
    void Tick()
    {
        if (ObservedActor.IsValid())  // Actor还活着
        {
            LogActorPosition(ObservedActor.Get());
        }
        else
        {
            // Actor已经被回收了，停止监控
            StopObserving();
        }
    }
};

// 场景2：缓存引用（避免循环引用）
class UMyComponent : public UActorComponent
{
    // 假设组件想引用它的Owner Actor
    // 但组件本身被Owner持有（通过UPROPERTY）
    // 如果组件也用UPROPERTY强引用Owner → 循环引用 → 内存泄漏！

    // 解决方案：用弱指针
    TWeakObjectPtr<AActor> OwnerWeakRef;  // 弱引用 → 不阻止GC回收Owner

    void BeginPlay() override
    {
        OwnerWeakRef = GetOwner();  // 记录Owner
    }

    void TickComponent(float DeltaTime, ELevelTick TickType,
                       FActorComponentTickFunction* ThisTickFunction) override
    {
        if (OwnerWeakRef.IsValid())
        {
            // 从Owner获取一些信息
            FVector OwnerLocation = OwnerWeakRef->GetActorLocation();
        }
    }
};

// 场景3：委托/事件回调中（防止被回调的对象已销毁）
class UMySubsystem
{
    // 子系统中注册了一个Actor的回调
    TWeakObjectPtr<AActor> RegisteredActor;

public:
    void RegisterActor(AActor* Actor)
    {
        RegisteredActor = Actor;
    }

    void BroadcastEvent()
    {
        // ✅ 先检查Actor是否还活着
        if (RegisteredActor.IsValid())
        {
            RegisteredActor->OnSomethingHappened();
        }
        // 如果Actor已经销毁了，弱指针的IsValid返回false
        // 我们安全地跳过回调，不会崩溃！
    }
};
```

### TWeakObjectPtr vs TSharedPtr 的区别

| 特性         | TWeakObjectPtr      | TSharedPtr（智能指针）           |
| ------------ | ------------------- | -------------------------------- |
| 管理对象类型 | 只能UObject及其子类 | 普通C++对象（非UObject）         |
| 阻止GC回收   | ❌ 不阻止           | ✅ 阻止（引用计数 > 0 就不释放） |
| 使用前检查   | 必须IsValid()       | 必须IsValid()                    |
| 所有权       | 无所有权（观察用）  | 共享所有权                       |
| 所属系统     | UE GC系统           | UE智能指针系统                   |

## 常见的GC相关Bug

### Bug 1：忘记UPROPERTY导致野指针崩溃（最常见的GC Bug）

```cpp
// ===== Bug场景 =====
UCLASS()
class AMyCharacter : public ACharacter
{
    GENERATED_BODY()

public:
    // ❌ 忘记UPROPERTY！
    class UWeaponComponent* WeaponComp;
    // 这个组件会在下一次GC时被回收
    // 因为GC不知道AMyCharacter持有它

    void BeginPlay() override
    {
        Super::BeginPlay();

        WeaponComp = NewObject<UWeaponComponent>(this);
        // WeaponComp创建成功，可以正常使用
    }

    void Tick(float DeltaTime) override
    {
        Super::Tick(DeltaTime);

        // 开始几分钟没问题...
        WeaponComp->UpdateWeapon();  // ✅ 正常运行

        // 60秒后，GC触发...
        // WeaponComp被GC回收了！

        // 又过了几秒...
        WeaponComp->UpdateWeapon();  // ❌ CRASH! 野指针！
        // 这个崩溃非常难查，因为：
        // 1. 不是100%复现（取决于GC时机）
        // 2. 崩溃堆栈不指向真正的Bug位置
        // 3. 可能在某些机器上从不崩溃，在另一些机器上频繁崩溃
    }
};

// ===== 修复方法 =====
// 只需要加一个UPROPERTY：
UPROPERTY()  // ← 就这一行就能避免崩溃
class UWeaponComponent* WeaponComp;
```

### Bug 2：TArray遍历时销毁元素

```cpp
// ===== Bug场景 =====
UPROPERTY()
TArray<AEnemy*> EnemyList;

void DestroyAllEnemies()
{
    // ❌ 错误：在遍历过程中修改数组
    for (AEnemy* Enemy : EnemyList)
    {
        Enemy->Destroy();  // 销毁Actor
        // 问题：Destroy()只是标记为PendingKill
        // 但EnemyList中的指针还在！
        // 而且如果EnemyList被其他代码修改，迭代器会失效
    }

    // ❌ 另一种错误写法
    for (int32 i = 0; i < EnemyList.Num(); i++)
    {
        EnemyList[i]->Destroy();
        EnemyList.RemoveAt(i);  // ❌ 删除元素导致索引错乱！
        // RemoveAt后，后面的元素前移，i++会跳过一个元素
    }

    // ===== 正确做法1：反向遍历 =====
    for (int32 i = EnemyList.Num() - 1; i >= 0; i--)
    {
        if (IsValid(EnemyList[i]))
        {
            EnemyList[i]->Destroy();
        }
        EnemyList.RemoveAt(i);  // ✅ 从后往前删，不影响前面的索引
    }

    // ===== 正确做法2：先收集再销毁 =====
    TArray<AEnemy*> EnemiesToDestroy = EnemyList;  // 拷贝一份
    for (AEnemy* Enemy : EnemiesToDestroy)
    {
        if (IsValid(Enemy))
        {
            Enemy->Destroy();  // 销毁（标记PendingKill）
        }
    }
    // Destroy不会立刻从数组中移除引用
    // 需要在下次Tick或适当时机清理数组中的无效指针
    EnemyList.RemoveAll([](AEnemy* Enemy)
    {
        return !IsValid(Enemy);  // 移除所有已被销毁的
    });

    // ===== 正确做法3：使用RemoveAll =====
    EnemyList.RemoveAll([](AEnemy* Enemy)
    {
        if (IsValid(Enemy))
        {
            Enemy->Destroy();
            return true;  // 从数组中移除
        }
        return false;
    });
}
```

### Bug 3：异步回调中访问已销毁的对象

```cpp
// ===== Bug场景：异步加载后访问对象 =====
void AMyActor::LoadWeaponAsync()
{
    // 发起异步加载请求
    FStreamableManager& Streamable = UAssetManager::GetStreamableManager();
    Streamable.RequestAsyncLoad(
        WeaponAssetPath,                    // 要异步加载的资源路径
        FStreamableDelegate::CreateLambda([this]()  // 加载完成后的回调
        {
            // ⚠️ 问题：这个Lambda在异步加载完成后才执行
            // 可能那时候AMyActor已经被销毁了！
            // this 变成了野指针！

            ApplyWeaponToCharacter();  // ❌ CRASH! this已被销毁
        })
    );
}

// ===== 修复方案1：使用弱引用 =====
void AMyActor::LoadWeaponAsyncSafe()
{
    // 创建指向this的弱引用
    TWeakObjectPtr<AMyActor> WeakThis(this);

    FStreamableManager& Streamable = UAssetManager::GetStreamableManager();
    Streamable.RequestAsyncLoad(
        WeaponAssetPath,
        FStreamableDelegate::CreateLambda([WeakThis]()  // 捕获弱引用
        {
            // ✅ 在回调中先检查this是否还活着
            if (WeakThis.IsValid())
            {
                WeakThis->ApplyWeaponToCharacter();  // ✅ 安全
            }
            else
            {
                UE_LOG(LogTemp, Warning, TEXT("Actor已销毁，跳过武器应用"));
            }
        })
    );
}

// ===== 修复方案2：使用Cancel + Delegate Handle =====
void AMyActor::LoadWeaponAsyncWithCancel()
{
    FStreamableManager& Streamable = UAssetManager::GetStreamableManager();
    TSharedPtr<FStreamableHandle> Handle = Streamable.RequestAsyncLoad(
        WeaponAssetPath,
        FStreamableDelegate::CreateUObject(this, &AMyActor::OnWeaponLoaded)
        // CreateUObject会绑定到this的生命周期
        // 但如果this被GC回收了，这个绑定也会失效
    );

    // 在EndPlay中取消未完成的异步请求
}

void AMyActor::EndPlay(const EEndPlayReason::Type EndPlayReason)
{
    Super::EndPlay(EndPlayReason);
    // 这里应该取消所有未完成的异步操作
    // 如果你保持了Handle，可以在这里Cancel
}

// ===== 修复方案3：使用延迟回调（最简单的安全检查）=====
void AMyActor::LoadWeaponAsyncSimple()
{
    FStreamableManager& Streamable = UAssetManager::GetStreamableManager();

    Streamable.RequestAsyncLoad(
        WeaponAssetPath,
        FStreamableDelegate::CreateUObject(this, &AMyActor::OnWeaponLoaded)
        // CreateUObject：当this被GC时，这个回调不会被触发
        // 但如果this只是被Destroy()（PendingKill），回调可能还会触发！
        // 所以仍然需要在回调中做IsValid检查
    );
}

void AMyActor::OnWeaponLoaded()
{
    // ✅ 双重保险：即使CreateUObject没挡住，IsValid也能拦
    if (!IsValid(this))
    {
        return;  // Actor已经被销毁了，别继续
    }

    ApplyWeaponToCharacter();
}
```

## FGCObject，非UObject类需要引用UObject时

### 问题：非UObject类不能使用UPROPERTY

```cpp
// 假设你有一个纯C++类（不从UObject继承）
class FMyDataProcessor  // 普通C++类，没有UObject基类
{
public:
    // ❌ 不能在这里加UPROPERTY！因为它不是UObject的子类
    // UPROPERTY() 只能用在UObject/UCLASS内部

    class UMyDataAsset* CachedData;  // 持有一个UObject指针
    // 问题：GC不知道这个非UObject类持有UObject引用
    // 所以GC可能回收CachedData指向的对象！
    // 崩溃！
};
```

### 解决方案：继承 FGCObject

```cpp
// FGCObject的完整头文件路径：
#include "UObject/GCObject.h"

// 让纯C++类继承FGCObject
class FMyDataProcessor : public FGCObject  // 继承FGCObject
{
public:
    class UMyDataAsset* CachedData;  // 引用的UObject

    // 步骤1：必须重写AddReferencedObjects
    // 这个函数告诉GC："这些UObject我还在用，别回收！"
    virtual void AddReferencedObjects(FReferenceCollector& Collector) override
    {
        // 把CachedData注册到GC的引用收集器中
        Collector.AddReferencedObject(CachedData);
        // 这样GC就知道FMyDataProcessor持有CachedData
        // CachedData就不会被GC回收了

        // 如果有多个UObject引用，全部注册：
        // Collector.AddReferencedObject(AnotherUObject);
        // Collector.AddReferencedObject(YetAnotherUObject);
    }

    // 步骤2：还需要重写GetStatId（UE5的统计系统要求）
    virtual FString GetReferencerName() const override
    {
        return TEXT("FMyDataProcessor");  // 返回一个名字用于调试/统计
    }
};
```

### 完整实例：一个数据管理器

```cpp
// DataManager.h
#pragma once

#include "CoreMinimal.h"
#include "UObject/GCObject.h"  // FGCObject需要这个头文件

// 纯C++类（非UObject），但需要安全引用UObject
class FDataManager : public FGCObject
{
private:
    // 这些是UObject引用，需要通过FGCObject保护
    class UDataTable* ItemDataTable;      // 物品数据表
    class UDataTable* SkillDataTable;     // 技能数据表
    TArray<class UTexture2D*> CachedIcons; // 缓存的图标

    // 普通C++数据（不需要GC保护）
    TMap<FName, int32> ItemCountMap;
    TMap<FName, float> SkillCooldownMap;

public:
    FDataManager()
        : ItemDataTable(nullptr)
        , SkillDataTable(nullptr)
    {
    }

    // 设置数据表引用
    void SetItemDataTable(UDataTable* InDataTable)
    {
        ItemDataTable = InDataTable;
    }

    void SetSkillDataTable(UDataTable* InDataTable)
    {
        SkillDataTable = InDataTable;
    }

    // 添加缓存的图标
    void CacheIcon(UTexture2D* Icon)
    {
        if (Icon)
        {
            CachedIcons.Add(Icon);
        }
    }

    // ═══════════════════════════════════════
    // FGCObject 接口实现
    // ═══════════════════════════════════════

    // 重写：告诉GC我们持有哪些UObject引用
    virtual void AddReferencedObjects(FReferenceCollector& Collector) override
    {
        // 注册所有UObject引用
        Collector.AddReferencedObject(ItemDataTable);
        Collector.AddReferencedObject(SkillDataTable);

        // 对于TArray中的UObject引用，用AddReferencedObjects重载
        Collector.AddReferencedObjects(CachedIcons);
    }

    // 重写：返回引用者名称（调试用）
    virtual FString GetReferencerName() const override
    {
        return TEXT("FDataManager");
    }

    // ===== 业务逻辑 =====
    bool HasValidDataTables() const
    {
        return IsValid(ItemDataTable) && IsValid(SkillDataTable);
    }
};
```

### 什么时候需要FGCObject

| 场景                            | 是否需要FGCObject                          |
| ------------------------------- | ------------------------------------------ |
| UObject类内部引用UObject        | ❌ 不需要，用UPROPERTY就好                 |
| 纯C++结构体（F开头）引用UObject | ✅ 需要！结构体不能放UPROPERTY             |
| 全局/单例管理器引用UObject      | ✅ 需要！避免全局管理器持有的UObject被回收 |
| 临时局部变量引用UObject         | ❌ 不需要，函数返回前GC不会回收它          |
| Slate UI类引用UObject           | ✅ 需要！Slate类不是UObject                |

## AddToRoot / RemoveFromRoot，防止GC回收

### 什么是"根"？

```cpp
// AddToRoot = "把这个对象加到GC根集中"
// 效果：只要对象在根集中，就永远不会被GC回收
// 代价：你必须手动RemoveFromRoot，否则永久内存泄漏

// 使用场景：你想让某个UObject在没有任何UPROPERTY引用时也不被回收
// 例如：全局单例对象、资源缓存等
```

### 基本用法

```cpp
// ===== 正确用法 =====
void KeepObjectAlive(UObject* Object)
{
    // 把Object添加到GC根集
    Object->AddToRoot();
    // 现在Object不会被GC回收了
    // 即使没有任何UPROPERTY指向它

    // ... 在需要的时间里使用Object ...

    // 用完了，从根集移除
    Object->RemoveFromRoot();
    // 现在Object可以被GC回收了
}

// ===== 配对原则 =====
// 每个AddToRoot必须有一个对应的RemoveFromRoot
// 就像new和delete一样要配对

// ✅ 正确：成对使用
void CorrectUsage()
{
    UMyObject* Obj = NewObject<UMyObject>();

    Obj->AddToRoot();        // 加根
    // ... 使用 ...
    Obj->RemoveFromRoot();   // 去根 ← 必须有这一行！
}

// ❌ 错误：只加不删 → 内存泄漏！
void WrongUsage()
{
    UMyObject* Obj = NewObject<UMyObject>();

    Obj->AddToRoot();        // 加根
    // ... 使用 ...
    // 忘记RemoveFromRoot！
    // Obj永远占据内存，直到程序退出！
}
```

### 典型使用场景

```cpp
// ===== 场景：全局资源缓存 =====
class UResourceCache : public UObject
{
    GENERATED_BODY()

public:
    // 缓存的资源
    UPROPERTY()
    TArray<UTexture2D*> CachedTextures;

    void Initialize()
    {
        // 让缓存自己不会被GC回收
        // 因为缓存可能没有被任何Actor引用
        // 但它需要在整个游戏期间存活
        AddToRoot();  // 把自己添加到根集
    }

    void Shutdown()
    {
        RemoveFromRoot();  // 允许自己被GC回收
    }
};

// ===== 场景：异步操作中的临时保护 =====
void LoadResourceAsync(const FString& Path)
{
    // 先创建一个空的数据容器
    UMyDataAsset* DataAsset = NewObject<UMyDataAsset>();

    // 临时保护：在异步加载完成前，不能让GC回收它
    DataAsset->AddToRoot();

    // 发起异步加载
    AsyncLoad(Path, [DataAsset](UObject* LoadedObject)
    {
        // 加载完成，填充数据
        DataAsset->InitializeFromAsset(LoadedObject);

        // 现在可以安全移除根保护了
        // （因为调用者应该保存了对DataAsset的UPROPERTY引用）
        DataAsset->RemoveFromRoot();
    });
}
```

### AddToRoot vs UPROPERTY vs FGCObject

| 方法             | 适用场景                | 优点                 | 缺点                         |
| ---------------- | ----------------------- | -------------------- | ---------------------------- |
| `UPROPERTY()`    | UObject子类持有引用     | 自动管理，最安全     | 只能在UObject类内部使用      |
| `FGCObject`      | 非UObject类需要持有引用 | 适用于纯C++类        | 需要手动实现接口             |
| `AddToRoot()`    | 全局对象/临时保护       | 强制保护，无视引用链 | 必须手动配对Remove，容易泄漏 |
| `TWeakObjectPtr` | 观察/监听但不持有       | 不阻止GC，无泄漏风险 | 不提供保护，对象可能被回收   |

## 强制GC

```cpp
// ===== 强制触发GC的几种方式 =====

// 方式1：通过GEngine（全局引擎实例）
if (GEngine)
{
    // 参数bFullPurge = true：执行完整GC
    // 包括标记、清除，还会清理一些额外的缓存
    GEngine->ForceGarbageCollection(true);

    // 参数bFullPurge = false：只清理PendingKill的对象
    // 不做完整标记-清除流程
    GEngine->ForceGarbageCollection(false);
}

// 方式2：通过UWorld
if (GetWorld())
{
    GetWorld()->ForceGarbageCollection(true);
}

// 方式3：通过命令行（调试用）
// 在游戏中按 ~ 打开控制台，输入：
// obj gc          - 执行一次GC
// obj list        - 列出所有UObject
// obj refs <name> - 查看某个对象的引用链

// ===== 何时需要手动GC =====

// ✅ 合理使用：卸载大关卡后立即回收
void OnLevelUnloaded()
{
    // 关卡卸载后，大量UObject变成不可达
    // 但可能要等60秒GC才会清理
    // 如果下一个关卡需要大量内存，手动GC是合理的
    GEngine->ForceGarbageCollection(true);
}

// ✅ 合理使用：编辑器工具（批量操作后）
void EditorTool_BatchProcess()
{
    // 处理了1000个资源...
    for (int32 i = 0; i < 1000; i++)
    {
        ProcessAsset(i);
    }
    // 清理临时对象
    GEngine->ForceGarbageCollection(true);
}

// ❌ 不合理使用：每帧都GC
void Tick(float DeltaTime)
{
    // ❌ 绝对不要这样！
    GEngine->ForceGarbageCollection(true);  // 严重卡顿！GC本身很耗时
}

// ❌ 不合理使用：反复触发
for (int i = 0; i < 100; i++)
{
    DoSomething();
    GEngine->ForceGarbageCollection(true);  // ❌ 太频繁，浪费时间
}
```
