# 17.1 简历与作品集

> **目标**：帮助你把本手册学到的知识和完成的实战项目，转化成一封能让面试官眼前一亮的简历，以及一个有说服力的作品集。

---

## 1. 简历的本质是什么？

先想清楚一个核心问题：**简历的目的是什么？**

很多人以为简历是"把自己做过的事情列出来"。这是错误的。

✅ **简历的真正目的**：在10-30秒的快速浏览中，让HR和面试官产生"这个人可以面试一下"的想法。

❌ **常见错误认知**：简历越长越好、把所有的经历都堆上去、用泛泛的空话填充篇幅。

请记住：**面试官的每一轮筛选都是淘汰制**。简历在HR手中平均只被看15秒。你的简历不是自传，而是一张"门票"——它的唯一使命就是让你进入面试环节。

---

## 2. 简历结构

一份标准的UE C++开发工程师简历，应该包含以下四个核心板块（按顺序）：

### 2.1 个人信息

```
┌──────────────────────────────────────────┐
│  姓名 | 求职意向 | 联系方式 | 作品链接    │
└──────────────────────────────────────────┘
```

**必须包含的内容**：
- 姓名（中文真实姓名）
- 求职意向：明确写"UE5 C++客户端开发工程师"或"UE5 游戏开发工程师"——不要写"软件工程师"这种笼统的描述
- 手机号码和邮箱
- GitHub个人主页链接（必须！）
- 如果有技术博客、知乎/掘金专栏、B站教程频道，一并列出

**可选内容**：
- 所在城市 / 期望工作城市
- 工作年限（如果有）
- 学历（学校、专业、毕业时间）

**不要写的内容**：
- ❌ 性别、年龄、民族、政治面貌（除非公司明确要求）
- ❌ 身高、体重、星座
- ❌ QQ号（显得不够专业，用邮箱代替）
- ❌ 自我评价段落（这部分应该通过项目经验来体现，而不是空洞地写"我学习能力强"）

**示例**：

```
张三
求职意向：UE5 C++客户端开发工程师 | 期望城市：上海
手机：138-xxxx-xxxx | 邮箱：zhangsan@example.com
GitHub：github.com/zhangsan | 技术博客：zhangsan.dev
```

### 2.2 技能清单

这是HR和面试官**最先会细看**的部分。你需要在这一节精准地告诉对方：我会什么，到什么程度。

#### ❌ 错误的写法

```
- 熟练掌握C++
- 了解游戏开发流程
- 熟悉数据结构与算法
- 会使用Unreal Engine
- 具备良好的团队协作能力
```

为什么这是错误的？
1. "熟练掌握C++"——每个人都在这么写，完全没有区分度
2. "熟悉数据结构与算法"——等于没说，面试官无法知道你具体会什么
3. "具备良好的团队协作能力"——技能清单里写软技能，浪费宝贵的版面

#### ✅ 正确的写法

技能清单应该按照**技术栈分层**，每一项都配套一个具体的技术点：

```markdown
## 技能清单

**游戏引擎**
- UE5：熟悉Actor/Component架构、Gameplay Framework、UI（UMG/Slate）
- 了解蓝图与C++的混合编程，能合理划分蓝图层与C++层

**核心系统**
- GAS（Gameplay Ability System）：能独立搭建Ability/Effect/Tag/Cue体系
- 网络同步：熟悉RPC机制、属性同步（Replication/RepNotify），理解客户端-服务器模型
- Enhanced Input System：熟悉InputAction/InputMappingContext，替代旧输入系统

**编程语言**
- C++（现代C++11/14/17）：智能指针、Lambda、模板、RAII
- 蓝图：可视化脚本编写与调试

**性能与调试**
- Unreal Insights、RenderDoc 基础使用
- 熟悉常见的性能优化手段（LOD、剔除、对象池、Draw Call优化）

**工具与协作**
- Git/SVN 版本控制
- Perforce（如有使用经验则写上）
- Visual Studio / Rider
```

**技能清单的编写原则**：
1. **写具体的技术名词**，不要写笼统的概念。比如写"GAS（Gameplay Ability System）"而不是"技能系统"
2. **突出UE专项技能**。通用C++开发能力只是基础，UE特有的技术才是你的竞争力
3. **诚实是最重要的**。技能清单上的每一项你都要准备好被深入提问。如果你只是看过文档但没有真正用过，就不要写"熟悉"，可以写"了解"
4. **分级表述**：精通 > 熟悉 > 了解。如果你只有学习阶段的经验，诚实地写"了解"并不会减分——但如果你写了"精通"却被两个问题问倒，就会直接出局

### 2.3 项目经验（最重要的部分）

对于游戏行业而言，**项目经验的重要程度远大于学历和技能清单**。

#### STAR法则

每个项目经验的描述，都应该遵循STAR法则：

| 字母 | 含义 | 说明 |
|------|------|------|
| **S**ituation | 情境 | 这个项目的背景是什么？ |
| **T**ask | 任务 | 你在项目中负责什么？ |
| **A**ction | 行动 | 你具体做了什么？用了什么技术方案？ |
| **R**esult | 结果 | 你做到了什么效果？有数据更好 |

#### 项目经验写作模板

对于每个项目，使用以下结构：

```markdown
## 项目名称 | 你的角色 | 项目时间

**项目简介**：（两句话说明这是什么项目，解决什么问题）

**技术栈**：UE5 C++、GAS、UMG、Replication...

**我的工作**：
- [STAR动作1]：为了实现XXX功能，我使用了YYY技术，最终达到了ZZZ的效果
- [STAR动作2]：...
- [STAR动作3]：...

**技术亮点**：
- 亮点1的简要说明
- 亮点2的简要说明

**项目链接**：[GitHub] [演示视频]
```

```markdown
**项目链接**：
- GitHub：github.com/zhangsan/DungeonExplorer
- 演示视频：bilibili.com/video/xxxxx
```

#### ❌ 错误的项目经验写法

```markdown
## 地牢探索者

这是一个地牢RPG游戏，我负责了全部开发工作。使用了UE5引擎开发，
实现了角色移动、技能、AI敌人、UI等功能。通过这个项目我学到了很多，
提升了我的C++和UE使用能力。
```

为什么这是错的？
1. **没有任何技术细节**："实现了角色移动"——是用CharacterMovementComponent还是自己写的物理？没有说清楚
2. **没有量化成果**："实现了AI敌人"——多少个敌人？什么行为？性能如何？
3. **没有体现技术决策**：为什么选择某个方案？遇到了什么问题？如何解决的？
4. **"学到了很多"属于无效信息**：面试官关心的是"你做了什么"，而不是"你学到了什么"

## 3. 简历排版建议

### 3.1 格式与长度

- **长度**：1页为最佳，最多不超过2页。对于初级岗位，1页完全足够
- **文件格式**：PDF（绝对不能发Word文档，因为不同设备打开可能排版混乱）
- **文件命名**：`姓名_UE5C++开发工程师_简历.pdf`——不要用`简历最终版3.pdf`这种文件名
- **字体**：正文使用思源黑体或微软雅黑，代码/技术名词可以使用等宽字体（Consolas / Source Code Pro）

### 3.2 排版原则

- 使用加粗突出关键词（技术名词、量化数据）
- 保持足够的留白，行间距1.5倍以上
- 使用项目符号（bullet points），不要用大段文字
- 颜色以黑白灰为主，不要使用过于花哨的配色
- 不要在简历中使用卡通头像或装饰图

### 3.3 检查清单

在投递简历之前，逐一核对：

- [ ] 没有错别字和语法错误（让朋友帮你看一遍）
- [ ] 所有链接都可以打开（GitHub、博客、视频）
- [ ] 电话号码和邮箱正确
- [ ] 技能清单上的每一项都经得起深入提问
- [ ] 文件名规范，格式为PDF
- [ ] 不需要翻页就能看到核心信息（技能+第一个项目）

## 5. 作品集的准备

简历帮你拿到面试机会，**作品集帮你通过面试**。在游戏行业，作品集的重要程度远超其他行业。

### 5.1 GitHub项目

#### ✅ 好的GitHub项目

一个好的UE C++ GitHub项目应该包含：

**项目仓库结构**：
```
DungeonExplorer/
├── README.md            ← 最重要！好README = 好第一印象
├── Source/
│   ├── DungeonExplorer/
│   │   ├── Character/
│   │   ├── AbilitySystem/
│   │   ├── AI/
│   │   └── UI/
├── Content/             ← 如果仓库不大，可以包含基础Content
├── Config/
├── Docs/                ← 架构文档、设计思路
│   ├── architecture.md
│   └── network-sync.md
├── Plugins/             ← 自定义插件
└── .gitignore           ← 必须，不要提交Intermediate/Binaries
```

**README.md 必备内容**：

```markdown
# Dungeon Explorer

基于UE5的动作RPG游戏Demo，演示GAS+网络同步架构。

## 功能演示（GIF/截图）
[放3-5张关键截图或GIF，展示战斗、UI、技能等核心功能]

## 技术特点
- GAS技能系统：Ability/Effect/Tag/Cue完整链路
- Client-Server网络：RPC+属性同步，100ms内稳定
- AI行为树：巡逻→追击→攻击→躲避
- Enhanced Input：键鼠+手柄双模式

## 架构概览
[放一张核心类的UML图或架构图]

## 如何运行
1. 克隆仓库
2. 右键 DungeonExplorer.uproject → Generate Visual Studio project files
3. 用Visual Studio编译
4. 在UE编辑器中打开

## 技术文档
- [技能系统设计](Docs/ability-system.md)
- [网络同步方案](Docs/network-sync.md)

## 许可证
MIT
```

**代码规范要点**（面试官会看你的代码！）：
- 变量命名遵循UE规范：`bIsDead`（bool前缀b）、`Health`（float/int首字母大写）
- 注释清晰但不啰嗦：每个类和关键函数有简要说明
- 没有注释掉的大段死代码
- 目录结构清晰，不把所有文件堆在一个文件夹里
- 有 `.gitignore` 且正确配置（排除Intermediate、Binaries、DerivedDataCache等）

#### ❌ 差的GitHub项目特征

- 没有README，或者README只有一行"UE5项目"
- 目录混乱，所有.cpp/.h混在一起
- 大量注释掉的代码没有删除
- 没有.gitignore，上传了编译中间文件
- 代码没有统一的命名风格，时而UE风格时而Google风格
- 没有功能截图或演示，面试官无法直观了解项目

### 5.2 可运行的Demo

手上有**可运行的游戏Demo**是一个巨大的加分项。面试官可以直观地评估你的技术水准。

#### 推荐做法（按优先级）：

1. **打包好的exe文件**（最佳）
   - 上传到百度网盘、123云盘或Google Drive
   - 在简历和GitHub README中提供下载链接
   - 确保打包配置正确，能在没有UE编辑器的机器上运行
   - 打包前检查：关闭Development模式的调试输出、确保必要的DLL都包含

2. **演示视频**（次佳，但更易传播）
   - 录制3-5分钟的功能演示视频
   - 上传到B站 / YouTube
   - 视频内容建议：
     - 0:00-0:30：快速展示核心玩法（战斗、技能）
     - 0:30-1:30：展示技术要点（UI、网络同步、AI行为）
     - 1:30-2:30：展示编辑器中的架构（蓝图节点、C++代码结构）
     - 2:30-3:00：总结技术栈
   - 如果涉及网络同步，建议用分屏（客户端1 + 客户端2 + 服务器）来展示

3. **WebGL版本**（如果有条件）
   - UE5支持打包为WebGL，方便面试官直接在浏览器中体验

---

### 5.3 技术博客

技术博客不是必须的，但它是**区分应聘者层次**的重要加分项。

#### 技术博客写什么？

✅ **好的技术博客主题**：
- "UE5 GAS技能同步的三种方案及性能对比"
- "深入理解UE5的垃圾回收机制"
- "从零实现一个UE5自定义Component"
- "UE5 Enhanced Input System源码解析"
- "解决UE5网络同步中的属性冲突问题"
- 你在项目中踩过的坑和解决方案

❌ **不建议写的内容**：
- "今天开始学习UE5"（流水账）
- 纯翻译官方文档
- 内容过于浅显（如"什么是Actor"）

#### 博客平台推荐
- **GitHub Pages + Hexo/Hugo**：技术含量最高，展示你的工程能力
- **知乎专栏**：国内游戏行业HR和技术负责人常用
- **掘金**：技术社区，覆盖面广
- **CSDN**：有流量但质量参差不齐，建议作为辅助

#### 博客的作用
- 证明你的技术深度（能写清楚 = 真正理解）
- 证明你的学习能力（持续输出）
- 作为面试中的谈资（面试官可能会顺着你的博客提问）

---

### 5.4 至少需要1-2个完整的UE C++项目

**一个项目是不够的**。为什么？

- 1个项目只能展示一个维度的能力（比如纯单机玩法）
- 2个项目可以展示广度（比如一个侧重Gameplay、一个侧重引擎扩展）
- 更多项目可以让面试官看到你的技术成长轨迹

**推荐的项目组合**：

| 项目类型 | 侧重点 | 示例 |
|----------|--------|------|
| Gameplay项目 | 综合游戏玩法 | 本手册第16章的"地牢探索者" |
| 专项技术Demo | 某个技术点的深入研究 | GAS技能同步Demo、AI行为树Demo |
| 工具/插件 | 编辑器扩展 | 自定义UE编辑器工具、自动化流程 |
| Game Jam作品 | 快速开发能力 | 48小时内完成的创意游戏 |

**如果你只有一个大项目怎么办？**
- 把大项目拆成多个技术模块，每个模块写一篇深度技术博客
- 单独提取某个技术点做一个独立Demo（比如从项目中提取GAS部分做成独立插件）
- 参加一次Game Jam，产出第二个项目

## 6. 没有工作经验怎么办？

大多数应届生和转行者都面临这个问题。好消息是：**游戏行业对个人项目的认可度远高于其他行业**。

### 6.1 用个人项目替代工作经验

在简历的"项目经验"部分，直接把个人项目放在最上面。不要留一个空的"工作经历"板块。

如果确实没有任何工作经验，建议的结构是：

```
个人信息
技能清单
项目经验（2-3个）
  ├── 项目A（你的主力UE C++项目）
  ├── 项目B（技术专项Demo）
  └── 项目C（Game Jam或其他小项目）
教育背景
开源贡献（如果有）
```

### 6.2 开源贡献

如果你能给知名的开源项目贡献代码，这是非常强的信号。

**可以从这些方向入手**：
- UE5 Marketplace上的免费插件，修复它们的bug或添加功能
- Unreal Engine官方源码的bug修复（通过Epic Games的GitHub提PR）
- 参与GameplayAbilitySystem相关的社区项目
- 翻译UE5文档或教程

即使只是修复了一个拼写错误或补充了一段注释，你也可以在简历上写：

> 为UE5官方文档贡献了中文本地化翻译（已合并3个PR）

### 6.3 实习

如果你还在校，**尽一切可能在毕业前拿到一份游戏开发的实习**。

- 大厂的实习经历是简历上最强的背书
- 即使是小工作室的实习，也好过完全没有
- 实习要趁早：大三暑假是最佳时机

## 7. 简历投递渠道

### 7.1 国内渠道（按推荐程度排序）

| 渠道 | 说明 | 适用情况 |
|------|------|----------|
| **内推** | 找在目标公司的学长/朋友/社群联系人帮你内推 | **最优先**，简历直达HR且处理速度更快 |
| **Boss直聘** | 目前国内游戏行业最活跃的招聘平台 | 应届生和社招都适合 |
| **拉勾网** | 偏互联网/科技领域 | 少数游戏公司也在这里招人 |
| **脉脉** | 职场社交平台，可以用来找内推和了解公司 | 适合获取信息而非直接投递 |
| **公司官网** | 大厂（腾讯、网易、米哈游等）的校招通道 | 校招必投 |

### 7.2 海外渠道

| 渠道 | 说明 |
|------|------|
| **LinkedIn** | 海外游戏公司最主要的招聘渠道 |
| **Indeed / Glassdoor** | 搜索海外游戏公司岗位 |
| **ArtStation** | 偏美术，但部分技术岗位也会发布 |

### 7.3 投递策略

✅ **应该做的**：
- 每个渠道的简历都根据公司和岗位做微调（至少改一下"求职意向"和项目排序）
- 每周固定时间集中投递（比如周一早上和周三晚上），保持规律
- 记录每次投递的公司名、岗位、投递时间、渠道（用Excel或Notion记录）
- 同一家公司最多投2个岗位，多了会让HR觉得你没有方向

❌ **不应该做的**：
- 用同一封简历海投所有公司
- 同一家公司一次性投5个岗位
- 简历里写"服从分配"或"任何岗位都行"——这传递的信号是你不知道自己想做什么
- 在社交媒体上公开询问"XX公司怎么样"——可能会被HR看到

### 7.4 投递后的跟进

- 投递后3-5个工作日没有回复 → 可以在招聘平台发一条礼貌的追问
- 如果被拒绝了 → 可以礼貌地问一下原因（但不一定会得到回复）
- **不要**反复催促HR，一天发3条消息只会适得其反

## 8. 简历投递的常见误区

| 误区 | 真相 |
|------|------|
| "简历越长越好" | 1页最佳，2页最多。HR没有耐心看长简历 |
| "技能写得越多越好" | 写得越多，被问倒的概率越大。只写你真正掌握的 |
| "没有工作经验就完了" | 游戏行业重视作品，好项目 > 普通工作经验 |
| "海投就能提高命中率" | 3封精心准备的简历 > 100封群发的简历 |
| "只要技术好简历随便写" | 好技术 + 烂简历 = HR直接淘汰，根本没机会展示技术 |
| "等准备好了再投" | 永远不会有"完全准备好"的时刻。边投边学是常态 |

## 9. 本章核心检查清单

在开始投递之前，请逐项确认：

- [ ] 你有一个完整的UE C++项目（或正在完善中）
- [ ] 你的GitHub仓库有清晰的README（含截图/架构图）
- [ ] 你的简历是PDF格式，长度1-2页
- [ ] 技能清单中的每一项都是具体的技术名词（不是笼统的描述）
- [ ] 项目描述使用了STAR法则（情境-任务-行动-结果）
- [ ] 你录制了一个项目演示视频（上传到B站或YouTube）
- [ ] 你至少写了一篇技术博客（哪怕只是记录项目中解决的一个问题）
- [ ] 你准备好了对技能清单上每一项被深入提问
