# 11.1 行为树基础：AI的大脑

> **目标**：理解行为树（Behavior Tree）是什么，为什么UE5选择它作为AI决策系统，以及它的核心节点类型和运行机制。

---

## 前言：一个"聪明"的AI需要什么？

想象你正在做一个敌人AI。它需要：

- 巡逻时按路线走路
- 看到玩家时追击
- 追到后攻击
- 血量低时逃跑
- 队友死了就愤怒

这些行为如果用`if-else`写，代码会长这样：

```cpp
// ❌ 用if-else管理复杂AI行为——噩梦开始的地方
void AEnemy::Tick(float DeltaTime)
{
    if (CurrentState == EState::Patrol)
    {
        if (SawPlayer())
        {
            CurrentState = EState::Chase;  // 转追击
        }
        else if (ReachedWaypoint())
        {
            MoveToNextWaypoint();  // 走向下一个路径点
        }
    }
    else if (CurrentState == EState::Chase)
    {
        if (LostPlayer())
        {
            CurrentState = EState::Patrol;  // 返回巡逻
        }
        else if (InAttackRange())
        {
            CurrentState = EState::Attack;  // 转攻击
        }
        else
        {
            MoveTowardPlayer();
        }
        // 每次加新条件都要改这坨面条代码...
    }
    // ... 还可能有十几个状态
}
```

这种写法叫做**有限状态机（Finite State Machine, FSM）**。它有两个致命的缺点：

| 状态机的缺点 | 具体表现                                 | 行为树的解决方案                             |
| ------------ | ---------------------------------------- | -------------------------------------------- |
| **难以扩展** | 加一个新状态要修改大量if-else            | 节点是独立的，加新行为不影响旧行为           |
| **难以复用** | "看到玩家"这个判断在每个状态里都要写一遍 | Decorator（装饰器）可复用条件判断            |
| **难以调试** | 一堆嵌套if-else，AI"卡住"了找不到原因    | 行为树编辑器可视化运行，哪个节点在跑一目了然 |
| **难以协作** | 程序员写代码，策划改不了AI逻辑           | 策划可以在编辑器里拖拽节点修改行为树         |

行为树就是来解决这些问题的。

---

## 什么是行为树？

行为树是一棵**倒置的树**——根在上面，叶子在下面。每一"帧"，AI都从根节点开始向下执行，像一条电流从树根流向某个叶子。

```
                    ┌───────────┐
                    │   Root    │  ← 入口，每一帧都从这里开始
                    └─────┬─────┘
                          │
                    ┌─────▼─────┐
                    │ Selector  │  ← "从左到右挑一个能执行的"
                    └┬──┬──┬───┘
                     │  │  │
          ┌──────────┘  │  └──────────┐
          ▼             ▼             ▼
    ┌─────────┐   ┌─────────┐   ┌─────────┐
    │ Sequence│   │ Sequence│   │ Idle    │  ← 叶子：具体的Task（任务）
    │ (战斗)  │   │ (巡逻)  │   │         │
    └────┬────┘   └────┬────┘   └─────────┘
         │             │
    ┌────▼────┐   ┌────▼────┐
    │ 追击玩家 │   │ 移动到  │
    │ (Task)  │   │ 路径点  │
    └─────────┘   │ (Task)  │
                  └─────────┘
```

> **核心思想**：把复杂的AI逻辑拆成一颗颗独立的"积木"（节点），然后在编辑器中像搭积木一样组合起来。

## 行为树 vs 状态机：用图表说话

假设我们做一个"巡逻→追击→攻击→返回"的敌人：

### 状态机版本（混乱）

```
         ┌──────看到玩家────→┌────────┐
         │                    │  追击  │
    ┌────┴───┐               └───┬────┘
    │  巡逻   │                  │
    └────┬───┘         进入攻击范围│
         │                         ▼
         │                   ┌────────┐    ┌────────┐
         │                   │  攻击  │───→│ 死亡   │← 死了
         │                   └───┬────┘    └────────┘
         │             丢失目标│
         └─────────────←──────┘

问题：每个箭头都是代码中的if判断，加一个"呼叫队友"状态需要画6条新箭头！
```

### 行为树版本（清晰）

```
Root (Selector)
├── Sequence (战斗分支) ← 有目标时执行
│   ├── Decorator: 目标是否存在？
│   ├── Task: 追击目标
│   └── Task: 攻击目标
└── Sequence (巡逻分支) ← 没目标时执行（备选）
    ├── Task: 移动到路径点
    └── Task: 等待2秒

加"呼叫队友" → 只需插入一个新Sequence，不影响旧逻辑！
```

## 行为树的四大核心节点类型

行为树中的所有节点都继承自 `UBTNode`。分为四大类：

```
UBTNode（基础节点——所有行为树节点的父类）
├── UBTCompositeNode（组合节点——控制子节点的执行顺序）
│   ├── UBTComposite_Selector       ← "或"逻辑
│   ├── UBTComposite_Sequence       ← "与"逻辑
│   └── UBTComposite_SimpleParallel ← "并行"逻辑
├── UBTTaskNode（任务节点——执行具体行为）
│   ├── UBTTask_MoveTo              ← 引擎内置：移动到目标
│   ├── UBTTask_Wait                ← 引擎内置：等待N秒
│   ├── UBTTask_RunBehavior         ← 引擎内置：运行子行为树
│   └── UBTTask_XXX                 ← 你自定义的C++ Task
├── UBTDecorator（装饰器——条件判断，挂在其他节点上）
│   ├── UBTDecorator_Blackboard     ← 引擎内置：检查黑板值
│   ├── UBTDecorator_Cooldown       ← 引擎内置：冷却时间
│   └── UBTDecorator_XXX            ← 你自定义的C++ Decorator
└── UBTService（服务——后台持续运行的逻辑）
    ├── UBTService_DefaultFocus     ← 引擎内置：默认注视
    └── UBTService_XXX              ← 你自定义的C++ Service
```

---

### 1. Composite节点（组合节点——管"顺序"的）

Composite节点本身不做具体事。它像一个**项目经理**，决定下属（子节点）以什么顺序、什么逻辑去执行。

#### Selector（选择器）——"或"逻辑

**执行规则**：从左到右执行子节点，**第一个返回Success的胜出**，后续不再执行。

```
Selector: "从左到右，谁行谁上"
┌──────────────────────────────────────┐
│              Selector                │
│                                      │
│  子节点1 ──→ 子节点2 ──→ 子节点3     │
│  (失败)      (成功!)      (跳过)     │
│                                      │
│  结果：成功（因为子节点2成功了）       │
└──────────────────────────────────────┘

全部子节点都失败 → Selector返回失败
```

**典型用法**：做"优先级行为"

```cpp
// 行为树编辑器中的等价概念（不是C++代码，是概念演示）
// Selector: 战斗 > 巡逻 > 待机
//   子1: Sequence(战斗分支)  ← 优先级最高，看到玩家就战斗
//   子2: Sequence(巡逻分支)  ← 优先级中等，没玩家就巡逻
//   子3: Task(待机)          ← 优先级最低，前面都失败就发呆
```

#### Sequence（序列）——"与"逻辑

**执行规则**：从左到右执行子节点，**必须全部成功才返回成功**，任何子节点失败则立即停止。

```
Sequence: "一步失败，全盘皆输"
┌──────────────────────────────────────┐
│              Sequence                │
│                                      │
│  子节点1 ──→ 子节点2 ──→ 子节点3     │
│  (成功)      (失败!)      (停止)     │
│                                      │
│  结果：失败（因子节点2中断）          │
└──────────────────────────────────────┘

全部子节点成功 → Sequence返回成功
```

**典型用法**：做"步骤链"

```cpp
// 行为树编辑器中的等价概念（不是C++代码，是概念演示）
// Sequence: 攻击步骤
//   子1: Decorator(敌人在攻击范围内？) ← 先检查条件
//   子2: Task(播放攻击动画)            ← 再执行步骤1
//   子3: Task(造成伤害)                ← 最后执行步骤2
//   任何一步失败→整条Sequence失败
```

#### 两种Composite的对比记忆

| 特性         | Selector                             | Sequence                                 |
| ------------ | ------------------------------------ | ---------------------------------------- |
| **逻辑**     | "或"（OR）                           | "与"（AND）                              |
| **成功条件** | 任一子节点成功                       | 所有子节点成功                           |
| **失败条件** | 所有子节点失败                       | 任一子节点失败                           |
| **类比**     | 选择外卖：先看麦当劳，没开就看肯德基 | 做饭：先买菜→再洗菜→再炒菜，缺一步都不行 |
| **使用场景** | 行为优先级（战斗>巡逻>待机）         | 步骤顺序（检查→执行→确认）               |

#### Simple Parallel（简单并行）

**执行规则**：同时运行一个主任务（左侧）和一个后台任务（右侧），主任务结束时整体结束。

```
Simple Parallel
├── 主Task（会阻塞到完成，比如"移动到目标"）
└── 后台Task（每一帧都在跑，比如"持续更新注视方向"）
```

> **注意**：Simple Parallel不是真正的多线程并行，它仍然在游戏线程的同一帧中执行。只是让两个节点"看起来"同时运行。实际项目中较少使用。

---

### 2. Task节点（任务节点——干"实事"的）

Task是行为树的**叶子节点**，执行具体的游戏行为。

```cpp
// Task的生命周期
ExecuteTask()   ← 接到任务时调用（只调用一次）
     │
     ├── 返回 Succeeded  ← 任务完成
     ├── 返回 Failed     ← 任务失败
     └── 返回 InProgress ← 任务还在进行中...
              │
         TickTask()      ← 如果返回InProgress，每帧调用（可选）
              │
              └── 最终返回 Succeeded 或 Failed
```

**引擎内置的常用Task：**

| 任务                      | 功能                             | 关键参数                 |
| ------------------------- | -------------------------------- | ------------------------ |
| `Move To`                 | 让AI移动到指定目标（Actor/位置） | 目标黑板Key、可接受半径  |
| `Wait`                    | 等待指定时间                     | 等待秒数、是否随机偏差   |
| `Rotate to Face BB Entry` | 旋转面向黑板中指定的目标         | 目标Key                  |
| `Run Behavior`            | 运行另一个行为树（子行为树）     | 目标行为树资产           |
| `Play Sound`              | 播放音效                         | 音频资产                 |
| `Finish With Result`      | 直接返回指定结果（测试用）       | 返回结果（Success/Fail） |

---

### 3. Decorator节点（装饰器——管"条件"的）

Decorator像一个**门卫**，挂在Composite或Task节点上，在节点执行前检查条件。

```
          Sequence (攻击)
          │
     ┌────┴────┐
     │Decorator│  ← 挂在Sequence上："目标是否在攻击范围内？"
     │ (条件)  │     条件不满足时 → Sequence直接跳过（失败）
     └────┬────┘
          │
     ┌────┴────┐
     │  追击   │    如果Decorator通过，这步才执行
     │  Task   │
     └─────────┘
```

**Decorator的检查时机（重要！）：**

| 检查时机                              | 含义                             | 使用场景                   |
| ------------------------------------- | -------------------------------- | -------------------------- |
| **On Node Activation**（节点激活时）  | 每次要执行这个节点时检查一次     | "目标是否存在？"           |
| **On Tree Start**（行为树开始时）     | 只在整个行为树启动时检查一次     | "AI是否处于战斗状态？"     |
| **Tick**（每帧都检查）                | 每一帧都重新检查条件             | "目标是否还在可视范围内？" |
| **Conditional Execution**（条件执行） | 当节点下方的某个子节点执行时检查 | "攻击冷却是否结束？"       |

**常用的内置Decorator：**

| 装饰器               | 功能                                |
| -------------------- | ----------------------------------- |
| `Blackboard`         | 检查黑板中的值是否满足条件          |
| `Cooldown`           | 该节点执行后有冷却时间              |
| `Loop`               | 让子节点循环执行N次                 |
| `Time Limit`         | 子节点超过指定时间没完成则失败      |
| `Force Success`      | 无论子节点返回什么都强制返回Success |
| `Does Path Exist`    | 检查A到B是否存在导航路径            |
| `Compare BB Entries` | 比较两个黑板值                      |

---

### 4. Service节点（服务——管"后台"的）

Service挂在Composite节点上，在Composite节点**激活期间**持续运行。

它和Decorator的区别：

| 对比维度     | Decorator             | Service                     |
| ------------ | --------------------- | --------------------------- |
| **做什么**   | 检查条件（返回真/假） | 更新数据（无返回值）        |
| **频率**     | 按配置的时机检查      | 按固定间隔执行（如每0.5秒） |
| **类比**     | 门卫（让不让过）      | 侦察兵（持续报告情报）      |
| **典型用途** | "能看到玩家吗？"      | "每0.5秒更新玩家位置到黑板" |

```
Selector
├── Sequence (战斗) ← 该Sequence激活时，Service持续运行
│    [Service: 每0.3秒更新"玩家位置"到黑板]  ← 挂在Sequence上
│   ├── Decorator: 目标存在？
│   ├── Task: 追击目标（使用黑板中的"玩家位置"）
│   └── Task: 攻击
└── Sequence (巡逻)
    [Service: 每1秒更新"下一个路径点"到黑板]
    ├── Task: 移动到路径点
    └── Task: 等待
```

**常用的内置Service：**

| 服务            | 功能                           |
| --------------- | ------------------------------ |
| `Default Focus` | 设置AI默认注视的目标           |
| `Run EQS`       | 运行环境查询系统（找最佳位置） |

## 行为树的运行机制

### "心跳"模型：每一帧的执行流程

```
游戏主循环（每帧）
  │
  ▼
AIController::Tick()（如果行为树在运行）
  │
  ▼
BehaviorTreeComponent::Tick(DeltaTime)
  │
  ▼
从Root开始，"电流"向下流动：

每一帧(AI大脑的"一次心跳"):
┌─────────────────────────────────────────────┐
│                                             │
│  Step1: 从Root开始                          │
│     │                                       │
│  Step2: 执行当前活跃的Composite子节点        │
│     │    └── 触发Service的执行（如果间隔到了）│
│     │                                       │
│  Step3: 检查Decorator（条件判断）             │
│     │    ├── 通过 → 继续向下                 │
│     │    └── 不通过 → 该分支失败，回退       │
│     │                                       │
│  Step4: 到达Task节点，执行具体行为            │
│     │    ├── InProgress → 记录当前位置，       │
│     │    │                 下一帧从这里继续   │
│     │    ├── Succeeded → 标记成功，向上传递   │
│     │    └── Failed → 标记失败，向上传递     │
│     │                                       │
│  Step5: 结果向上冒泡                         │
│     │    └── Composite根据规则决定              │
│     │         下一个执行谁                   │
│                                             │
└─────────────────────────────────────────────┘
```

### 关键概念：树的"激活"和"休眠"

```cpp
// 概念演示——行为树内部的工作逻辑（不是你要写的代码）

// 当树首次激活一个节点时：
OnNodeActivation(Node)
{
    1. 调用该节点（及父节点）上挂的Service的OnBecomeRelevant()
    2. Service开始按间隔Tick
    3. Decorator执行条件检查
    4. 如果条件通过，进入Task的ExecuteTask()
}

// 当树离开一个节点时：
OnNodeDeactivation(Node)
{
    1. 调用该节点上挂的Service的OnCeaseRelevant()
    2. Service停止Tick
    3. 如果Task还在InProgress，调用AbortTask()
}

// 当树完全停止时（比如AI死了）：
OnTreeStop()
{
    // 所有正在运行的Service停止
    // 所有正在运行的Task被Abort
    // BehaviorTreeComponent进入空闲状态
}
```

## 行为树的三种"停用"（关键概念）

当一个正在运行的Task被更高优先级的节点打断时，UE有三种处理方式：

| 终止模式           | 英文           | 含义                                 |
| ------------------ | -------------- | ------------------------------------ |
| **Both**           | Both           | 当前节点及其所有子节点都被终止       |
| **Self**           | Self           | 只终止当前节点，子节点保留           |
| **Lower Priority** | Lower Priority | 终止所有比当前节点优先级低的兄弟节点 |

> 这是行为树编辑器中的`Abort Mode`设置，配置在Decorator上，控制条件触发时的行为。

## 常见的行为树设计模式

### 模式1：优先级巡逻

```
Root (Selector)
├── Sequence [优先级：高]
│   ├── Decorator: 看到玩家？
│   └── Task: 追击玩家
├── Sequence [优先级：中]
│   ├── Decorator: 听到枪声？
│   └── Task: 调查声音源
└── Sequence [优先级：低]
    ├── Task: 走向路径点A
    └── Task: 走向路径点B
```

### 模式2：条件检查链

```
Sequence (攻击流程)
├── Decorator: 敌人在攻击范围吗？
├── Task: 播放攻击动画
├── Decorator: 攻击命中了吗？
└── Task: 造成伤害
```

### 模式3：冷却/间隔执行

```
Sequence
├── Decorator: [Cooldown: 5秒]
└── Task: 释放技能
```

> Cooldown Decorator保证Task每5秒最多执行一次。

## ❌ 常见误区

| 误区                             | 为什么错                                           | ✅ 正确做法                           |
| -------------------------------- | -------------------------------------------------- | ------------------------------------- |
| 把Selector当Sequence用           | Selector是"或"逻辑，第一个子节点成功就停止         | 步骤顺序用Sequence                    |
| 在Task中写死循环                 | 行为树每帧执行Task，死循环会让游戏卡死             | 需要等待时返回InProgress，下一帧继续  |
| Decorator不设置检查时机          | 默认只在节点激活时检查，条件变了不会重新评估       | 需要持续检查的设置Tick                |
| 不用Blackboard直接在Task里找对象 | 多个Task各自做FindAllActorsOfClass，性能极差       | 用Service统一更新Blackboard，Task只读 |
| 一棵行为树管理所有AI             | 不同AI（近战兵、远程兵、Boss）共用一棵树会极其复杂 | 每类AI一棵行为树，可复用子行为树      |
