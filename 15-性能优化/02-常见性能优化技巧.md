## 1. CPU 优化

CPU 优化是游戏性能优化中最常见的类别。CPU 瓶颈通常表现为：`stat unit` 中的 `Game` 时间远超 16.67ms。

### 1.1 减少 Tick 频率

Tick 是 UE 中最常见的性能杀手 —— 场景里每个 Actor 默认每帧都会 Tick。如果有 1000 个 Actor，每帧就要调用 1000 次 Tick 函数。

#### 1.1.1 禁用不需要的 Tick

```cpp
// ---------- AMyStaticProp.h ----------
#pragma once
#include "CoreMinimal.h"
#include "GameFramework/Actor.h"
#include "AMyStaticProp.generated.h"

UCLASS()
class MYPROJECT_API AMyStaticProp : public AActor
{
    GENERATED_BODY()

public:
    AMyStaticProp()
    {
        // ✅ 构造函数中直接关闭 Tick
        // 树、石头、路灯等静态装饰物根本不需要每帧更新
        // 关闭 Tick 后，这个 Actor 只占用内存，不消耗 CPU
        PrimaryActorTick.bCanEverTick = false;
        //                   └设为 false = 这个 Actor 永远不会 Tick
        //                    设为 true  = 这个 Actor 需要 Tick（默认值）
    }
};
```

```cpp
// ---------- AMyEnemy.cpp ----------
// ❌ 错误示范：函数体内判断后 return，但 Tick 仍然被调用了
void AMyEnemy::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    // 这个 Actor 的 Tick 每帧都被调用
    // 即使每次进来就 return，函数调用本身也有开销！
    // 如果有 1000 个敌人，等于每帧调用了 1000 次空函数
    if (bIsDead)
    {
        return;  // 已经死了，什么都不做
    }

    // ... 正常的 Tick 逻辑
}

// ✅ 正确做法1：在构造时就决定是否需要 Tick
AMyEnemy::AMyEnemy()
{
    PrimaryActorTick.bCanEverTick = true;   // 允许 Tick
    PrimaryActorTick.bStartWithTickEnabled = true;  // 生成时开始 Tick
}

void AMyEnemy::Die()
{
    // 敌人死亡后，关闭它的 Tick
    // SetActorTickEnabled 是开关，随时可以开/关
    SetActorTickEnabled(false);
    //                 └false = 停止 Tick
    //                  true  = 恢复 Tick

    // 这比在 Tick 里写 if(bIsDead) return; 高效得多
    // 因为引擎根本不会调用这个 Actor 的 Tick 函数
}

// ✅ 正确做法2：降低 Tick 间隔（不需要每帧 Tick 的 Actor）
void AMyEnemy::BeginPlay()
{
    Super::BeginPlay();

    // 设置 Tick 间隔为 0.5 秒（即每 0.5 秒 Tick 一次，而不是每帧）
    // 默认值是 0.0，表示每帧都 Tick
    PrimaryActorTick.TickInterval = 0.5f;
    //                            └单位：秒
    //                             0.5 = 每秒只 Tick 2 次
    //                             0.0 = 每帧都 Tick（60fps下每秒60次）

    // 🎯 性能提升计算：
    // 如果场景有 500 个敌人，原本每帧 500 次 Tick
    // 设置 TickInterval = 0.5 后，每帧只 Tick 约 17 次（500 / 30）
    // 性能提升约 30 倍！
}
```

#### 1.1.2 用 Timer 替代 Tick

```cpp
// ---------- AMyPatrolGuard.h ----------
#pragma once
#include "CoreMinimal.h"
#include "GameFramework/Actor.h"
#include "AMyPatrolGuard.generated.h"

UCLASS()
class MYPROJECT_API AMyPatrolGuard : public AActor
{
    GENERATED_BODY()

public:
    AMyPatrolGuard()
    {
        // ✅ 关闭 Tick —— 我们不需要每帧更新
        PrimaryActorTick.bCanEverTick = false;
    }

protected:
    virtual void BeginPlay() override;

    // 定时器句柄，用于管理定时器的生命周期
    FTimerHandle PatrolTimerHandle;
    //            └用于取消定时器的"身份证"
    //             不需要时可以调用 GetWorldTimerManager().ClearTimer(PatrolTimerHandle)

    // 巡逻检查函数 —— 每 2 秒执行一次
    void CheckPatrolPoint();

    // 巡逻点数组
    UPROPERTY(EditAnywhere, Category = "巡逻设置")
    TArray<AActor*> PatrolPoints;    // 储存巡逻点的Actor引用

    UPROPERTY(EditAnywhere, Category = "巡逻设置")
    float PatrolCheckInterval = 2.0f;  // 巡逻检查间隔（秒）
};

// ---------- AMyPatrolGuard.cpp ----------
#include "AMyPatrolGuard.h"
#include "TimerManager.h"  // 必需，TimerManager 在这里定义

void AMyPatrolGuard::BeginPlay()
{
    Super::BeginPlay();

    // ✅ 使用 Timer 替代 Tick：每 2 秒检查一次巡逻点
    GetWorldTimerManager().SetTimer(
        PatrolTimerHandle,          // 1. 定时器的"身份证"（用于后续取消）
        this,                        // 2. 回调函数所属的对象（this = 当前巡逻兵）
        &AMyPatrolGuard::CheckPatrolPoint, // 3. 要定时调用的函数地址
        PatrolCheckInterval,         // 4. 定时间隔（2.0秒）
        true                         // 5. 是否循环（true = 每隔2秒执行一次，直到手动取消）
        //                       └如果是 false，只执行一次就自动取消
    );

    // 对比：
    // ❌ 用 Tick：每秒执行 60 次（60fps），60 * 2 = 120 次不必要的空跑
    // ✅ 用 Timer：每 2 秒只执行 1 次
    // 性能差距：120 倍！
}

void AMyPatrolGuard::CheckPatrolPoint()
{
    // 这个函数每 2 秒才执行一次，而不是每帧
    // 适合不需要实时响应的逻辑：巡逻路径检查、血量恢复、资源刷新等

    if (PatrolPoints.Num() == 0)
    {
        return;  // 没有巡逻点，直接返回
    }

    // 获取当前要去的巡逻点
    int32 CurrentIndex = FMath::RandRange(0, PatrolPoints.Num() - 1);
    AActor* TargetPoint = PatrolPoints[CurrentIndex];

    if (TargetPoint)
    {
        UE_LOG(LogTemp, Log, TEXT("巡逻兵正在前往巡逻点: %s"), *TargetPoint->GetName());
        // 移动逻辑...
    }
}
```

**Timer vs Tick 的选择原则：**

| 需求                                 | 推荐方案                   | 原因                 |
| ------------------------------------ | -------------------------- | -------------------- |
| 每帧都需要更新（如角色移动输入）     | Tick                       | Timer 做不到每帧执行 |
| 每隔一段时间检查一次（如巡逻、回复） | Timer                      | 比 Tick 高效数十倍   |
| 延迟执行一次性操作（如3秒后爆炸）    | Timer（不循环）            | 比手动计时要优雅     |
| 根据条件动态开关                     | Tick + SetActorTickEnabled | Timer 不好频繁开/关  |

### 1.2 避免在 Tick 中做复杂计算

```cpp
// ---------- ❌ 错误示范：在 Tick 中做重复的昂贵计算 ----------
void AMyCharacter::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    // ❌ 错误1：每帧都查找所有敌人（场景有500个敌人时非常慢）
    TArray<AActor*> AllEnemies;
    UGameplayStatics::GetAllActorsOfClass(GetWorld(), AEnemy::StaticClass(), AllEnemies);
    //                 └这个函数要遍历场景中所有 Actor，
    //                  在 500 个敌人的场景中，每帧都要扫描一遍全部 Actor

    // ❌ 错误2：每帧都对所有敌人排序（500个元素排序很慢）
    AllEnemies.Sort([](const AActor& A, const AActor& B) {
        float DistA = FVector::DistSquared(GetActorLocation(), A.GetActorLocation());
        float DistB = FVector::DistSquared(GetActorLocation(), B.GetActorLocation());
        return DistA < DistB;
    });

    // ❌ 错误3：每帧都做碰撞检测（物理计算非常昂贵）
    FHitResult Hit;
    FCollisionQueryParams Params;
    GetWorld()->SweepSingleByChannel(Hit, Start, End, FQuat::Identity,
        ECC_Pawn, FCollisionShape::MakeSphere(1000.0f), Params);
}

// ---------- ✅ 正确示范：缓存结果，降低更新频率 ----------
// ---------- AMyCharacter.h ----------
UCLASS()
class MYPROJECT_API AMyCharacter : public ACharacter
{
    GENERATED_BODY()

protected:
    virtual void BeginPlay() override;
    virtual void Tick(float DeltaTime) override;

private:
    // 缓存：存储"最近的敌人"查询结果
    // 我们不需要每帧重新查找，可以每隔一段时间刷新一次缓存
    UPROPERTY()
    TArray<AActor*> CachedEnemyList;  // 缓存的敌人列表

    float LastEnemyUpdateTime = 0.0f;  // 上次更新敌人列表的时间
    float EnemyUpdateInterval = 0.5f;   // 更新间隔（0.5秒更新一次）

    AActor* CachedNearestEnemy = nullptr;  // 缓存的最近敌人
    float CachedNearestDistance = FLT_MAX;  // 缓存的最近距离（初始化为最大值）
};

// ---------- AMyCharacter.cpp ----------
void AMyCharacter::BeginPlay()
{
    Super::BeginPlay();

    // 设置 Tick 间隔 —— 既然只是查敌人，不需要每帧都查
    PrimaryActorTick.TickInterval = 0.1f;  // 每 0.1 秒 Tick 一次（每秒10次）
}

void AMyCharacter::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    // 获取从游戏开始到现在的时间（秒）
    float CurrentTime = GetWorld()->GetTimeSeconds();

    // ✅ 只在到达更新间隔时才刷新敌人列表
    if (CurrentTime - LastEnemyUpdateTime >= EnemyUpdateInterval)
    {
        // 更新缓存
        UGameplayStatics::GetAllActorsOfClass(GetWorld(), AEnemy::StaticClass(), CachedEnemyList);
        // ↑ 现在每 0.5 秒才调用一次，而不是每帧

        // 找出最近的敌人（也只在这里计算一次）
        CachedNearestEnemy = nullptr;
        CachedNearestDistance = FLT_MAX;
        FVector MyLocation = GetActorLocation();

        for (AActor* Enemy : CachedEnemyList)
        {
            float Dist = FVector::DistSquared(MyLocation, Enemy->GetActorLocation());
            if (Dist < CachedNearestDistance)
            {
                CachedNearestDistance = Dist;
                CachedNearestEnemy = Enemy;
            }
        }

        LastEnemyUpdateTime = CurrentTime;  // 记录更新时间
    }

    // ✅ Tick 主体只使用缓存的结果，不做重复计算
    if (CachedNearestEnemy)
    {
        // 直接使用缓存的最近敌人来旋转角色面向敌人
        FVector Direction = CachedNearestEnemy->GetActorLocation() - GetActorLocation();
        Direction.Z = 0.0f;  // 忽略高度差
        Direction.Normalize();  // 归一化，只保留方向信息
        SetActorRotation(Direction.Rotation());  // 面向敌人
    }
}
```

> 💡 **缓存原则**：如果一段代码的结果在同一帧或相邻几帧内不会变化，就把结果缓存起来，不要重复计算。

### 1.3 对象池替代频繁 Spawn/Destroy

在 UE 中，`SpawnActor`（生成）和 `Destroy`（销毁）是非常耗时的操作，因为它们涉及内存分配/释放、组件注册、物理系统通知等一系列复杂流程。如果你的游戏需要频繁创建和销毁对象（如子弹、粒子、敌人），应该使用**对象池**模式。

```cpp
// ---------- 理解为什么 Spawn/Destroy 很慢 ----------
// 在 Tick 中频繁 Spawn 和 Destroy 的典型坏例子：
void AMachineGun::Fire()
{
    // ❌ 每次开火都 Spawn 一颗新子弹
    ABullet* NewBullet = GetWorld()->SpawnActor<ABullet>(
        ABullet::StaticClass(),  // 要生成的类
        MuzzleLocation,          // 生成位置
        MuzzleRotation           // 生成旋转
    );

    // 这个过程涉及：
    // 1. 在堆上分配内存（malloc/new）
    // 2. 调用构造函数
    // 3. 注册所有组件到世界
    // 4. 初始化物理代理
    // 5. 通知所有相关系统"有新Actor生成了"
    // ... 单次看很快，但每秒 600 发子弹就是巨大的开销

    // 然后子弹飞了 2 秒击中墙壁后：
    // bullet->Destroy();
    // 销毁过程涉及：
    // 1. 从世界注销所有组件
    // 2. 清理物理代理
    // 3. 等待垃圾回收
    // 4. 释放内存
}

// ---------- 对象池的核心思想 ----------
// 与其反复"生成→销毁→生成→销毁"，
// 不如一次性生成足够多的子弹，用完后"藏起来"（禁用），
// 下次需要时再"拿出来"（启用）。
//
// 比喻：
// ❌ 反复 Spawn/Destroy = 每天都买新水杯，用完就扔
// ✅ 对象池          = 买一打水杯，用完洗干净放回柜子，下次再用

// 具体的对象池实现放在本文件夹的 03-章节案例.md 中
// 请参考那里关于 TPooledObject 模板类的详细实现
```

✅ **适合对象池的项目**：子弹、粒子效果、敌人(波次模式)、UI提示文字、音效播放器
❌ **不适合对象池的项目**：玩家角色、BOSS、关卡机关（数量少、生成不频繁）

### 1.4 合理使用距离剔除（Cull Distance）

距离剔除是一种"远的就别管了"的优化思想：离相机很远的 Actor，玩家根本看不见或看不清，可以降低更新频率甚至完全不更新。

```cpp
// ---------- AMyDistantEnemy.h ----------
UCLASS()
class MYPROJECT_API AMyDistantEnemy : public AActor
{
    GENERATED_BODY()

public:
    AMyDistantEnemy()
    {
        PrimaryActorTick.bCanEverTick = true;
    }

    // ✅ 使用 Net Cull Distance（网络剔除距离）
    // 虽然名字里有 "Net"，但单机游戏也能用
    // 引擎会自动管理：当 Actor 离玩家超过这个距离，停止网络同步和Tick

    UPROPERTY(EditAnywhere, Category = "优化")
    float NetCullDistanceSquared = 25000000.0f;
    //                             └默认 2500万 单位平方（约 5000 单位的距离）
    //                              距离 = sqrt(25000000) ≈ 5000 单位

    // 自定义距离检查函数
    bool ShouldUpdateThisFrame() const;
};

// ---------- AMyDistantEnemy.cpp ----------
bool AMyDistantEnemy::ShouldUpdateThisFrame() const
{
    // 获取玩家控制器
    APlayerController* PC = GetWorld()->GetFirstPlayerController();
    if (!PC)
    {
        return true;  // 没有玩家控制器，默认更新
    }

    // 获取玩家控制的 Pawn（角色）
    APawn* PlayerPawn = PC->GetPawn();
    if (!PlayerPawn)
    {
        return true;
    }

    // 计算敌人与玩家之间距离的平方
    // 使用 DistSquared 而不是 Dist，因为平方根运算比较慢
    float DistSquared = FVector::DistSquared(
        GetActorLocation(),        // 自己的位置
        PlayerPawn->GetActorLocation()  // 玩家的位置
    );

    // 近处的敌人：每帧更新
    // 远处的敌人：跳过更新
    return DistSquared <= NetCullDistanceSquared;
}

void AMyDistantEnemy::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    // ✅ 开局就检查，不需要更新就直接返回
    // 虽然 Tick 仍然被调用了，但逻辑几乎不执行
    if (!ShouldUpdateThisFrame())
    {
        return;  // 太远了，跳过所有逻辑
    }

    // 以下是正常的 Tick 逻辑
    UpdateAI();        // 更新 AI 决策
    UpdateAnimation(); // 更新动画
    CheckForPlayer();  // 检测玩家
}

// ✅ 更激进的优化：在 Tick 的开销主要在函数调用时
// 可以直接关闭 Tick，用 Timer 和距离判断结合：
void AMyDistantEnemy::BeginPlay()
{
    Super::BeginPlay();

    // 根据到玩家的距离动态调整 Tick 间隔
    float Distance = FVector::Dist(GetActorLocation(),
        GetWorld()->GetFirstPlayerController()->GetPawn()->GetActorLocation());

    if (Distance < 2000.0f)
    {
        PrimaryActorTick.TickInterval = 0.0f;   // 近处：每帧更新
    }
    else if (Distance < 5000.0f)
    {
        PrimaryActorTick.TickInterval = 0.1f;   // 中距离：每秒 10 次
    }
    else if (Distance < 10000.0f)
    {
        PrimaryActorTick.TickInterval = 0.5f;   // 远处：每秒 2 次
    }
    else
    {
        // 超远处：完全关闭 Tick，只在玩家靠近时重新开启
        SetActorTickEnabled(false);
    }
}
```

### 1.5 事件驱动替代轮询

**轮询** = 每帧都问一遍"有事吗？"，大多数时候回答是"没有"。
**事件驱动** = 有事的时候通知你，没事就不打扰。

```cpp
// ---------- ❌ 轮询模式（Polling）：每帧都检查 ----------
void AMyDoor::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    // ❌ 每帧都要检查玩家是否在范围内
    // 即使玩家在大老远，也要做一次距离计算
    APlayerController* PC = GetWorld()->GetFirstPlayerController();
    if (PC && PC->GetPawn())
    {
        float Dist = FVector::Dist(GetActorLocation(), PC->GetPawn()->GetActorLocation());
        if (Dist < 200.0f)
        {
            OpenDoor();  // 玩家靠近了，开门
        }
    }
    // 问题：99% 的时间玩家都不在门旁边，
    // 但门每帧都要计算一次距离 —— 白白浪费 CPU
}

// ---------- ✅ 事件驱动模式（Event-Driven）：用碰撞检测替代 ----------
// 步骤1：给门加一个碰撞盒子（Box Collision 或 Sphere Collision）
// 步骤2：在蓝图中把这个盒子设为 Overlap 触发
// 步骤3：绑定碰撞事件

// ---------- AMyDoor.h ----------
UCLASS()
class MYPROJECT_API AMyDoor : public AActor
{
    GENERATED_BODY()

public:
    AMyDoor()
    {
        // ✅ 门的 Tick 完全可以关闭
        PrimaryActorTick.bCanEverTick = false;
        // 门不需要每帧检查 —— 等有人碰到它时再处理
    }

protected:
    // 碰撞盒子组件 —— 用来检测玩家接近
    UPROPERTY(VisibleAnywhere)
    class UBoxComponent* TriggerBox;

    // 门网格体
    UPROPERTY(VisibleAnywhere)
    class UStaticMeshComponent* DoorMesh;

    // 碰撞开始时的回调（玩家进入门范围时触发）
    UFUNCTION()
    void OnPlayerEnterRange(
        UPrimitiveComponent* OverlappedComponent,  // 自己的碰撞组件
        AActor* OtherActor,                         // 碰到门的那个Actor
        UPrimitiveComponent* OtherComp,             // 那个Actor的碰撞组件
        int32 OtherBodyIndex,                       // 碰撞体索引
        bool bFromSweep,                            // 是否是扫描碰撞
        const FHitResult& SweepResult              // 碰撞详细信息
    );

    // 碰撞结束时的回调（玩家离开门范围时触发）
    UFUNCTION()
    void OnPlayerLeaveRange(
        UPrimitiveComponent* OverlappedComponent,
        AActor* OtherActor,
        UPrimitiveComponent* OtherComp,
        int32 OtherBodyIndex
    );
};

// ---------- AMyDoor.cpp ----------
#include "AMyDoor.h"
#include "Components/BoxComponent.h"
#include "Components/StaticMeshComponent.h"

void AMyDoor::BeginPlay()
{
    Super::BeginPlay();

    // 绑定碰撞事件 —— 这就是"事件驱动"的关键！
    // 编译器会自动处理碰撞检测，不需要你手动在 Tick 里计算距离
    if (TriggerBox)
    {
        // 绑定：当有人进入碰撞盒子时 → 自动调用 OnPlayerEnterRange
        TriggerBox->OnComponentBeginOverlap.AddDynamic(
            this,                    // 接收者（this = 这个门）
            &AMyDoor::OnPlayerEnterRange  // 要调用的函数
        );

        // 绑定：当有人离开碰撞盒子时 → 自动调用 OnPlayerLeaveRange
        TriggerBox->OnComponentEndOverlap.AddDynamic(
            this,
            &AMyDoor::OnPlayerLeaveRange
        );
    }
    // 一行 Tick 代码都不需要！完全靠引擎的事件系统驱动
}

void AMyDoor::OnPlayerEnterRange(
    UPrimitiveComponent* OverlappedComponent,
    AActor* OtherActor,
    UPrimitiveComponent* OtherComp,
    int32 OtherBodyIndex,
    bool bFromSweep,
    const FHitResult& SweepResult)
{
    // 只有当玩家真正进入范围时，这个函数才会被调用
    // 不是每帧检查，而是"事件发生时"才调用

    // 检查碰到的是不是玩家角色
    if (OtherActor && OtherActor->ActorHasTag(FName("Player")))
    {
        UE_LOG(LogTemp, Log, TEXT("玩家靠近了门，开门！"));
        // 开门动画/逻辑...
    }
}

void AMyDoor::OnPlayerLeaveRange(
    UPrimitiveComponent* OverlappedComponent,
    AActor* OtherActor,
    UPrimitiveComponent* OtherComp,
    int32 OtherBodyIndex)
{
    if (OtherActor && OtherActor->ActorHasTag(FName("Player")))
    {
        UE_LOG(LogTemp, Log, TEXT("玩家离开了门，关门！"));
        // 关门动画/逻辑...
    }
}
```

**轮询 vs 事件驱动对比：**

| 特性       | 轮询（Polling）    | 事件驱动（Event-Driven）    |
| ---------- | ------------------ | --------------------------- |
| CPU 开销   | 每帧都执行         | 只在事件发生时执行          |
| 响应速度   | 取决于轮询间隔     | 即时响应                    |
| 代码复杂度 | 简单               | 稍复杂（需要设置碰撞/绑定） |
| 适用场景   | 必须每帧检测的场景 | 大多数"检测触发"的场景      |

> 💡 **经验法则**：能用碰撞/触发/委托解决的，就不要用 Tick 轮询。

### 1.6 虚拟函数调用的开销

```cpp
// ---------- 理解虚函数调用的额外开销 ----------

// 假设有这样的类层次结构：
class ABaseEnemy : public AActor
{
public:
    // 虚函数：子类可以重写这个行为
    virtual void Attack()
    {
        // 默认攻击逻辑
    }

    // 非虚函数：子类不能重写，调用更快
    void Die()
    {
        // 统一的死亡逻辑
    }
};

class AMeleeEnemy : public ABaseEnemy
{
public:
    // 重写虚函数
    virtual void Attack() override
    {
        // 近战攻击逻辑
    }
};

// ---------- 虚函数为什么有额外开销？----------
// 调用非虚函数 Die() 时：
//     编译器直接知道要调用哪个函数 → 直接跳转过去 → 1 步完成
//
// 调用虚函数 Attack() 时：
//     1. 通过对象指针找到"虚函数表"（vtable）
//     2. 在虚函数表中查找 Attack 的地址
//     3. 跳到正确的地址执行
//     → 比非虚函数多了两次内存读取（查表操作）

// ---------- 实际建议 ----------

// ✅ 可以忽略虚函数开销的场景：
//    - 每帧只调用几次的函数
//    - 事件响应函数（碰撞、输入等）
//    - 初始化函数（BeginPlay等）
//    这些场景下虚函数开销可以忽略不计

// ❌ 需要注意虚函数开销的场景（热路径）：
//    - 每帧被调用数千次的函数
//    - 在大量循环中调用的函数
//    - 性能分析显示为瓶颈的函数
//    在这些"热路径"上，考虑用非虚函数或模板替代

// 示例：如果有一个函数在 Tick 中对 1000 个敌人循环调用：
void ProcessAllEnemies(TArray<ABaseEnemy*>& Enemies)
{
    for (ABaseEnemy* Enemy : Enemies)
    {
        // 如果 Attack 是虚函数，每次循环都要查虚函数表
        // 1000 个敌人 = 1000 次虚表查找
        Enemy->Attack();  // ← 虚函数调用（如果只有几种子类还好，太多就慢）
    }
}
```

> 💡 **关键认知**：虚函数开销对新手来说通常不是首要问题。先关注算法复杂度、Spawn/Destroy、Tick 频率这些大问题。只有在 Unreal Insights 明确显示虚函数调用是瓶颈时，再考虑优化它。过早优化是万恶之源。

## 2. 内存优化

内存优化不像帧率那样容易被感知（内存不足通常表现为卡顿或崩溃而非持续的帧率下降），但同样至关重要。

### 2.1 TArray 预分配（Reserve）

```cpp
// ---------- TArray 的动态扩容机制 ----------
// TArray 在添加元素时，如果内部缓冲区满了，会自动扩容
// 但扩容操作涉及：分配新内存 → 拷贝旧数据 → 释放旧内存
// 如果频繁扩容，会有大量"分配-拷贝-释放"的开销

// ---------- ❌ 错误做法：不预分配 ----------
void BadExample()
{
    TArray<int32> Numbers;
    // 此时 Numbers 的内部缓冲区大小是 0

    // 循环添加 10000 个元素
    for (int32 i = 0; i < 10000; ++i)
    {
        Numbers.Add(i);
        // 当内部缓冲区满了时，TArray 会：
        //   第1次扩容：容量 0 → 4
        //   第2次扩容：容量 4 → 16
        //   第3次扩容：容量 16 → 64
        //   第4次扩容：容量 64 → 256
        //   第5次扩容：容量 256 → 1024
        //   第6次扩容：容量 1024 → 4096
        //   第7次扩容：容量 4096 → 16384
        // 共扩容了约 log2(10000) ≈ 14 次
        // 每次扩容都要"分配新内存 + 拷贝旧数据 + 释放旧内存"
    }
}

// ---------- ✅ 正确做法：预分配 ----------
void GoodExample()
{
    TArray<int32> Numbers;

    // ✅ 提前告知 TArray 我们需要多少空间
    // Reserve 一次性分配足够的内存，避免后续反复扩容
    Numbers.Reserve(10000);
    //      └一次性分配 10000 个元素的空间
    //       后续 Add 操作不再触发扩容

    for (int32 i = 0; i < 10000; ++i)
    {
        Numbers.Add(i);  // 不会扩容，直接写入已分配的内存
    }
}

// ---------- Reserve vs Resize vs SetNum ----------
// 这三个函数容易混淆，理解它们的区别很重要：

// Reserve(N)：预留空间，但不改变元素数量
TArray<int32> Arr;
Arr.Reserve(100);  // 内部分配了 100 个位置的内存
// Arr.Num() == 0   ← 元素数量还是 0！
// 后续 Add 时不会扩容，直接使用预留的空间

// SetNum(N)：直接设置元素数量
TArray<int32> Arr2;
Arr2.SetNum(100);   // 数组现在有 100 个元素（默认值是0）
// Arr2.Num() == 100  ← 元素数量就是 100
// Arr2[0] ~ Arr2[99] 都可以直接访问

// Resize(N)：类似 SetNum，但可以选择是否保留旧数据
// 通常在需要清空数组但又想保留内存时使用
Arr.Reset();       // 清空元素，但保留内存（Num = 0，Max不变）
Arr.Empty();       // 清空元素，释放内存（Num = 0，Max = 0）
```

### 2.2 避免不必要的拷贝（传 const 引用）

```cpp
// ---------- 值传递 vs 引用传递 ----------

// 声明一个较重的结构体（包含大量数据）
struct FPlayerData
{
    FString Name;            // 玩家名称（字符串，可能很长）
    TArray<FString> Items;   // 背包物品列表（可能有很多个）
    TMap<FName, int32> Stats; // 属性表（力量、敏捷、智力等）
    FVector Position;         // 位置
    FRotator Rotation;        // 旋转
};
// FPlayerData 包含字符串和数组，拷贝一次会很贵（需要深度拷贝所有内部数据）

// ---------- ❌ 值传递：每次调用都拷贝整个结构体 ----------
void PrintPlayerData_Bad(FPlayerData Data)
//                        └没有 &，表示"值传递"
//                         调用时会把整个 FPlayerData 拷贝一份给 Data
{
    // Data 是原始数据的一个完整副本
    // 拷贝 FPlayerData 意味着：
    // 1. 拷贝 Name（分配新内存，复制字符串内容）
    // 2. 拷贝 Items（分配新内存，复制整个数组和所有字符串）
    // 3. 拷贝 Stats（分配新内存，复制整个 Map）
    // 4. 拷贝 Position 和 Rotation（12+12=24字节，忽略不计）
    // 总开销：大量内存分配和字符串复制！
    UE_LOG(LogTemp, Log, TEXT("玩家: %s"), *Data.Name);
}

// ---------- ✅ const 引用传递：零拷贝 ----------
void PrintPlayerData_Good(const FPlayerData& Data)
//                        └const引用，表示"借来看，不拷贝，也不能修改"
//                         调用时只传递一个指针（8字节），零拷贝开销
{
    // Data 是原始数据的一个"只读视图"
    // 没有发生任何拷贝！直接访问原始数据
    UE_LOG(LogTemp, Log, TEXT("玩家: %s"), *Data.Name);
}

// ---------- 传递规则速查表 ----------
//                          用这个类型        示例
// 基本类型(int,float,bool)：值传递           void Foo(int X)
// 小结构体(FVector)：       值传递也可       void Foo(FVector V)
// 大结构体/类：             const引用        void Foo(const FBigStruct& S)
// 需要修改原值：            引用              void Foo(FBigStruct& S) ← 注意没有const
// 字符串(FString)：         const引用        void Foo(const FString& S)
// TArray/TMap：             const引用        void Foo(const TArray<int>& A)

// ---------- ❌ 常见陷阱：返回局部引用 ----------
// const FString& GetName_Bad()
// {
//     FString LocalName = TEXT("局部变量");
//     return LocalName;  // ❌ 危险！返回的是局部变量的引用
//                        // 函数返回后 LocalName 被销毁，引用指向已释放的内存！
// }

// ✅ 返回大对象时，直接返回值（编译器会优化，不会实际拷贝）
FString GetName_Good()
{
    FString LocalName = TEXT("局部变量");
    return LocalName;  // ✅ 安全！编译器会用"返回值优化"(RVO)免去拷贝
}
```

### 2.3 使用 FName 而非 FString 做标识符

```cpp
// ---------- FName vs FString vs FText ----------

// FString：通用的可变字符串，类似 std::string
// - 每次创建都需要分配堆内存
// - 比较两个 FString 需要逐字符对比（O(n) 复杂度）
// - 适合：日志输出、文件路径、可变的文本内容

// FName：不可变的哈希字符串标识符，类似内嵌的"ID号"
// - 相同的字符串在全局只存一份（字符串驻留/String Interning）
// - 比较两个 FName 只比较它们的哈希索引（O(1) 复杂度）
// - 适合：资源路径、属性名、标签、键值

// FText：用于向玩家显示的本地化文本
// - 支持翻译和多语言
// - 适合：UI文字、对话、任何玩家能看到的文本

// ---------- 实际使用对比 ----------
void NameVsString()
{
    // ❌ 用 FString 做大量查找比较（慢）
    TMap<FString, int32> ItemDatabase_String;  // 键是 FString
    FString Key1 = FString(TEXT("Sword_Of_Flame"));
    FString Key2 = FString(TEXT("Sword_Of_Flame"));
    // Key1 和 Key2 是两个不同的内存块，比较它们需要逐字符对比

    // ✅ 用 FName 做查找比较（快）
    TMap<FName, int32> ItemDatabase_Name;  // 键是 FName
    FName Key3 = FName(TEXT("Sword_Of_Flame"));
    FName Key4 = FName(TEXT("Sword_Of_Flame"));
    // Key3 和 Key4 指向全局表中的同一个条目
    // 比较它们只需比较一个整数索引，快得多！
    // 这只是第一次从字符串创建 FName 时的开销，之后用相同的字符串获取 FName 是很快的
}

// ---------- 使用建议 ----------
// ✅ 用 FName 的场景：
//    - TMap 的键值
//    - Actor 标签 (Tags)
//    - 资源引用路径
//    - 属性名
//    - Socket 名称

// ✅ 用 FString 的场景：
//    - 需要频繁修改的字符串
//    - 需要子串操作的字符串
//    - 格式化的日志输出

// ✅ 用 FText 的场景：
//    - UI 文本
//    - 对话文本
//    - 任何需要本地化的文本
```

### 2.4 关卡流式加载（World Composition / Level Streaming）

如果一个关卡有 10 平方公里的地图和 10000 个物体，全部一次性加载会占用大量内存，而且加载时间很长。关卡流式加载解决了这个问题：只加载玩家附近的区域，走远了的区域自动卸载。

```cpp
// ---------- 手工流式加载子关卡 ----------

#include "Kismet/GameplayStatics.h"  // UGameplayStatics 在这里
#include "Engine/LevelStreamingDynamic.h"  // 动态流式加载

void AMyLevelManager::LoadNearbyLevel(const FName& LevelName)
{
    // ✅ 异步加载子关卡（不阻塞游戏主线程）
    // 关卡在后台加载，加载完成后自动出现在世界中
    UGameplayStatics::LoadStreamLevel(
        this,          // 世界上下文对象（任意 Actor 或 Component）
        LevelName,     // 关卡名称（在 Levels 窗口中设置的名称）
        true,          // bMakeVisibleAfterLoad：加载后是否立即可见
        false          // bShouldBlockOnLoad：是否阻塞游戏直到加载完成（false=异步）
    );

    UE_LOG(LogTemp, Log, TEXT("正在后台加载子关卡: %s"), *LevelName.ToString());
}

void AMyLevelManager::UnloadFarLevel(const FName& LevelName)
{
    // ✅ 卸载远处的子关卡，释放内存
    UGameplayStatics::UnloadStreamLevel(
        this,          // 世界上下文对象
        LevelName,     // 要卸载的关卡名称
        FLatentActionInfo(), // 延迟动作信息（通常留空）
        false          // bShouldBlockOnUnload：是否阻塞
    );
}

// ---------- 流式加载的设置步骤（编辑器操作）----------
// 1. 在主关卡中，打开 Window → Levels 面板
// 2. 点击 Levels 下拉菜单 → Add Existing... 添加子关卡
// 3. 在 Levels 面板中，右键子关卡 → 选择 Streaming Method
//    - Always Loaded：永远加载（小关卡用这个）
//    - Blueprint：由蓝图或C++代码控制加载/卸载（大地图推荐）
// 4. 在C++代码中调用 LoadStreamLevel / UnloadStreamLevel

// ---------- 流式加载的性能收益 ----------
// 假设一个开放世界有 10 个子关卡，每个 500MB：
// 全部加载：500MB × 10 = 5GB 内存占用 ❌
// 只加载周围的 3 个：500MB × 3 = 1.5GB 内存占用 ✅
// 节省了 3.5GB 内存！
```

### 2.5 异步加载资产（FStreamableManager）

```cpp
// ---------- 理解同步加载 vs 异步加载 ----------

// ❌ 同步加载：LoadObject，会阻塞游戏线程直到资产加载完成
void BadLoadAsset()
{

    UStaticMesh* HeavyMesh = LoadObject<UStaticMesh>(

// ---------- ✅ 异步加载：FStreamableManager ----------
// ---------- AMyAssetLoader.h ----------
#pragma once
#include "CoreMinimal.h"
#include "GameFramework/Actor.h"
#include "Engine/StreamableManager.h"  // FStreamableManager 在这里
#include "AMyAssetLoader.generated.h"

UCLASS()
class MYPROJECT_API AMyAssetLoader : public AActor
{
    GENERATED_BODY()

public:
    AMyAssetLoader()
    {
        PrimaryActorTick.bCanEverTick = false;
    }

    // 异步加载 Boss 模型
    void LoadBossModelAsync();

    // 加载完成后的回调函数
    void OnBossModelLoaded();

protected:
    // 流式管理器 —— 管理所有异步加载请求
    // Shared 版本在多个对象间共享内部缓存，更省内存
    FStreamableManager StreamableManager;
    //                  └不需要初始化，声明即用

    // 引用已加载的网格体（UPROPERTY防止被垃圾回收）
    UPROPERTY()
    class UStaticMesh* LoadedBossMesh;

    // 共享指针，持有异步加载请求的句柄
    TSharedPtr<FStreamableHandle> LoadHandle;
};

// ---------- AMyAssetLoader.cpp ----------
#include "AMyAssetLoader.h"
#include "Engine/StaticMesh.h"

void AMyAssetLoader::LoadBossModelAsync()
{
    // 步骤1：构造要加载的资产路径
    // FSoftObjectPath 是资产的"软引用"路径，不会阻止资产被垃圾回收
    FSoftObjectPath MeshPath(
        TEXT("/Game/Art/Characters/Boss/BossMesh.BossMesh")
        //     └包路径────────────────────────┘ └资产名──┘
    );
    // 包路径和资产名用 . 分隔，通常资产名和包的最后一个部分相同

    // 步骤2：发起异步加载请求
    // RequestAsyncLoad：在后台线程加载资产，不阻塞游戏
    LoadHandle = StreamableManager.RequestAsyncLoad(
        MeshPath,                                    // 要加载的资产路径
        FStreamableDelegate::CreateUObject(           // 加载完成后的回调
            this,                                     // 回调所属对象
            &AMyAssetLoader::OnBossModelLoaded       // 回调函数
        )
    );
    // 此时游戏继续正常运行，加载在后台线程中进行

    UE_LOG(LogTemp, Log, TEXT("Boss模型正在后台异步加载，游戏不卡顿..."));
}

void AMyAssetLoader::OnBossModelLoaded()
{
    // 步骤3：加载完成！获取已加载的资产
    // 这个函数在资产加载完成后由引擎自动调用

    // 方式1：使用 LoadSynchronous 在回调中获取（此时已加载完毕，实际是同步的）
    // 因为回调触发时资产已经在内存中，所以不会阻塞
    LoadedBossMesh = LoadObject<UStaticMesh>(
        nullptr,
        TEXT("/Game/Art/Characters/Boss/BossMesh.BossMesh")
    );

    if (LoadedBossMesh)
    {
        UE_LOG(LogTemp, Log, TEXT("Boss模型加载成功！"));
        // 把模型应用到某个组件上...
    }
    else
    {
        UE_LOG(LogTemp, Error, TEXT("Boss模型加载失败，请检查文件路径"));
    }

    // 加载完成后释放句柄
    LoadHandle.Reset();
}
```

> 💡 **关键理解**：异步加载 = 游戏不卡顿的前提下在后台加载资产。适合加载大模型、大纹理、音频文件等大型资产。

## 3. 渲染优化

### 3.1 LOD 系统（Level of Detail —— 细节层次）

LOD 的核心思想：近处的物体用高精度模型（高面数），远处的物体用低精度模型（低面数），因为远处的物体在屏幕上只占几个像素，高面数模型纯粹是浪费 GPU。

```cpp
// ---------- LOD 在编辑器中的设置（主要操作在编辑器，C++辅助）----------

// 步骤1：导入多个精度的模型
// 例如一面墙（Wall）：
//   Wall_LOD0.fbx  →  2000 个面（高精度，用于近距离）
//   Wall_LOD1.fbx  →  800 个面（中精度，用于中距离）
//   Wall_LOD2.fbx  →  100 个面（低精度，用于远距离）

// 步骤2：在静态网格编辑器中设置 LOD
// 1. 打开静态网格体编辑器（双击 Wall_LOD0）
// 2. 在 LOD Settings 中添加 LOD 层级
// 3. 为每个 LOD 层级指定对应的网格体
// 4. 设置每层 LOD 的切换距离（Screen Size）

// 步骤3：在 C++ 中动态控制 LOD
void AMyLODController::ForceLODOnActor(AActor* TargetActor, int32 LODLevel)
{
    if (!TargetActor) return;

    // 获取 Actor 上的所有 StaticMeshComponent
    TArray<UStaticMeshComponent*> MeshComponents;
    TargetActor->GetComponents<UStaticMeshComponent>(MeshComponents);
    //                      └模板函数：只获取指定类型的组件
    //                       返回所有 StaticMeshComponent（可能有多个）

    for (UStaticMeshComponent* MeshComp : MeshComponents)
    {
        // ✅ 强制使用指定的 LOD 级别
        // ForcedLodModel = 0 表示使用 LOD0（最高精度）
        // ForcedLodModel = 1 表示使用 LOD1（中等精度）
        // ForcedLodModel = 2 表示使用 LOD2（最低精度）
        MeshComp->ForcedLodModel = LODLevel;
        //         └设置为 0, 1, 2... 强制使用对应的LOD级别
        //          设置为 0 表示不强制（引擎自动选择合适的LOD）
    }

    UE_LOG(LogTemp, Log, TEXT("已强制 %s 使用 LOD%d"), *TargetActor->GetName(), LODLevel);
}

// ---------- LOD 的典型设置参数 ----------
// Screen Size（屏幕占比）：决定 LOD 切换的距离
// Screen Size 1.0 = 物体占满整个屏幕时用 LOD0
// Screen Size 0.5 = 物体占半屏时切换到 LOD1
// Screen Size 0.25 = 物体占1/4屏时切换到 LOD2
// 数值越小 = 切换距离越远

// ❌ 常见错误：所有物体都用同一套 LOD 设置
// ✅ 正确做法：大型建筑切换距离远一些，小装饰物切换距离近一些
```

### 3.2 合批（Instanced Static Mesh —— ISM）

Draw Call 是 UE 中最常见的渲染瓶颈。每画一个独立的 StaticMeshComponent 就是一次 Draw Call，CPU 需要为每次 Draw Call 准备渲染状态，1000 次 Draw Call 的开销远超 1 次。

```cpp
// ---------- ❌ 错误做法：为每个物体创建独立的 Actor ----------
// 场景中种 1000 棵树
void BadTreeSpawning()
{
    for (int32 i = 0; i < 1000; ++i)
    {
        // 每棵树都是独立的 Actor，各有各的 StaticMeshComponent
        // 1000 棵树 = 1000 次 Draw Call
        // CPU 要向 GPU 发送 1000 次"画一棵树"的指令
        AActor* Tree = GetWorld()->SpawnActor<AActor>(
            AActor::StaticClass(),
            FVector(FMath::RandRange(-1000, 1000), FMath::RandRange(-1000, 1000), 0),
            FRotator::ZeroRotator
        );
        // 给每个 Actor 挂一个 StaticMeshComponent
        UStaticMeshComponent* MeshComp = NewObject<UStaticMeshComponent>(Tree);
        MeshComp->SetStaticMesh(TreeMesh);
        MeshComp->RegisterComponent();
        Tree->SetRootComponent(MeshComp);
    }
}

// ---------- ✅ 正确做法：使用 ISM 将所有树合并为 1 次 Draw Call ----------
// ---------- AForestManager.h ----------
UCLASS()
class MYPROJECT_API AForestManager : public AActor
{
    GENERATED_BODY()

public:
    AForestManager()
    {
        PrimaryActorTick.bCanEverTick = false;

        // 创建一个 InstancedStaticMeshComponent 作为根组件
        // ISM 组件的工作方式：
        //   你添加多个"实例"到这个组件里，每个实例有自己的 Transform
        //   GPU 一次性画完所有实例 —— 只需要 1 次 Draw Call！
        TreeISM = CreateDefaultSubobject<UInstancedStaticMeshComponent>(TEXT("TreeISM"));
        //                   └这就是 ISM 组件，它可以画同一网格体的多个实例
        SetRootComponent(TreeISM);  // 设为根组件
    }

    // 生成整个森林
    UFUNCTION(BlueprintCallable, Category = "森林")
    void GenerateForest(int32 TreeCount, float AreaRadius);

protected:
    UPROPERTY(VisibleAnywhere)
    class UInstancedStaticMeshComponent* TreeISM;

    // 要在蓝图中设置的树木模型
    UPROPERTY(EditAnywhere, Category = "森林")
    class UStaticMesh* TreeMesh;
};

// ---------- AForestManager.cpp ----------
#include "AForestManager.h"
#include "Components/InstancedStaticMeshComponent.h"

void AForestManager::GenerateForest(int32 TreeCount, float AreaRadius)
{
    if (!TreeMesh || !TreeISM)
    {
        UE_LOG(LogTemp, Error, TEXT("TreeMesh 或 TreeISM 为空！请在编辑器中设置"));
        return;
    }

    // 步骤1：设置 ISM 使用哪种网格体
    TreeISM->SetStaticMesh(TreeMesh);
    //                    └所有这1000棵树都用同一个模型
    //                     但每棵树可以有独立的 Transform

    // 步骤2：循环添加实例
    for (int32 i = 0; i < TreeCount; ++i)
    {
        // 随机位置（在指定半径的圆内）
        FVector RandomLocation(
            FMath::RandRange(-AreaRadius, AreaRadius),  // X 随机
            FMath::RandRange(-AreaRadius, AreaRadius),  // Y 随机
            0.0f                                         // Z = 地面
        );

        // 随机旋转（让每棵树看起来朝向不同）
        FRotator RandomRotation(0.0f, FMath::RandRange(0.0f, 360.0f), 0.0f);

        // 随机缩放（模拟大小不同的树）
        float RandomScale = FMath::RandRange(0.8f, 1.2f);

        // 构造 Transform
        FTransform InstanceTransform(
            RandomRotation,                    // 旋转
            RandomLocation,                    // 位置
            FVector(RandomScale, RandomScale, RandomScale)  // 缩放（XYZ统一缩放）
        );

        // ✅ 添加一个实例 —— 不创建新Actor，只记录Transform！
        TreeISM->AddInstance(InstanceTransform);
        //      └这个操作非常轻量，只是往数组里加一个 Transform
        //       不会创建 Actor，不会创建新的 Component
        //       最终 GPU 一次性画出所有实例
        //       1000 棵树 = 1 次 Draw Call！
    }

    UE_LOG(LogTemp, Log, TEXT("森林已生成：%d 棵树，只需 1 次 Draw Call！"), TreeCount);
}

// ---------- ISM 的性能对比 ----------
// 1000 棵独立的树（每个一个 StaticMeshComponent）：
//   Draw Calls: 1,000
//   GameThread: ~15ms（CPU在准备渲染数据时忙不过来）
//
// 1000 棵 ISM 树（一个 InstancedStaticMeshComponent）：
//   Draw Calls: 1
//   GameThread: ~0.5ms
//
// 性能提升：约 30 倍！
```

> ⚠️ **ISM 的局限**：
>
> - 所有实例**共用同一个材质**（不能每棵树用不同的材质）
> - 不支持单个实例的 LOD（所有实例使用相同的 LOD 级别）
> - 不支持逐实例的碰撞（碰撞需要另外处理）
> - 如果实例需要各自不同的逻辑（如被砍倒的树），不适合用 ISM

遮挡剔除的意思是：被墙壁挡住的物体，根本不需要画。

    //   ☑ Cast Hidden Shadow（投射隐藏阴影，即使是看不见的面也投射阴影）

        // Window → Place Actors → Volumes → Cull Distance Volume
        // Cull Distance Volume 会覆盖场景中的一个小物件的绘制距离
    }
}

### 3.4 阴影优化

阴影是最常见的 GPU 瓶颈之一。每个投射阴影的光源，每个投射阴影的物体，都在消耗 GPU 时间。

```cpp
// ---------- 阴影优化的几个关键设置 ----------

// 1. 减少投射阴影的动态光源数量
// ❌ 场景中有 20 个 Point Light，每个都投射动态阴影
//    每个 Point Light 的阴影需要单独渲染一次 → 20 次阴影渲染
// ✅ 只在主方向光和少数关键点光源上启用阴影投射

// 2. 对于不投射动态阴影的光源，设置为 Static（烘焙）或 Stationary（混合）
//    编辑器中选择光源 → Mobility：
//    - Static：完全烘焙，零运行时开销，运行时不改动
//    - Stationary：半烘焙，零阴影开销，颜色/强度可运行时调整
//    - Movable：全动态，阴影开销大，光源可自由移动

// 3. 降低阴影精度
// 在光源的详情面板中：
//    Dynamic Shadow Distance：动态阴影的可见距离（越远越模糊，但开销也大）
//    Num Dynamic Shadow Cascades：级联阴影贴图数量（方向光，通常3-4级）
//    Shadow Resolution：阴影贴图分辨率（默认1024或2048）

// ---------- 在 C++ 中动态控制阴影 ----------
void OptimizeShadowForActor(AActor* TargetActor)
{
    TArray<UStaticMeshComponent*> MeshComps;
    TargetActor->GetComponents<UStaticMeshComponent>(MeshComps);

    for (UStaticMeshComponent* MeshComp : MeshComps)
    {
        // ✅ 关闭该物体的阴影投射
        // 小装饰物（如小石头、草）通常不需要投射阴影
        MeshComp->SetCastShadow(false);
        //                    └false = 不投射阴影（节省GPU）
        //                     true  = 投射阴影（默认）

        // ✅ 远处的阴影精度自动降低（需在引擎/项目设置中配置）
        // 这是在项目层面的设置，不需要在 C++ 里逐物体设置
    }
}
```

## 4. 蓝图优化

蓝图的优点是开发速度快，代价是运行速度慢。在某些关键计算上，蓝图比 C++ 慢 10-100 倍。

### 4.1 为什么蓝图比 C++ 慢？

蓝图运行在虚拟机（VM）上，类似 Java 的 JVM。每个蓝图节点都是一个独立的操作：

```
C++ 执行: a = b + c * d
→ 编译为 3 条 CPU 指令，直接执行

蓝图执行: a = b + c * d
→ 需要：
  1. 从 VM 栈读取变量 c（1 次内存访问）
  2. 从 VM 栈读取变量 d（1 次内存访问）
  3. 执行 Multiply 节点（VM 函数调用）
  4. 从 VM 栈读取变量 b（1 次内存访问）
  5. 执行 Add 节点（VM 函数调用）
  6. 将结果写入变量 a（1 次内存访问）
  → 每个节点都是一个函数调用，而 C++ 中一切都是内联的
```

### 4.2 复杂计算放 C++

```cpp

// ❌ 蓝图方案：用蓝图节点实现 A* 寻路
//    每个敌人寻路需要约 200 个蓝图节点
//    500 个敌人 × 200 个节点 = 100,000 次 VM 函数调用
//    耗时约 45ms（灾难性的！）

// ✅ C++ 方案：把寻路逻辑写在 C++ 中，暴露给蓝图调用

// ---------- UPathfindingLibrary.h ----------
#pragma once

#include "Kismet/BlueprintFunctionLibrary.h"
#include "UPathfindingLibrary.generated.h"

// 蓝图函数库：包含可被蓝图调用的纯 C++ 静态函数
UCLASS()
class MYPROJECT_API UPathfindingLibrary : public UBlueprintFunctionLibrary
{
    GENERATED_BODY()

public:
    // BlueprintCallable：这个函数可以从蓝图中调用
    // 函数的具体实现在 C++ 中（执行速度快）
    // 蓝图只需要调用它，不需要自己实现寻路逻辑
    UFUNCTION(BlueprintCallable, Category = "寻路")
    static TArray<FVector> FindPath(
        const FVector& StartLocation,  // 起始位置
        const FVector& EndLocation,    // 目标位置
        bool bAllowPartialPath         // 是否允许不完整的路径
    );
};

// ---------- UPathfindingLibrary.cpp ----------
#include "UPathfindingLibrary.h"
#include "NavigationSystem.h"  // UNavigationSystemV1 在这里

TArray<FVector> UPathfindingLibrary::FindPath(
    const FVector& StartLocation,
    const FVector& EndLocation,
    bool bAllowPartialPath)
{
    // 这个函数全部在 C++ 中执行，速度是蓝图的 50-100 倍

    TArray<FVector> PathPoints;  // 结果路径点数组

    // 获取导航系统（全局单例）
    UNavigationSystemV1* NavSys = UNavigationSystemV1::GetCurrent(GetWorld());
    if (!NavSys)
    {
        UE_LOG(LogTemp, Error, TEXT("FindPath：找不到导航系统！"));
        return PathPoints;  // 返回空数组
    }

    // 使用 UE 自带的寻路系统进行路径查找
    FPathFindingQuery Query;
    // ... 配置寻路查询（实际代码省略以保持简洁）...

    // 在 C++ 中执行路径查找 —— 纯引擎代码，无 VM 开销
    FPathFindingResult Result = NavSys->FindPathSync(Query);

    if (Result.IsSuccessful())
    {
        // 提取路径点
        for (const FNavPathPoint& Point : Result.Path->GetPathPoints())
        {
            PathPoints.Add(Point.Location);
        }
    }

    return PathPoints;
}
```

### 4.3 蓝图 Nativization（蓝图转 C++）

蓝图 Nativization 是 UE 的一个功能，能把蓝图脚本自动翻译成 C++ 代码并编译，从而获得接近 C++ 的性能。

**操作步骤：**

1. `编辑（Edit）` → `项目设置（Project Settings）`
2. 左侧导航找 `打包（Packaging）` 分类
3. 找到 `Blueprint Nativization Method`（蓝图本地化方法）
4. 设置为 `Inclusive（包含式）` 或 `Exclusive（排他式）`
   - Inclusive：把所有蓝图都转成 C++（编译时间长，但全部优化）
   - Exclusive：只转你手动标记的蓝图（更灵活，推荐）
5. 如果选 Exclusive，需要在你想要优化的蓝图上：
   - 打开蓝图类 → `类设置（Class Settings）` → 勾选 `Nativize（本地化）`

> ⚠️ **注意事项**：
>
> - Nativization 只在**打包（Cooked Build）**时生效，编辑器 PIE 不生效
> - 不是所有蓝图都能安全地 Nativize，尤其是动态委托和反射使用较多的蓝图可能出错
> - 编译时间会变长（需要把蓝图翻译成C++再编译）
> - 考虑打包时间：Nativization 会增加打包时间

## 5. 资产优化

资产优化是最容易被新手忽视的性能维度 —— 一个 4096x4096 的贴图看似没什么，但 100 个这样的贴图就是灾难。

### 5.1 贴图压缩和大小

在 UE 中，贴图通常是最大的内存消耗者之一。

**编辑器操作（主要在贴图编辑器中进行，而非 C++）：**

| 贴图用途               | 推荐分辨率           | 压缩格式                  | 说明                 |
| ---------------------- | -------------------- | ------------------------- | -------------------- |
| 角色皮肤（主角）       | 2048x2048            | BC7（桌面）/ ASTC（移动） | 主角近看，可高分辨率 |
| 角色皮肤（NPC）        | 1024x1024            | BC7 / ASTC                | NPC 通常不需要太高   |
| 环境贴图（地面、墙壁） | 1024x1024 或 512x512 | BC1 或 BC7                | 大多数情况够用       |
| 小装饰物               | 512x512 或 256x256   | BC1                       | 很小的物体           |
| UI 图标                | 跟实际显示大小匹配   | 看需求                    | UI 需要像素级清晰    |

**操作步骤：**

1. 在 Content Browser 中双击贴图资源
2. 在贴图编辑器中查看 `Compression` 设置
3. 调整 `Maximum Texture Size`（最大贴图尺寸）
4. 选择合适的压缩格式

**压缩格式对比：**

| 格式       | 每像素占用 | 质量              | 适用平台          |
| ---------- | ---------- | ----------------- | ----------------- |
| BC1 (DXT1) | 0.5 字节   | 有损，不支持Alpha | 通用（桌面+主机） |
| BC3 (DXT5) | 1 字节     | 有损，支持Alpha   | 通用              |
| BC7        | 1 字节     | 高质量            | 桌面（DX11+）     |
| ASTC       | 可变       | 可调              | 移动平台          |

```
一个 2048x2048 BC7 贴图的内存占用：
  2048 × 2048 × 1 字节 = 4 MB
  （加上 Mipmap 大约 5-6 MB）

100 个这样的贴图 = 500+ MB！
所以在保证质量的前提下，尽量降低贴图分辨率和选择合适的压缩格式
```

### 5.2 模型面数控制

| 物体类型              | 推荐三角面数范围 | 说明                       |
| --------------------- | ---------------- | -------------------------- |
| 玩家角色              | 20,000 - 60,000  | 主角，需要高精度           |
| 敌人角色              | 5,000 - 20,000   | 根据重要程度调整           |
| 大型建筑              | 1,000 - 5,000    | 依赖纹理和法线贴图表现细节 |
| 小型道具              | 100 - 500        | 箱子、木桶等               |
| 植被（单棵）          | 50 - 500         | 通常做法面片+Alpha贴图     |
| 远景物体（LOD2/LOD3） | 10 - 50          | 低面数代理模型             |

### 5.3 音频压缩

音频文件在 UE 中通常以 WAV 格式导入，但在打包时会被压缩。

**编辑器操作：**

1. 在 Content Browser 中双击音频资源（Sound Wave）
2. 查看 `Compression` 部分
3. 选择合适的压缩格式

| 音频类型               | 推荐格式              | 说明                 |
| ---------------------- | --------------------- | -------------------- |
| 短音效（枪声、脚步声） | PCM（不压缩）或 ADPCM | 短音效不占多少空间   |
| 背景音乐               | Ogg Vorbis / ADPCM    | 压缩率高，质量可接受 |
| 语音对话               | Ogg Vorbis 高压缩     | 语音不需要高保真     |

> 💡 音频优化的主力战场在编辑器资产设置中，C++ 侧一般不需要手动管理音频压缩。
