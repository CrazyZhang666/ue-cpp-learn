## 1. Unreal Insights —— 最核心的性能分析工具

### 1.1 什么是 Unreal Insights？

Unreal Insights 是 Epic 官方出品的性能分析工具，类似给游戏做"全身 CT 扫描"。它能记录每一帧里 CPU 和 GPU 到底在忙什么，帮你找到最耗时的代码。

你可以把 Unreal Insights 理解为一台"游戏性能录像机"：

- **录制**：运行游戏时记录下每一帧的所有操作
- **回放**：用 Insights 桌面程序打开录制的文件进行逐帧分析

### 1.2 如何启用（命令行 -trace 参数）

Unreal Insights 通过"Trace Channel"来决定记录哪些数据。启用方式取决于你的运行环境。

#### 方式 A：编辑器内启用（最常用）

**操作步骤：**

1. 打开你的 UE5 项目
2. 点击顶部菜单栏的 `工具（Tools）` → `会话前端（Session Frontend）`
3. 在打开的窗口中选择 `Profiler` 标签页
4. 点击 `Start` 按钮开始录制
5. 点击 `Stop` 按钮停止录制，trace 文件会自动保存

**也可以用控制台命令：**

```
Trace.Start    // 开始录制
Trace.Stop     // 停止录制
```

在编辑器中按 `~`（波浪号键）打开控制台，输入上面两个命令即可。

#### 方式 B：命令行参数启动（适合打包后的独立游戏）

当你双击 `.exe` 运行游戏时，可以通过命令行参数自动开启录制：

```bash
# 最常用的 trace 启动命令
MyProject.exe -trace=cpu,gpu,frame,log,bookmark

# 参数含义：
# cpu      - 记录CPU上所有函数的耗时（最重要）
# gpu      - 记录GPU上所有渲染命令的耗时
# frame    - 记录帧级别的信息（帧率、帧号）
# log      - 记录游戏日志与帧的对应关系
# bookmark - 记录你在代码中手动标记的时间点
```

**操作步骤：**

1. 找到你的打包后 `.exe` 文件位置（例如 `F:\MyGame\Windows\MyProject.exe`）
2. 在 `.exe` 同级目录下创建一个文本文件
3. 将文件重命名为 `启动并录制.bat`
4. 右键编辑，输入以下内容：

```batch
@echo off
echo 正在启动游戏并开启性能录制...
MyProject.exe -trace=cpu,gpu,frame,log,bookmark
pause
```

5. 双击运行这个 `.bat` 文件即可

#### 常用的 Trace Channel 说明

| Channel 名称 | 记录内容               | 性能开销 | 推荐程度          |
| ------------ | ---------------------- | -------- | ----------------- |
| `cpu`        | CPU 上所有函数的计时   | 中等     | ⭐⭐⭐⭐⭐ 必开   |
| `gpu`        | GPU 上的渲染指令耗时   | 中等     | ⭐⭐⭐⭐⭐ 必开   |
| `frame`      | 帧号、帧率等基础帧信息 | 很低     | ⭐⭐⭐⭐⭐ 必开   |
| `log`        | 游戏日志与帧的绑定关系 | 很低     | ⭐⭐⭐⭐ 推荐     |
| `bookmark`   | 代码中自定义的性能标记 | 很低     | ⭐⭐⭐⭐ 推荐     |
| `loadtime`   | 资产加载时间           | 低       | ⭐⭐⭐ 按需       |
| `memory`     | 内存分配记录           | 高       | ⭐⭐ 调试内存时开 |
| `file`       | 文件 I/O 操作记录      | 低       | ⭐⭐ 按需         |

> ⚠️ **注意**：`memory` channel 的性能开销很高，会明显降低游戏帧率。只在排查内存问题时才开启。

✅ **正确做法**：日常优化使用 `-trace=cpu,gpu,frame,log`
❌ **错误做法**：无脑全开所有 channel，导致录制出的数据不准确

### 1.3 如何录制 Trace

录制 trace 有三种方式：

#### 方式 1：编辑器内录制（最方便）

```
# 在编辑器控制台（按 ~ 键打开）中输入：
Trace.Start    # 开始录制，会在右下角显示"Recording"字样

# 操作你的游戏，复现你要分析的卡顿场景

Trace.Stop     # 停止录制
```

录制完成后，UE5 会自动弹出一个对话框，告诉你 trace 文件保存的位置。默认路径是：

```
你的项目目录/Saved/Profiling/UnrealInsights/
```

文件名类似：`YourProject-2024.06.15-14.30.45.utrace`

#### 方式 2：使用 UnrealTrace 命令行工具录制（适合测试打包版本）

```bash
# 先启动游戏（不指定 -trace 参数）
MyProject.exe

# 另开一个命令行，使用 UnrealTrace 工具连接到运行中的游戏进行录制
# UnrealTrace.exe 位于引擎安装目录下
# 例如：E:\UE_5.3\Engine\Binaries\Win64\UnrealTrace.exe

UnrealTrace.exe start    # 开始录制
UnrealTrace.exe stop     # 停止录制，自动保存 .utrace 文件
```

#### 方式 3：DevOps/自动化测试录制

```bash
# 在自动化测试脚本中用命令行参数
MyProject.exe -trace=cpu,gpu,frame -ExecCmds="Automation RunTests MyTest; Quit"
# -ExecCmds 参数指定运行完测试后自动退出
```

### 1.4 如何分析 Trace 文件

#### 步骤 1：启动 Unreal Insights 桌面程序

Unreal Insights 是一个独立的桌面应用程序，不在编辑器内部。

**找到它：**

```
你的引擎目录/Engine/Binaries/Win64/UnrealInsights.exe

# 例如：
E:\UE_5.3\Engine\Binaries\Win64\UnrealInsights.exe
```

✅ **建议**：把这个 `.exe` 发送到桌面快捷方式，以后双击就能打开。

#### 步骤 2：打开 Trace 文件

1. 双击 `UnrealInsights.exe` 启动程序
2. 点击顶部菜单 `File` → `Open...`
3. 导航到你的 `.utrace` 文件位置（如 `项目目录/Saved/Profiling/UnrealInsights/`）
4. 选中 `.utrace` 文件，点击打开

或者直接把 `.utrace` 文件拖到 Unreal Insights 窗口中。

#### 步骤 3：等待解析

稍等几秒，Unreal Insights 会解析文件并展示一个时间线视图。如果你录制了较长的时间（如几分钟），解析可能需要十几秒。

### 1.5 CPU Timeline 分析

这是最核心的分析视图，能让你看到每一帧、每一个函数花了多少时间。

#### 界面布局说明

打开 trace 文件后，你会看到：

```
┌─────────────────────────────────────────────────────┐
│  工具栏（顶部）  [放大] [缩小] [选择范围] [导出]      │
├─────────────────────────────────────────────────────┤
│  帧时间线（上方）- 绿色竖线代表每一帧的开始和结束      │
│  ┃ ┃ ┃ ┃  ████████████████  ███████  █████████████  │
├─────────────────────────────────────────────────────┤
│  CPU 线程视图（主体）                                 │
│  ┌─ GameThread ──────────────────────────────────┐  │
│  │ █████ Tick() ██ Physics ████ Render ████████ │  │
│  └───────────────────────────────────────────────┘  │
│  ┌─ RenderThread ────────────────────────────────┐  │
│  │ █████ InitViews ████ BasePass ████████████   │  │
│  └───────────────────────────────────────────────┘  │
│  ┌─ TaskGraph NP0 ───────────────────────────────┐  │
│  │ ██ Worker █████ Worker ██ Worker ██████████  │  │
│  └───────────────────────────────────────────────┘  │
├─────────────────────────────────────────────────────┤
│  详情面板（底部）- 选中某个函数后显示详细信息         │
│  函数名: AActor::Tick        耗时: 15.234 ms        │
│  调用次数: 1000              占比: 42%              │
└─────────────────────────────────────────────────────┘
```

#### 如何找到性能瓶颈

**操作步骤（跟着一步步来）：**

1. **找到一帧特别长的帧**
   - 在帧时间线上，看到绿色的竖条有高有低
   - 找到一个特别高的竖条（说明这一帧特别慢）
   - 鼠标左键点击这个位置

2. **放大查看**
   - 滚动鼠标滚轮放大时间线
   - 或者用工具栏的放大镜图标
   - 一直放大到能看到单个函数调用的程度

3. **识别最耗时的函数**
   - 在 GameThread（游戏线程）上，看哪些色块最宽（宽度 = 耗时）
   - 点击最宽的色块
   - 底部详情面板会显示函数名和具体耗时

4. **记录问题**
   - 记录下：函数名、单次耗时、这一帧内被调用了多少次
   - 例如："AActor::Tick 耗时 15ms，单帧被调用了 1000 次"

#### 解读示例

假设你看到这样一段 GameThread 时间线：

```
帧耗时: 52ms（正常是 16.67ms，严重超了！）
┌──────────────────────────────────────────────────────┐
│ GameThread                                            │
│ ██████ WorldTick ██ NetworkTick ████ TickActors ████████████████████████████ │
│  3ms             1ms           2ms        46ms ← 罪魁祸首！
│                                                      │
│ 展开 TickActors：                                       │
│ █ AActor::Tick █ AMyEnemy::Tick ██████████████████████████████████████ │
│   1ms             1ms              44ms ← 98%的时间都在这里！
│                                                      │
│ 展开 AMyEnemy::Tick：                                   │
│ █ MoveTo █ FindTarget ██ UpdateAnimation ████████████████████████████ │
│   1ms      2ms          5ms              36ms ← 动画更新是瓶颈！
└──────────────────────────────────────────────────────┘
```

**分析结论**：

- 瓶颈：`AMyEnemy::Tick` 中的 `UpdateAnimation` 函数耗时 36ms
- 原因：可能有 500 个敌人同时在做复杂的动画计算
- 优化方向：减少动画更新频率、使用 LOD 降低远处敌人的动画精度

### 1.6 GPU Timeline 分析

GPU Timeline 与 CPU Timeline 类似，但展示的是 GPU 上的工作。

#### 关键线程

在 GPU 视图中，你会看到两个重要线程：

| 线程             | 含义                | 说明                                               |
| ---------------- | ------------------- | -------------------------------------------------- |
| **RenderThread** | 渲染线程（在CPU上） | 准备渲染命令，生成 Draw Call。CPU 侧的渲染准备工作 |
| **RHIThread**    | RHI 线程（在CPU上） | 将渲染命令提交给 GPU 驱动。CPU 和 GPU 之间的桥梁   |
| **GPU 0**        | GPU 本身的执行      | 真正在显卡上执行渲染工作的时间                     |

> 💡 **关键理解**：RenderThread 和 GPU 是**并行工作**的。RenderThread 准备第 3 帧时，GPU 可能在渲染第 2 帧。这就是为什么 GPU 瓶颈和 CPU 瓶颈是两种完全不同的问题。

#### GPU Timeline 常见色块含义

| 色块名称         | 含义                           | 如果它很宽说明什么                     |
| ---------------- | ------------------------------ | -------------------------------------- |
| `BasePass`       | 基础渲染通道，画所有不透明物体 | 场景物体太多或材质太复杂               |
| `ShadowDepths`   | 渲染阴影贴图                   | 阴影设置太高，投射阴影的光源和物体太多 |
| `Translucency`   | 渲染半透明物体                 | 半透明物体太多或覆盖面积太大           |
| `PostProcessing` | 后处理效果（Bloom、DOF等）     | 后处理设置太激进                       |
| `Velocities`     | 运动矢量计算（TAA抗锯齿需要）  | 运动中的物体太多                       |
| `Lights`         | 光照计算（Deferred Rendering） | 动态光源太多或影响范围太大             |

#### 分析示例

```
GPU 帧耗时: 42ms（目标 16.67ms，严重超了！）
┌──────────────────────────────────────────────────────┐
│ GPU 0                                                │
│ █ BasePass ████ ShadowDepths ██████████████████████ █ PostProcess │
│   8ms        24ms ← 阴影占了57%的时间！         4ms   │
│                                                      │
│ 展开 ShadowDepths：                                    │
│ █ DirectionalLight ████ PointLight_01 ████ PointLight_02 ██... │
│   4ms                5ms                5ms            │
│   （一共20个点光源，每个都要单独画一遍阴影！）                │
└──────────────────────────────────────────────────────┘
```

**分析结论**：

- 瓶颈：`ShadowDepths` 占了 24ms（57% 的总 GPU 时间）
- 原因：场景里 20 个动态点光源全部投射动态阴影
- 优化方向：减少投射阴影的光源数量，或使用静态/烘焙光照替代动态光照

### 1.7 帧分析（Frame Analysis）

帧分析让你跳脱宏观视角，深入观察**某一帧**的所有细节。

#### 操作步骤

1. 在时间线上找到性能异常的那一帧（最高的一根绿条）
2. 双击这帧，进入"帧分析"视图
3. 你会看到这一帧的完整调用树，类似一个文件夹结构：

```
Frame 1547 (54.23 ms)                     ← 总耗时
├── GameThread (48.12 ms)                  ← 游戏线程
│   ├── UWorld::Tick (45.80 ms)            ← 世界更新
│   │   ├── FTickTaskManager::RunTickGroup (38.20 ms)  ← Actor Tick
│   │   │   ├── AActor::Tick (34.50 ms)   ← 所有Actor的Tick
│   │   │   │   ├── AMyEnemy::Tick (28.00 ms)  ← 敌人的Tick
│   │   │   │   │   ├── FindTarget (15.00 ms)  ← 寻找目标
│   │   │   │   │   └── MoveTo (12.00 ms)      ← 移动逻辑
│   │   │   │   └── AMyBullet::Tick (6.00 ms)  ← 子弹的Tick
│   │   │   └── FTimerManager::Tick (2.00 ms)  ← 定时器
│   │   └── UWorld::SendAllEndOfFrameUpdates (1.50 ms)
│   └── FEngineLoop::Tick (2.00 ms)
├── RenderThread (3.50 ms)
│   └── FSceneRenderer::Render (3.20 ms)
└── GPU 0 (42.00 ms)                       ← GPU 花了 42ms
    ├── BasePass (10.00 ms)
    ├── ShadowDepths (24.00 ms)            ← GPU 瓶颈在此
    └── PostProcessing (5.00 ms)
```

从这个树形结构中，你可以清晰看到：

- 这一帧总共 54.23ms（目标 16.67ms，超了 3 倍多）
- 游戏线程（CPU）花了 48ms，其中 `AMyEnemy::Tick` 是罪魁祸首
- GPU 花了 42ms，其中 `ShadowDepths` 是瓶颈
- CPU 和 GPU 都有问题，需要双管齐下优化

> 💡 **经验法则**：
>
> - 如果 `GameThread > 16.67ms` 而 `GPU < 16.67ms`：你的 CPU 是瓶颈
> - 如果 `GPU > 16.67ms` 而 `GameThread < 16.67ms`：你的 GPU 是瓶颈
> - 如果两者都超过 16.67ms：两者都需要优化

## 2. Stat 命令 —— 游戏内实时统计

Stat 命令是 UE 中最方便的性能监控工具，直接在游戏运行中输入控制台命令就能实时看到性能数据。

### 2.1 stat fps —— 显示帧率

这是最简单的命令，显示当前帧率。

```
stat fps
```

**效果**：屏幕右上角会显示实时的 FPS 数字。

```
FPS: 45.2
```

- 60 FPS = 每帧 16.67ms（目标帧率）
- 30 FPS = 每帧 33.33ms（可接受的底线，再低就明显卡顿了）
- 15 FPS = 每帧 66.67ms（严重卡顿，必须优化）

✅ **正确用法**：在 PIE（Play In Editor）或独立运行时使用
❌ **错误用法**：只在编辑器不运行时看 FPS — 编辑器的额外开销会让 FPS 偏低

### 2.2 stat unit —— 帧耗时分解（最重要！）

```
stat unit
```

**效果**：屏幕右侧显示详细的帧耗时分解：

```
Frame: 54.23 ms        ← 这一帧总共花了多少时间
Game:  42.15 ms        ← 游戏线程（CPU逻辑）花费的时间
Draw:   3.50 ms        ← 渲染线程（CPU提交渲染命令）花费的时间
GPU:   38.00 ms        ← GPU 实际渲染花费的时间
```

**如何解读 stat unit 的结果：**

假设你看到：

```
Frame: 54.23 ms     （目标 16.67ms，不达标！）
Game:  42.15 ms     ← 最长！CPU 游戏逻辑是瓶颈
Draw:   3.50 ms     ← 正常
GPU:   38.00 ms     ← 也长！GPU 也有瓶颈
```

**结论**：你的游戏 CPU 和 GPU 都有瓶颈，但 CPU 问题（Game 42ms）比 GPU 问题（GPU 38ms）更严重一些。优先优化 CPU。

另一种情况：

```
Frame: 48.00 ms
Game:   5.00 ms     ← 正常
Draw:   4.00 ms     ← 正常
GPU:   44.00 ms     ← 极长！GPU 是唯一瓶颈
```

**结论**：CPU 很快，所有时间都在等 GPU 完成渲染。你需要降低画质、减少Draw Call、优化Shader。

> 💡 **关键认知**：`Frame = max(Game, GPU) + 少量开销`。最终帧时间取决于最慢的那个环节。如果 Game 线程等 GPU 完成，Game 时间也会膨胀（因为它在等）。所以最准确的方式是看谁"主动"耗时最长。

✅ **正确做法**：每次做性能分析首先敲 `stat unit`
❌ **错误做法**：只看 `stat fps` 就下结论。FPS 告诉你"有问题"，stat unit 告诉你"哪里有问题"

### 2.3 stat game —— 游戏线程详细统计

```
stat game
```

**效果**：显示游戏线程上各个子系统的耗时明细：

```
                    Count   Inclusive    Exclusive
Tick Time                      42.15ms       ---
  Tick Actors        1500       38.00ms    36.00ms   ← Actor Tick 占了绝大部分时间
  Timer Manager        50        2.00ms     1.50ms
  Physics              32        1.80ms     1.80ms
  Network              10        0.35ms     0.35ms
```

**列解释：**

- `Count`：调用次数。1500 说明场景里有 1500 个 Actor 每帧都在 Tick
- `Inclusive`：包含此函数内部调用子函数的总时间
- `Exclusive`：只计入此函数自身的代码，不包含它调用的子函数

**如何分析：**

- 如果 `Tick Actors` 的 `Count` 很大（比如 1000+），说明场景里 Actor 太多了
- 如果 `Inclusive` 和 `Exclusive` 差距很大（比如 `Inclusive = 38ms`，`Exclusive = 2ms`），说明函数本身不慢，但它调用的子函数慢，需要展开分析

### 2.4 stat gpu —— GPU 统计

```
stat gpu
```

**效果**：显示 GPU 上各个渲染阶段的耗时：

```
                                          ms
BasePass                                8.50       ← 基础渲染
Shadows                                24.00       ← 阴影，注意这个数字！
  ShadowDepths                         22.00
Translucency                            2.00
Post Processing                         5.00
  Bloom                                 1.50
  DOF                                   1.00
  Motion Blur                           0.80
Lights                                  3.00
Velocities                              1.20
```

**如何分析：**

- 找到耗时最高的条目
- 如果 `Shadows` 很高 → 减少动态阴影、降低阴影精度
- 如果 `BasePass` 很高 → 减少 Draw Call、简化材质
- 如果 `Post Processing` 很高 → 降低后处理质量设置
- 如果 `Translucency` 很高 → 减少半透明物体的数量和覆盖面积

### 2.5 stat scenerendering —— 渲染统计

```

**效果**：显示渲染相关的详细信息，特别是 Draw Call 数量：

```

**关键指标解读：**

| 指标                | 桌面平台合理值  | 移动平台合理值 | 说明                         |
| ------------------- | --------------- | -------------- | ---------------------------- |
| Mesh Draw Calls     | < 3000          | < 500          | 每次 Draw Call 都有 CPU 开销 |

| Visible Static Mesh | 无硬限制，看ISM | < 200          | ISM 可以合并 Draw Call       |

> 💡 **Draw Call 是游戏性能中最常见的瓶颈之一**。每画一个物体就需要一次 Draw Call。1000 个独立的墙壁，每个都需要一次 Draw Call = 1000 次。但如果用 Instanced Static Mesh（ISM）打包在一起，就只需要 1 次 Draw Call。

## 5. UE_LOG 使用技巧

UE_LOG 是 UE 中的日志宏，类似 `printf` 或 `cout`，但更强大，支持日志分类和详细级别。

### 5.1 UE_LOG 基本语法

```cpp
// UE_LOG 的基本语法（每一部分都有注释说明）
// UE_LOG(日志分类, 详细级别, TEXT("格式字符串"), 参数...);

// 示例：输出一条警告级别的日志
UE_LOG(LogTemp, Warning, TEXT("玩家生命值: %d"), CurrentHealth);
//     └─LogTemp ─┘ └Warning─┘ └────文本内容─────────┘ └──参数──┘
//       分类标签    级别颜色       格式化文本            替换%d的值

// 更完整的示例 —— 在 Actor 的 BeginPlay 中：
void AMyActor::BeginPlay()
{
    Super::BeginPlay();

    // 普通日志：输出到控制台和日志文件，显示为灰色
    UE_LOG(LogTemp, Log, TEXT("AMyActor::BeginPlay - 角色初始化开始"));

    // 警告日志：显示为黄色，表示需要注意但不致命的问题
    UE_LOG(LogTemp, Warning, TEXT("角色 %s 的血量很低: %d"), *GetName(), CurrentHealth);

    // 错误日志：显示为红色，表示出错
    if (CurrentHealth <= 0)
    {
        UE_LOG(LogTemp, Error, TEXT("初始化失败：角色血量为 %d，这不正常！"), CurrentHealth);
    }
}
```

### 5.2 Verbosity 级别（详细程度）

Verbosity 从低到高排列：

| 级别          | 含义     | 颜色 | 何时使用                                               |
| ------------- | -------- | ---- | ------------------------------------------------------ |
| `VeryVerbose` | 极度详细 | 灰色 | 逐帧输出，只在深度调试时用。会刷屏，千万不要在正式版开 |
| `Verbose`     | 详细     | 灰色 | 详细调试信息                                           |
| `Log`         | 普通日志 | 灰色 | 正常的流程信息，如"任务已完成"                         |
| `Display`     | 显示级别 | 白色 | 比 Log 更重要的信息，默认在控制台显示                  |
| `Warning`     | 警告     | 黄色 | 可能需要关注的问题，如"内存接近上限"                   |
| `Error`       | 错误     | 红色 | 出错了，如"文件加载失败"                               |
| `Fatal`       | 致命错误 | 红色 | 程序崩溃前的最后一条日志，输出后程序会终止             |

```cpp
// ✅ 正确使用 Verbosity 的示例
// Log级别：正常的流程信息，不影响性能
UE_LOG(LogTemp, Log, TEXT("玩家 %s 进入了关卡 %s"), *PlayerName, *LevelName);

// Warning级别：潜在问题，提醒开发者关注
if (ActorCount > 5000)
{
    UE_LOG(LogTemp, Warning, TEXT("场景中Actor数量过多: %d，可能影响性能"), ActorCount);
}

// Error级别：明确的错误
if (!MyAsset)
{
    UE_LOG(LogTemp, Error, TEXT("关键资产加载失败，游戏可能无法正常运行"));
}

// ❌ 错误用法：滥用 Warning 级别
// 不要把Log级别的信息标为Warning，会导致团队"狼来了"效应
UE_LOG(LogTemp, Warning, TEXT("玩家向右移动了一步")); // 这明明是普通信息！
```

### 5.3 日志分类（Log Category）

```cpp
// 自定义日志分类 —— 在 .h 或 .cpp 文件顶部声明
// 这样你在代码中就能用 UE_LOG(LogMyGame, ...) 来输出日志
DEFINE_LOG_CATEGORY_STATIC(LogMyGame, Log, All);
//                         └分类名──┘ └默认级别──┘ └编译级别──┘
//                         自定义名字  默认展示Log及以上  编译所有级别

// 更完整的自定义示例：
// ---------- MyGameStatics.h ----------
#pragma once
#include "CoreMinimal.h"

// 声明日志分类（在头文件中）
DECLARE_LOG_CATEGORY_EXTERN(LogMyGame, Log, All);
// 使用 EXTERN 版本表示这个分类会在 .cpp 中定义

// ---------- MyGameStatics.cpp ----------
#include "MyGameStatics.h"

// 定义日志分类（在源文件中）
DEFINE_LOG_CATEGORY(LogMyGame);

// 现在可以在任何包含该头文件的地方使用
void UMyGameStatics::DoSomething()
{
    // 使用自定义分类输出日志
    UE_LOG(LogMyGame, Log, TEXT("DoSomething 被调用，参数值: %f"), SomeValue);

    // 你可以在编辑器的"输出日志"窗口中按分类过滤
    // 只显示 LogMyGame 的输出，非常方便调试
}
```

### 5.4 用 UE_LOG 辅助性能分析

```cpp
// 场景：你想知道某个函数到底执行了多长时间

void AMyActor::HeavyCalculation()
{
    // ✅ 方法1：使用 FPlatformTime（推荐，精度高）
    // 记录起始时间戳（单位：秒，但包含小数部分，精度很高）
    double StartTime = FPlatformTime::Seconds();

    // ... 你的复杂计算代码 ...
    for (int32 i = 0; i < 100000; ++i)
    {
        // 模拟复杂计算
        float Result = FMath::Sqrt(static_cast<float>(i)) * FMath::Sin(static_cast<float>(i));
    }

    // 计算耗时（秒）
    double ElapsedTime = FPlatformTime::Seconds() - StartTime;

    // 将秒转换为毫秒（1秒 = 1000毫秒），输出日志
    UE_LOG(LogTemp, Warning, TEXT("HeavyCalculation 耗时: %.3f ms"), ElapsedTime * 1000.0);
    // %.3f 表示保留3位小数，ms 表示毫秒

    // ❌ 不推荐：使用 FDateTime（精度只有1秒，不足以测量毫秒级操作）
    // FDateTime Start = FDateTime::Now();
    // ...代码...
    // FDateTime End = FDateTime::Now();  // 精度不够！
}
```

### 5.5 使用 SCOPE_CYCLE_COUNTER（更专业的计时方式）

```cpp
#include "Stats/Stats.h"  // 必须先包含这个头文件

// 声明一个统计计数器（放在 .cpp 文件顶部）
DECLARE_CYCLE_STAT(TEXT("我的角色Tick耗时"), STAT_MyActorTick, STATGROUP_Game);
//                  └显示名称────────┘ └统计ID──────┘ └所属组──────┘
//                  Insights中显示的名字  代码引用的ID   在stat命令中的分组

// 在函数中使用
void AMyActor::Tick(float DeltaTime)
{
    // 这一行会在 Unreal Insights 和 stat game 中自动记录此函数的总耗时
    SCOPE_CYCLE_COUNTER(STAT_MyActorTick);

    Super::Tick(DeltaTime);

    // 函数内的所有代码时间都会被记录
    UpdateMovement(DeltaTime);   // 更新移动
    CheckForEnemies();            // 检查敌人
    UpdateAnimation(DeltaTime);   // 更新动画
}
// 不需要手动计算时间差，SCOPE_CYCLE_COUNTER 自动完成一切

// 🎯 用了 SCOPE_CYCLE_COUNTER 后，你会在这些地方看到数据：
// 1. stat game 命令输出中会出现 "STAT_MyActorTick"
// 2. Unreal Insights 的时间线上会出现 "我的角色Tick耗时" 色块
// 3. CSV Profiler 导出的数据中会包含这项统计
```

> 💡 **对比**：`UE_LOG + FPlatformTime` = 手动秒表，适合临时测量；`SCOPE_CYCLE_COUNTER` = 自动秒表，适合需要长期跟踪的指标。

## 6. Visual Studio 调试器与 UE

Visual Studio 调试器是 C++ 开发者最重要的日常工具。掌握它，你就能在代码中"暂停时间"，逐行观察程序的运行。

### 6.1 前置准备：选择合适的编译配置

在 Visual Studio 顶部工具栏中，你会看到编译配置下拉框：

```
DebugGame Editor | Win64 | YourProject
```

UE5 项目的编译配置说明：

| 配置名称               | 优化级别 | 调试符号 | 性能 | 用途                          |
| ---------------------- | -------- | -------- | ---- | ----------------------------- |
| **DebugGame Editor**   | 禁用优化 | 完整     | 很慢 | 🐛 日常开发调试代码逻辑       |
| **Development Editor** | 启用优化 | 完整     | 正常 | ⚡ 接近打包的体验，但仍可断点 |
| **Shipping**           | 全部优化 | 无       | 最快 | 🚀 最终发布的版本，不能调试   |

```cpp
// Debug 模式 vs Development 模式的代码差异示意

// 假设你写了这段代码：
void AMyActor::Tick(float DeltaTime)
{
    // Debug 模式：这行会原样执行，方便你单步跟踪
    // Development 模式：编译器可能把这行优化掉或内联展开
    float Temp = GetActorLocation().X + 100.0f;
}

// Debug模式下：
// ✅ 可以逐行跟踪每一行代码
// ✅ 可以查看所有变量的值（即使它们被优化掉也不会丢失）
// ❌ 帧率很低（因为没有任何优化）
// ❌ 不能用来测试性能 —— 测出来的结果毫无意义

// Development模式下：
// ✅ 性能接近最终打包版本
// ✅ 仍然可以打断点（但不能保证断在你想要的行）
// ❌ 有些变量可能因为优化而"不可查看"
// ❌ 当优化器重组代码后，单步跟踪可能"乱跳"
```

✅ **正确做法**：日常调试用 DebugGame，测试性能用 Development 或打包版本
❌ **错误做法**：在 DebugGame 配置下做性能测试，然后说"我的游戏太慢了"

### 6.2 断点调试（Breakpoint）

断点是调试器最基础的功能：让程序在指定行暂停，让你检查当时的状态。

**操作步骤：**

1. 在 Visual Studio 中打开你的 `.cpp` 文件
2. 在你想暂停的那一行代码的**行号左边**（灰色区域）点一下
3. 会出现一个红色圆点，这就是断点
4. 在 UE 编辑器中运行游戏（PIE）
5. 当程序执行到这一行时，Visual Studio 会自动弹出，程序暂停

```
// 假设你在第 42 行设置了断点
40: void AMyEnemy::TakeDamage(float Damage)
41: {
42:     CurrentHealth -= Damage;      // ← 红色断点在这里
43:     if (CurrentHealth <= 0)
44:     {
45:         Die();
46:     }
47: }
```

**断点暂停后你可以做的事：**

| 操作           | 快捷键             | 作用                                     |
| -------------- | ------------------ | ---------------------------------------- |
| **继续运行**   | F5                 | 程序继续执行直到下一个断点               |
| **逐过程**     | F10                | 执行当前这一行，不进入函数内部           |
| **逐语句**     | F11                | 执行当前这一行，如果是函数则进入函数内部 |
| **跳出**       | Shift+F11          | 跳出当前函数，回到调用者                 |
| **查看变量值** | 鼠标悬停           | 把鼠标放在变量名上，会弹出当前值         |
| **监视窗口**   | 拖拽变量到监视窗口 | 持续观察变量的变化                       |

### 6.3 条件断点（Conditional Breakpoint）

当某个操作在一帧内被执行了 500 次，但只有第 347 次有问题时，普通断点会让你手抽筋。条件断点可以解决这个问题。

**操作步骤：**

1. 先设置一个普通断点（红色圆点）
2. 右键点击红色圆点 → 选择 `条件（Conditions）`
3. 在弹出的窗口中输入条件表达式

```cpp
// 示例：只在第 100 个敌人受到伤害时暂停
void AMyEnemy::TakeDamage(float Damage)
{
    // 条件断点写在这行：
    // 条件：EnemyIndex == 100
    //                    └只有第100号敌人受伤时才暂停
    CurrentHealth -= Damage;

    // 另一个例子：只在伤害很高时暂停
    // 条件：Damage > 100.0f
    //                   └只有超过100点伤害时才暂停

    // 更复杂的条件：
    // 条件：CurrentHealth < 50.0f && Damage > 20.0f
    //                    └生命值低于50 且 伤害大于20时才暂停
}
```

> ⚠️ 条件断点会显著降低程序运行速度，因为调试器需要每执行到这一行就评估一次条件。只在必要时使用。

### 6.4 数据断点（监视变量变化）

数据断点是最强大的调试功能之一：当**某个变量的值被修改**时，程序会自动暂停。这对于追踪"谁改了不该改的变量"非常有用。

**操作步骤：**

1. 先在普通断点处暂停程序（你需要程序运行到能访问该变量的位置）
2. 在监视窗口中添加你要监视的变量（例如 `CurrentHealth`）
3. 右键点击监视窗口中的变量 → 选择 `断点` → `数据断点（Data Breakpoint）`
4. 在弹出的窗口中设置字节数（一般默认 4 字节即可）

```cpp
// 场景：你的 CurrentHealth 莫名其妙变成了负数，但不知道是谁改的
// 使用数据断点后，无论哪行代码修改了 CurrentHealth，程序都会暂停

// 比如这段代码里，某处偷偷把血量改成了负数：
void SomeRandomFunction()
{
    // 数据断点会在这里暂停！
    CurrentHealth = -999.0f;  // ← 抓到你了！
}
```

> ⚠️ 数据断点的限制：
>
> - 每次最多设置 4 个硬件数据断点（CPU 硬件限制）
> - 监视范围通常 1-8 字节（监视 float 用 4 字节即可）
> - 会显著降低程序运行速度

### 6.5 调用堆栈分析（Call Stack）

调用堆栈告诉你"程序是怎么运行到这一行的"——从最开始的 main 函数，一层一层调用，最终到达你断点所在的行。

**操作步骤：**

当程序在断点处暂停时：

1. 在 Visual Studio 中找到 `调用堆栈（Call Stack）` 窗口
   - 如果没看到，去菜单 `调试（Debug）` → `窗口（Windows）` → `调用堆栈（Call Stack）`
2. 你会看到类似这样的层级结构：

```
MyProject.exe!AMyEnemy::TakeDamage(float Damage) Line 42    ← 你断点的位置
MyProject.exe!UDamageSystem::ApplyDamageToAll() Line 156     ← 谁调用了 TakeDamage
MyProject.exe!AMyGameMode::ProcessTurn() Line 89             ← 谁调用了 ApplyDamageToAll
MyProject.exe!AActor::Tick(float DeltaTime) Line 234        ← 溯源到 Tick
MyProject.exe!UWorld::Tick(...) Line 1024                   ← 引擎内部
MyProject.exe!FEngineLoop::Tick() Line 3456                  ← 引擎最外层
MyProject.exe!WinMain() Line 123                             ← 程序入口
[外部代码]                                                    ← Windows 系统调用
```

3. **双击任意一行**，可以跳转到对应的代码位置，查看上下文

**调用堆栈的实用技巧：**

```cpp
// 场景：你发现某个函数被错误地调用了，但不知道是从哪里调用的
void MyFunction()
{
    // 在这里打断点，然后看调用堆栈窗口
    // 调用堆栈会告诉你"谁"在什么条件下调用了这个函数
    DoSomethingWrong(); // ← 断点在这里
}

// 你会发现调用链可能是：
// UInputComponent::ProcessAction → AMyCharacter::OnFirePressed → MyFunction
// 这样你就知道是因为玩家按了开火键触发的
```

### 6.6 附加到进程（Attach to Process）

你不需要每次调试都从 VS 启动 UE 编辑器。可以先打开编辑器，再把 VS 调试器"附加"上去。

**操作步骤：**

1. 正常运行 UE5 编辑器，打开你的项目
2. 在 Visual Studio 中打开你的项目解决方案文件（`.sln`）
3. 菜单：`调试（Debug）` → `附加到进程（Attach to Process）`
4. 在进程列表中找 `UnrealEditor.exe`
5. 点击 `附加（Attach）`
6. 现在你在 VS 中设置的断点就能生效了

✅ **优点**：不需要每次调试都重新启动编辑器（省去漫长的启动时间）
❌ **缺点**：附加后编辑器会明显变慢（因为调试器在监控所有代码执行）

### 6.7 实战：完整的调试流程

假设你遇到一个 Bug：敌人的生命值在受到伤害后没有正确地更新到 UI 上。

```
你的调试流程应该是这样的：

第1步：在 TakeDamage 函数的最后一行打断点
      → 确认伤害确实被应用了（CurrentHealth 确实减少了）
      → 结果：CurrentHealth 减少了，说明逻辑没问题 ✓

第2步：在 UI 更新函数（UpdateHealthBar）处打断点
      → 确认这个函数是否被调用了
      → 结果：函数没被调用！说明是调用链断了 ✗

第3步：在调用 UpdateHealthBar 的地方打断点
      → 发现 if 条件判断写错了，条件永远为 false
      → 修复 bug：把 if (HealthChanged = true) 改成 if (HealthChanged)
      //                         ↑ 赋值！         ↑ 比较！
      //                         这是个经典的 C++ 陷阱

第4步：重新运行，验证修复
      → 伤害正常显示了 ✓
```
