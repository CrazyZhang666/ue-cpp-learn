# 05 — 打包发布

> **章节目标**
> 将地牢探索者从开发环境中的项目打包为用户可以双击运行的独立应用程序（.exe）。本章覆盖打包前的检查清单、项目优化、打包步骤、常见错误修复，以及发布到Steam/Epic商店的初步介绍。

---

## 5.1 打包前检查清单

打包之前必须逐项检查，任何一项遗漏都可能导致打包后的游戏无法运行。这个清单可以打印出来贴在显示器旁边。

### 5.1.1 资产引用检查

**使用Reference Viewer检查资产完整性**：

```
操作步骤：
1. 打开每个关卡（MainMenu, Dungeon_Floor1, Dungeon_Floor2, Dungeon_BossRoom）
2. 对关卡中的关键蓝图（BP_DungeonCharacter, BP_DungeonEnemy等）
   → 右键 → Reference Viewer
3. 检查是否有红色标记的丢失引用

Reference Viewer 图例：
  - 白色实线 = 硬引用（打包时会包含）
  - 蓝色虚线 = 软引用（按需加载）
  - 红色标记  = 丢失的引用（必须修复！）
```

✅ **正确做法**：每个关卡都打开Reference Viewer检查一遍
❌ **错误做法**：只在编辑器里跑了一遍没问题就觉得可以打包了——编辑器和打包后的环境不同！

**使用Size Map检查包体大小**：

```
1. 在 Content Browser 中右键文件夹 → Size Map
2. 检查是否有异常巨大的资产：
   - 单个纹理超过2K分辨率 → 可能导致包体膨胀
   - 未使用的测试资产 → 应该移除或设置不打包
   - 重复的纹理/模型 → 去重
```

### 5.1.2 关卡设置检查

对每个关卡，打开 World Settings（工具栏 → Settings → World Settings）：

| 检查项 | 应该设置的值 | 为什么 |
|--------|-------------|--------|
| GameMode Override | `BP_DungeonGameMode` | 如果没有GameMode，游戏不知道规则 |
| Default Pawn Class | `BP_DungeonCharacter` | 玩家出生时使用哪个角色 |
| Player Start (Actor) | 放置在关卡中 | 没有 PlayerStart，玩家不知道在哪里出生 |
| Player Controller Class | `BP_DungeonPlayerController` | 处理玩家输入 |
| HUD Class | `BP_DungeonHUD` | 显示游戏界面（血条等） |

✅ **正确做法**：每个关卡都独立检查World Settings
❌ **错误做法**：只在第一个关卡设置了GameMode，其他关卡忘记设置

### 5.1.3 输入配置检查

```
检查路径：Edit → Project Settings → Engine → Input

确认以下映射存在且正确：
□ Axis Mappings:
  - MoveForward (W: 1.0, S: -1.0)
  - MoveRight (D: 1.0, A: -1.0)
  - LookRight (Mouse X: 1.0)
  - LookUp (Mouse Y: -1.0)
  - ZoomCamera (Mouse Wheel Axis: 1.0)

□ Action Mappings:
  - Jump (Space Bar)
  - Dodge (Left Shift)
  - LightAttack (Left Mouse Button)
  - HeavyAttack (Right Mouse Button)
  - Skill1~Skill4 (Q, E, R, F)
  - Inventory (Tab)
  - Pause (Escape)
```

### 5.1.4 性能基线检查

在编辑器中运行以下控制台命令，收集性能数据：

```
按 ` 键打开控制台，输入：

stat fps          — 显示帧率
stat unit         — 显示帧时间（Frame/Game/Draw/GPU）
stat memory       — 显存和系统内存使用
stat scenerendering — 渲染统计
```

**目标性能基线**：

| 指标 | 最低目标 | 理想目标 |
|------|----------|----------|
| FPS（帧率） | 稳定30 FPS以上 | 稳定60 FPS |
| Game Thread（游戏线程） | < 16ms | < 8ms |
| Draw Calls（绘制调用） | < 3000 | < 1500 |
| GPU Time（GPU时间） | < 16ms | < 8ms |
| Memory（内存使用） | < 4GB | < 2GB |

## 5.2 项目设置优化

### 5.2.1 默认地图设置

```
路径：Edit → Project Settings → Project → Maps & Modes

设置项目：
  - Game Default Map: MainMenu （启动游戏时默认加载主菜单）
  - Editor Startup Map: Dungeon_Floor1 （编辑器中Play时加载第一层，方便测试）
  - Game Instance Class: BP_DungeonGameInstance
  - Global Default GameMode: BP_DungeonGameMode

Transition Map 设置：
  - 创建一个简单的黑屏关卡（全黑带加载图标）
  - 设为 Transition Map，这样关卡切换时有平滑过渡
```

### 5.2.2 渲染设置优化

```
路径：Edit → Project Settings → Engine → Rendering

1. Forward Shading（前向渲染）：
   □ 如果项目主要用于PC，保持 Deferred Shading（延迟渲染）
   □ 如果希望移动端兼容，开启 Forward Shading

2. 默认分辨率设置：
   - Default Resolution: 1920x1080
   - 不要写死分辨率，让玩家可以在设置中调整

3. LOD（细节层次）：
   - Static Mesh LOD Distance Scale: 1.0 （默认值）
   - 如果性能不足，可以适当降低

4. 抗锯齿：
   - Anti-Aliasing Method: TSR (Temporal Super Resolution) [UE5默认]

5. 动态分辨率：
   □ 开启 Dynamic Resolution（根据负载自动调整分辨率）
   - 最低分辨率比例: 50%
```

### 5.2.3 音频设置

```
路径：Edit → Project Settings → Engine → Audio

- Default Sound Class: Master
- 最大音频通道数: 32 （默认，可根据项目调整）
- 音频质量: 默认（如果包体太大可以降低）
```

### 5.2.4 打包专用设置

这些设置只在打包版本中生效，不影响编辑器开发环境。

```
路径：Edit → Project Settings → Project → Packaging

□ 关键设置：
  - Build Configuration: Shipping （发布版本，去除调试信息）
  - Full Rebuild: ✓ （完全重新编译）
  - Use Pak File: ✓ （将资源打包为.pak文件）
  - Include Prerequisites: ✓ （包含VC++运行库等必要组件）

□ 路径设置：
  - Staging Directory: 留空（使用默认的 Saved/StagedBuilds/）
  - 自定义输出目录: D:/Builds/DungeonCrawler/

□ 资源包含：
  - 使用 "List of maps to include in a packaged build" 来精确控制
  - 只勾选实际使用的关卡：MainMenu, Dungeon_Floor1, Dungeon_Floor2, Dungeon_BossRoom
  - 不要勾选测试关卡和空白关卡

□ 剔除未使用的资产：
  - 开启 "Share Material Shader Code"（共享材质着色器代码）
  - 开启 "Shared Material Native Libraries"
```

## 5.3 打包步骤详解

### 5.3.1 打包前准备

```
第0步：全面测试
  - 从头到尾完整玩一遍游戏
  - 测试所有功能：移动、战斗、升级、装备、存档、关卡切换
  - 记录发现的任何Bug
  - 修复Bug后重新测试一次

第1步：清理项目
  - File → Cook Content for Windows （预备烘焙，提前发现资产问题）
  - 清理未使用的资源：右键Content Browser → Fix Up Redirectors
  - 在 Content Browser 中搜索 "Redirector"，清除所有重定向器

第2步：保存所有文件
  - Ctrl+Shift+S （Save All）
  - 关闭所有未保存的蓝图编辑器窗口
```

### 5.3.2 执行打包

```
方法一：通过编辑器菜单打包
===============================

1. File → Package Project → Windows → Windows (64-bit)
   （或者选择 Build Target = Shipping 以获得最优性能）

2. 在弹出的文件夹选择对话框中，选择输出目录
   例如: D:/Builds/DungeonCrawler/

3. 编辑器底部会出现打包进度条：
   - "Building C++ code..." （编译C++代码）
   - "Cooking content..." （烘焙/转换资产）
   - "Packaging..." （打包）
   - 整个过程可能需要 10-60分钟，取决于项目大小

4. 等待完成。Output Log 会显示：
   LogPackageBuild: "PACKAGING COMPLETED SUCCESSFULLY"

5. 前往输出目录，应该能看到：
   DungeonCrawler/
   ├── DungeonCrawler.exe
   ├── Engine/
   ├── DungeonCrawler/
   │   ├── Content/
   │   │   └── Paks/
   │   │       └── DungeonCrawler-WindowsClient.pak  （所有资产都在这里）
   │   └── Binaries/
   │       └── Win64/
   └── Manifest files...
```

```
方法二：通过命令行打包（适合CI/CD自动化）
=============================================

在项目根目录打开命令行，运行：

# Windows 64-bit Shipping Build
RunUAT.bat BuildCookRun ^
  -project="D:/DungeonCrawler/DungeonCrawler.uproject" ^
  -platform=Win64 ^
  -clientconfig=Shipping ^
  -cook ^
  -stage ^
  -pak ^
  -archive ^
  -archivedirectory="D:/Builds/DungeonCrawler"
```

### 5.3.3 测试打包后的exe

```
1. 前往打包输出目录
2. 双击 DungeonCrawler.exe
3. 测试以下功能（用手柄和键鼠都测试一遍）：

   □ 游戏是否正常启动（无崩溃、无黑屏）
   □ 主菜单是否能正常操作（开始新游戏/继续/设置/退出）
   □ 进入游戏后角色是否正常生成
   □ 移动/跳跃/闪避/攻击是否流畅
   □ 敌人是否正常生成和攻击
   □ UI是否正常显示（血条/法力条/技能栏）
   □ 关卡切换是否正常（第一层 → Boss房 → 第二层）
   □ 存档/读档是否正常
   □ 设置面板是否可以调整音量/分辨率
   □ 退出游戏是否正常（不残留进程）

4. 在不同配置的电脑上测试（如果条件允许）：
   - 低配置（集成显卡）
   - 中配置（GTX 1060级别）
   - 高配置（RTX 3060+）
```

## 5.4 常见打包错误及解决

### 错误1：C++编译失败

```
错误信息：
  Error: Compilation failed. See log for details.

原因分析：
  - C++代码中有语法错误
  - Build.cs 中缺少模块依赖
  - 头文件include顺序错误

解决方法：
  1. 先在VS/Rider中编译（Development配置），确认编译通过
  2. 检查 Output Log 中的第一个错误（后续错误可能是连锁反应）
  3. 常见修复：检查 Build.cs 中是否包含了所有需要的模块
  4. 常见修复：确认 .generated.h 文件在所有include的最后
```

### 错误2：资产缺失/引用丢失

```
错误信息：
  LogLinker: Warning: Failed to load '/Game/Blueprints/BP_Enemy'
  LogLinker: Warning: Can't find file for asset '/Game/Data/DT_Monsters'

原因分析：
  - 蓝图或资源在编辑器中移动/重命名后，其他引用未更新
  - 某些资产没有被包含在打包列表中

解决方法：
  1. 对缺失资产 → 右键 → Reference Viewer → 查找谁引用了它 → 更新引用
  2. 对已被删除的资产 → 找到引用者 → 移除无效引用或替换新资产
  3. 使用 Tools → Validate Assets → 查找所有资产问题
  4. 在 Project Settings → Packaging 中确保所有需要的关卡都在打包列表中
```

### 错误3：关卡加载失败

```
错误信息：
  UEngine::Browse: Browse Failed: World '/Game/Maps/Dungeon_Floor2' not found

原因分析：
  - 关卡没有包含在打包列表中
  - 关卡的路径字符串写错了（C++中硬编码的关卡名与实际不符）

解决方法：
  1. 检查 Project Settings → Packaging → List of maps to include
  2. 确认所有需要打包的关卡都在列表中
  3. 检查 C++ 代码中的 OpenLevel 调用，确认路径带 _C 后缀
     正确: FName(TEXT("/Game/Maps/Dungeon_Floor2"))
     错误: FName(TEXT("Dungeon_Floor2"))
```

### 错误4：打包后UI显示异常

```
错误现象：
  - UMG界面控件位置错乱
  - 字体变成方块
  - 图片不显示

原因分析：
  - UI使用了 Anchors/Alignment 而不是固定位置（不同分辨率下错位）
  - 字体资产没有被打包
  - 图片资产的引用路径变了

解决方法：
  1. 所有 UMG 控件使用 Anchors（锚点）定位，不要用绝对像素位置
  2. 在 Project Settings → Packaging → Additional Asset Directories to Cook
     中添加字体文件夹
  3. 检查图片的引用路径，使用Reference Viewer确认
```

### 错误5：打包后输入无效

```
错误现象：
  - 键盘/鼠标操作无响应
  - 手柄无法使用

原因分析：
  - Plugin 未启用（如 Windows 输入插件）
  - 增强输入（EnhancedInput）的 InputAction 资产未被引用

解决方法：
  1. 检查 Edit → Plugins → Input Devices → Windows 输入是否启用
  2. 如果使用 Enhanced Input：
     - 确认 Enhanced Input Plugin 已启用
     - 确认 InputAction 资产在 Content 中存在
     - 确认 Input Mapping Context 正确配置并加载
```

### 错误6：存档功能在打包版中失效

```
错误现象：
  - 编辑器里存档正常，打包后存不了档

原因分析：
  - 存档路径在打包后发生了变化
  - 打包版缺少文件写入权限

解决方法：
  1. 打包后存档路径会自动变为：
     %LOCALAPPDATA%/DungeonCrawler/Saved/SaveGames/
  2. 确保 UPROPERTY 标记的存档数据类完整
  3. 如果是Shipping配置，确保没有把SaveGame相关代码用 #if WITH_EDITOR 条件编译排除
```

### 错误7：nvwgf2umx.dll 崩溃

```
错误现象：
  - 游戏启动后黑屏、崩溃
  - 崩溃日志显示 nvwgf2umx.dll

原因分析：
  - NVIDIA 显卡驱动问题
  - DX12 兼容性问题

解决方法：
  1. 更新显卡驱动到最新版本
  2. 在 Project Settings → Platforms → Windows → Targeted RHIs
     中尝试添加 DX11 作为备用渲染器
  3. 设置 Default RHI 为 DX12，备用为 DX11
```

## 5.5 版本管理和更新策略

### 5.5.1 版本号规划

使用语义化版本号（Semantic Versioning）：`主版本.次版本.修订号`

```
v0.1.0  - 内部测试版（Alpha版，核心功能完成）
v0.5.0  - 封闭测试版（Beta版，所有功能完成，有Bug）
v0.9.0  - 公开测试版（Early Access，修了大部分Bug）
v1.0.0  - 正式发布版（首个稳定版本）
v1.1.0  - 内容更新（新技能、新敌人、新楼层）
v1.1.1  - 平衡性调整 + Bug修复
```

### 5.5.2 更新机制

```cpp
// ── 简单的版本检查和更新机制 ──
// 在 GameInstance 初始化时检查存档版本

void UDungeonGameInstance::CheckAndMigrateSaveData()
{
    // 加载旧的存档
    UDungeonSaveGame* OldSave = Cast<UDungeonSaveGame>(
        UGameplayStatics::LoadGameFromSlot(CurrentSaveSlot, 0)
    );

    if (!OldSave) return;

    // 检查存档版本
    if (OldSave->SaveGameVersion < CurrentGameVersion)
    {
        UE_LOG(LogTemp, Warning, TEXT("存档版本过旧（%d），需要进行迁移到版本 %d"),
            OldSave->SaveGameVersion, CurrentGameVersion);

        // 执行版本迁移
        MigrateSaveData(OldSave, OldSave->SaveGameVersion, CurrentGameVersion);

        // 重新保存为新版本
        UGameplayStatics::SaveGameToSlot(OldSave, CurrentSaveSlot, 0);
    }
}

void UDungeonGameInstance::MigrateSaveData(UDungeonSaveGame* Save, int32 FromVersion, int32 ToVersion)
{
    // 示例：v1 → v2 迁移
    // 在v2中增加了 Endurance 属性，旧存档中这个值为0（默认），需要修正
    if (FromVersion < 2 && ToVersion >= 2)
    {
        // 如果耐力为0（旧存档），给它一个合理的默认值
        if (Save->Endurance <= 0.0f)
        {
            Save->Endurance = 10.0f;  // 默认耐力值
        }
    }

    // v2 → v3 迁移
    // 在v3中修改了技能数据结构，需要转换旧格式
    // ...
}
```

## 5.6 商店上架简介

### 5.6.1 Steam上架流程概览

```
步骤1：准备工作
  - 支付 Steam Direct 费用（$100）
  - 准备游戏宣传物料（宣传图、Logo、截图、预告片）
  - 准备商店页面文本（游戏描述、系统需求、更新公告）

步骤2：创建商店页面
  - 登录 https://partner.steamgames.com/
  - 创建应用 → 填写基本信息
  - 上传所有物料
  - 设置标签（如：动作、角色扮演、地牢探索、第三人称）
  - 提交审核（Steam审核通常1-2天）

步骤3：集成 Steamworks SDK
  - 下载 Steamworks SDK
  - 在项目中启用 OnlineSubsystemSteam 插件
  - 配置 steam_appid.txt
  - 实现成就系统（可选）
  - 实现云存档（使用 Steam Cloud API）

步骤4：测试
  - 使用 Steamworks 提供的测试工具
  - 测试成就解锁
  - 测试云存档同步
  - 测试 Steam Overlay（Shift+Tab）

步骤5：发布
  - 上传游戏包体（通过 SteamPipe）
  - 设置定价（建议：$9.99~$19.99 对于独立游戏）
  - 设置发布时间
  - 点击"发布"按钮
```

### 5.6.2 Steamworks SDK 集成（最小配置）

```cpp
// ── 在 Build.cs 中添加 OnlineSubsystem 依赖 ──
PublicDependencyModuleNames.AddRange(new string[]
{
    "OnlineSubsystem",
    "OnlineSubsystemSteam",
});

// ── 在 DefaultEngine.ini 中配置 ──
// 在 Config/DefaultEngine.ini 添加：

[/Script/Engine.GameEngine]
+NetDriverDefinitions=(DefName="GameNetDriver",DriverClassName="OnlineSubsystemSteam.SteamNetDriver",DriverClassNameFallback="OnlineSubsystemUtils.IpNetDriver")

[OnlineSubsystemSteam]
bUseSteamNetworking=true
```

### 5.6.3 Epic商店上架概览

```
Epic Games Store 与 Steam 的关键区别：
- EGS 不收取上架费，但审核更严格（人工审核）
- 使用 Epic Online Services (EOS) 而非 Steamworks
- 分成比例：Epic 12%，开发者 88%（比 Steam的30%更友好）

集成 EOS SDK：
1. 下载 Epic Online Services SDK
2. 在项目中启用 EOS Shared Plugin
3. 配置 EOS 参数（Product ID, Sandbox ID, Deployment ID）
4. 实现成就、排行榜、云存档
```

## 5.7 安装程序制作

### 5.7.1 使用WiX Toolset创建安装包

单纯的复制exe和pak不够专业。为玩家创建一个安装程序，可以设置桌面快捷方式、开始菜单、卸载程序。

```xml
<!-- 简化版 WiX 安装配置文件 (Installer.wxs) -->
<?xml version="1.0" encoding="UTF-8"?>
<Wix xmlns="http://schemas.microsoft.com/wix/2006/wi">
  <Product Id="*" Name="地牢探索者" Language="2052" Version="1.0.0.0"
           Manufacturer="YourStudio" UpgradeCode="PUT-GUID-HERE">

    <Package InstallerVersion="200" Compressed="yes"
             InstallScope="perMachine" />

    <!-- 安装界面 -->
    <MediaTemplate EmbedCab="yes" />

    <!-- 安装目录 -->
    <Directory Id="TARGETDIR" Name="SourceDir">
```
