# 16.1 项目规划与架构

> **目标**：在写下任何一行代码之前，先完成游戏的整体规划与架构设计。这一步决定了整个开发过程是井井有条还是一团乱麻。本章将带你从零开始设计一个完整可玩的"地牢探索者"游戏。

---

## 1.1 项目概述

### 1.1.1 这是什么游戏？

我们要制作的是一款 **"地牢探索者"（Dungeon Crawler）**——第三人称动作角色扮演游戏。玩家操控一名冒险者，深入层层地牢，击败怪物、收集装备、提升等级，最终击败守护Boss，通关整个地牢。

**游戏类型**：第三人称动作RPG（Action RPG）
**视角**：第三人称追尾视角（类似《黑暗之魂》《原神》）
**战斗方式**：实时动作战斗，支持连击、格挡、闪避、技能释放
**目标平台**：Windows PC（64位）

### 1.1.2 为什么选择这个项目？

| 原因     | 说明                                                 |
| -------- | ---------------------------------------------------- |
| 覆盖面广 | 涵盖角色控制、战斗、AI、UI、存档等几乎所有UE核心系统 |
| 难度可控 | 从简单到复杂，可以逐步迭代                           |
| 成就感强 | 做完是一个真正能玩的完整游戏                         |
| 面试加分 | 动作RPG是游戏行业最常见类型，作品集含金量高          |

## 1.2 核心玩法设计

### 1.2.1 游戏循环

```
┌──────────────────────────────────────────────────┐
│                  游戏主循环                         │
│                                                    │
│  进入地牢 → 探索房间 → 遭遇怪物 → 战斗击败          │
│     ↑                                    ↓         │
│     └──── 升级/换装 ← 获得经验和装备 ←──┘          │
│                        ↓                           │
│              收集钥匙 → 开启Boss房间                │
│                        ↓                           │
│              击败Boss → 进入下一层                   │
│                        ↓                           │
│              （重复直到通关或死亡）                   │
└──────────────────────────────────────────────────┘
```

### 1.2.2 详细玩法清单

**探索**

- 地牢由多个房间组成，房间之间由走廊连接
- 部分房间上锁，需要找到对应钥匙
- 房间内可以找到宝箱、药水、装备等道具
- 小地图显示已探索区域

**战斗**

- 鼠标左键：轻攻击（可三连击）
- 鼠标右键：重攻击
- 鼠标左键 + 鼠标右键组合：上挑终结技
- 右键按住：格挡（时机精准触发完美格挡）
- Shift：闪避（带无敌帧）
- Q / E / R / F：四个技能

**成长**

- 击败怪物获得经验值
- 升级获得属性点，自由分配
- 击败怪物掉落装备
- 装备可提升攻击、防御等属性

**Boss战**

- 每层最后有Boss房间（需要Boss钥匙）
- Boss拥有独特的攻击模式和阶段变化
- 击败Boss获得大量经验和稀有装备
- 击败Boss后解锁通往下一层的楼梯

### 1.3.3 数据流设计

**游戏状态管理**

游戏状态由 `ADungeonGameState` 集中管理，所有需要跨客户端同步的状态都放在这里（如果是多人游戏）。单人游戏中，`ADungeonGameState` 仍然有用——它存储当前楼层的全局状态：

```cpp
// 游戏状态枚举 — 控制游戏的整体流程
UENUM(BlueprintType)
enum class EGamePhase : uint8
{
    MainMenu     UMETA(DisplayName = "主菜单"),     // 主菜单界面
    Exploring    UMETA(DisplayName = "探索中"),     // 玩家正在探索
    Combat       UMETA(DisplayName = "战斗中"),     // 玩家处于战斗状态
    BossFight    UMETA(DisplayName = "Boss战"),     // Boss战阶段
    Paused       UMETA(DisplayName = "暂停"),       // 游戏暂停
    GameOver     UMETA(DisplayName = "游戏结束"),   // 玩家死亡
    Victory      UMETA(DisplayName = "通关")        // Boss被击败，通关
};
```

**事件系统**

使用UE原生的 **多播委托（Multicast Delegate）** 进行模块间通信。这种方法解耦了各个系统——战斗系统不需要知道UI系统的存在，只需广播事件，UI系统自己监听即可。

```cpp
// 在 DungeonGameMode 中定义全局事件
DECLARE_MULTICAST_DELEGATE_OneParam(FOnEnemyKilled, ADungeonEnemy*);
DECLARE_MULTICAST_DELEGATE_OneParam(FOnItemPickedUp, UDungeonItemBase*);
DECLARE_MULTICAST_DELEGATE_OneParam(FOnPlayerLevelUp, int32);
DECLARE_MULTICAST_DELEGATE(FOnBossDefeated);
DECLARE_MULTICAST_DELEGATE_OneParam(FOnGamePhaseChanged, EGamePhase);
```

**数据流向图**：

```
输入设备
   │
   ▼
PlayerController ──→ DungeonCharacter ──→ 执行动作
   │                      │
   │                      ├──→ 伤害系统（计算伤害）
   │                      │       │
   │                      │       └──→ Enemy（扣血/死亡）
   │                      │              │
   │                      │              └──→ 事件广播
   │                      │                    │
   │                      │         ┌─────────┘
   │                      │         ▼
   │                      └──→ GameMode ←── 事件聚合
   │                                 │
   │                    ┌────────────┼────────────┐
   │                    ▼            ▼            ▼
   │                GameState     UI更新      存档系统
   │                （状态记录）  （界面刷新） （数据持久化）
   │
   └──→ UI输入（暂停、菜单、背包等）
```

### 1.4.2 ADungeonCharacter（玩家角色）详细设计

```cpp
// ============================================================================
// ADungeonCharacter — 玩家操控的角色
// ============================================================================
// 继承自 ACharacter，自带 CapsuleComponent 和 CharacterMovementComponent
// 核心职责：移动、战斗、属性管理、装备管理、状态管理
// ============================================================================

UCLASS()
class ADungeonCharacter : public ACharacter
{

public:

    ADungeonCharacter();

    // 当前生命值（0~最大生命值）
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float CurrentHealth;

    // 最大生命值
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float MaxHealth;

    // 当前法力值
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float CurrentMana;

    // 最大法力值
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float MaxMana;

    // 当前体力值
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float CurrentStamina;

    // 最大体力值
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float MaxStamina;

    // 基础攻击力（不含装备加成）
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float BaseAttack;

    // 基础防御力（不含装备加成）
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float BaseDefense;

    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    float SpeedMultiplier;

    // 当前等级（1级起步）

    int32 CharacterLevel;

    int32 CurrentExperience;

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    int32 ExperienceToNextLevel;

    // 可用属性点（升级获得，手动分配）
    UPROPERTY(EditAnywhere, BlueprintReadWrite, Replicated)
    int32 AvailableAttributePoints;

    // ── 装备系统 ──
    // 装备槽 — 武器
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    class UDungeonEquipment* WeaponSlot;

    // 装备槽 — 头盔
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    class UDungeonEquipment* HelmetSlot;

    // 装备槽 — 铠甲
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    class UDungeonEquipment* ArmorSlot;

    // 装备槽 — 鞋子
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    class UDungeonEquipment* BootsSlot;

    // 装备槽 — 戒指1
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    class UDungeonEquipment* RingSlot1;

    // 装备槽 — 戒指2
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    class UDungeonEquipment* RingSlot2;

    // ── 背包系统 ──
    // 物品背包（最大容量）
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    TArray<class UDungeonItemBase*> Inventory;

    // 背包最大容量
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    int32 MaxInventorySize;

    // ── 战斗系统 ──
    // 当前是否在攻击动画中
    UPROPERTY(BlueprintReadOnly)
    bool bIsAttacking;

    // 当前是否在格挡中
    UPROPERTY(BlueprintReadOnly)
    bool bIsBlocking;

    // 是否处于无敌状态（闪避/受击后短暂无敌）
    UPROPERTY(BlueprintReadOnly)
    bool bIsInvincible;

    // ── 状态机 ──
    // 当前角色状态
    UPROPERTY(BlueprintReadOnly)
    ECharacterState CurrentState;

    // 状态枚举
    UENUM(BlueprintType)
    enum class ECharacterState : uint8
    {
        Idle      UMETA(DisplayName = "待机"),

        Attacking UMETA(DisplayName = "攻击中"),

        Dodging   UMETA(DisplayName = "闪避中"),

    // ── 方法 ──

    void HeavyAttack();                 // 重攻击
    void StartBlocking();               // 开始格挡

    void UseSkill(int32 SkillIndex);    // 使用技能

    // 属性相关
    float GetTotalAttack() const;       // 计算总攻击力（基础+装备）
    float GetTotalDefense() const;      // 计算总防御力（基础+装备）
    void TakeDamage(float Damage);      // 受到伤害
    void Heal(float Amount);            // 恢复生命
    void AddExperience(int32 Amount);   // 获得经验
    void LevelUp();                     // 升级处理
    void AllocateAttributePoint(EAttributeType Type); // 分配属性点

    // 装备相关
    void EquipItem(UDungeonEquipment* Equipment);  // 装备物品
    void UnequipItem(EEquipmentSlot Slot);         // 卸下装备
    void AddToInventory(UDungeonItemBase* Item);   // 添加到背包
    void RemoveFromInventory(UDungeonItemBase* Item); // 从背包移除

    // 状态相关
    void SetState(ECharacterState NewState);       // 切换状态
    void Die();                                     // 死亡处理
};
```

### 1.4.3 ADungeonEnemy（敌人基类）详细设计

```cpp
// ============================================================================
// ADungeonEnemy — 敌人基类
// ============================================================================
// 所有敌人的基类，包含AI所需的基础属性和行为
// 不同类型的敌人通过蓝图或子类实现
// ============================================================================

UCLASS()
class ADungeonEnemy : public ACharacter
{
    GENERATED_BODY()

public:
    ADungeonEnemy();

    // ── 基础属性 ──
    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    float MaxHealth;           // 最大生命值

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    float CurrentHealth;       // 当前生命值

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    float AttackPower;         // 攻击力

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    float DefensePower;        // 防御力

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    float MoveSpeed;           // 移动速度

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    int32 ExperienceReward;    // 击败后给予的经验值

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    float DetectionRange;      // 侦查范围（发现玩家的距离）

    UPROPERTY(EditAnywhere, BlueprintReadWrite)
    float AttackRange;         // 攻击范围
```

## 1.5 开发顺序建议

### 1.5.1 分阶段开发路线图

正确的开发顺序极其重要。不要跳跃——每一阶段都是下一阶段的基础。

```
第一阶段：地基搭建（预计2-3天）
├── 步骤1: 创建C++项目，设置目录结构
├── 步骤2: 实现 ADungeonCharacter 基础移动（WASD + 鼠标视角）
├── 步骤3: 实现第三人称摄像机（弹簧臂 + 缩放）
├── 步骤4: 实现 ADungeonGameMode 基础框架
└── 步骤5: 搭建测试关卡（白色盒子即可）

第二阶段：战斗系统（预计3-4天）
├── 步骤6: 实现 ADungeonCharacter 轻攻击/重攻击
├── 步骤7: 实现连击系统（攻击动画 + 输入缓冲）
├── 步骤8: 实现伤害系统（伤害计算、暴击判定）
├── 步骤9: 实现格挡和闪避
├── 步骤10: 实现 ADungeonEnemy 基础类
└── 步骤11: 敌人受击反馈（动画 + 硬直）

第三阶段：成长系统（预计2-3天）
├── 步骤12: 实现属性系统（生命/法力/体力/攻/防/速）
├── 步骤13: 实现升级系统（经验值 + 属性分配）
├── 步骤14: 实现装备系统（装备槽 + 属性加成）
├── 步骤15: 实现物品系统（UDataAsset + 拾取）
└── 步骤16: 实现背包UI

第四阶段：AI与技能（预计3-4天）
├── 步骤17: 实现敌人AI（行为树：巡逻→侦查→追击→攻击）
├── 步骤18: 实现技能系统（4技能槽 + 冷却管理）
├── 步骤19: 实现Boss AI（多阶段AI）
└── 步骤20: 实现掉落系统

第五阶段：存档与数据（预计2天）
├── 步骤21: 实现 UDungeonSaveGame（序列化）
├── 步骤22: 实现 DataTable（怪物/装备/技能数据）
├── 步骤23: 实现 UDungeonGameInstance（跨关卡数据）
└── 步骤24: 实现存档管理UI

第六阶段：UI与关卡（预计2-3天）
├── 步骤25: HUD（血条/法力条/体力条/技能栏）
├── 步骤26: 主菜单和暂停菜单
├── 步骤27: 角色面板UI
├── 步骤28: 搭建多个楼层关卡
└── 步骤29: 关卡切换过渡

第七阶段：打磨发布（预计2天）
├── 步骤30: 性能优化（LOD、遮挡剔除、光照优化）
├── 步骤31: 音效和特效整合
├── 步骤32: Bug修复和平衡性调整
├── 步骤33: 打包Windows版本
└── 步骤34: 测试打包后的exe
```

### 1.5.2 为什么是这个顺序？

| 阶段           | 理由                             |
| -------------- | -------------------------------- |
| **先做移动**   | 角色必须能动，否则什么也测不了   |
| **再做战斗**   | 战斗是核心玩法，必须尽早验证手感 |
| **然后做成长** | 有了战斗，才能测试经验获取和升级 |
| **之后做AI**   | 有了角色和战斗，才能让AI与之交互 |
| **最后做存档** | 存档依赖前面的数据已经稳定       |
| **UI贯穿始终** | UI可以先用临时蓝图，最后再打磨   |

**黄金法则**：**先让游戏能跑起来，再逐步添加功能**。每完成一个步骤立即测试，不要等到全部写完了再测试。

✅ **正确做法**：实现移动 → 立即测试移动 → 实现攻击 → 立即测试攻击 → ...
❌ **错误做法**：花一周写完所有代码 → 第一次编译 → 100+个错误 → 心态崩溃

## 1.7 模块依赖配置

### 1.7.1 Build.cs 文件

```csharp
// ============================================================================
// DungeonCrawler.Build.cs — 模块构建配置
// ============================================================================
// 这个文件告诉UE的构建系统我们需要哪些模块。
// 缺少依赖会导致编译错误（找不到头文件），必须在此声明。
// ============================================================================

using UnrealBuildTool;

public class DungeonCrawler : ModuleRules
{
    public DungeonCrawler(ReadOnlyTargetRules Target) : base(Target)
    {
        // 使用显式PCH，加快编译速度
        PCHUsage = PCHUsageMode.UseExplicitOrSharedPCHs;

        // ── 公共依赖模块（头文件中用到的模块）──
        PublicDependencyModuleNames.AddRange(new string[]
        {
            "Core",               // UE核心（FString, TArray等基础类型）
            "CoreUObject",        // UObject系统（UCLASS, UPROPERTY等反射宏）
            "Engine",             // 引擎基础（AActor, UActorComponent等）
            "InputCore",          // 输入系统（按键绑定、Action映射）
            "EnhancedInput",      // UE5增强输入系统
        });

        // ── 私有依赖模块（.cpp文件中用到的模块，不暴露给外部）──
        PrivateDependencyModuleNames.AddRange(new string[]
        {
            "Slate",              // UI框架底层
            "SlateCore",          // UI框架底层核心
            "UMG",                // UMG UI系统（蓝图UI）
            "AIModule",           // AI系统（行为树、黑板、AI感知）
            "GameplayTasks",      // AI任务系统（行为树依赖）
            "NavigationSystem",   // 导航系统（AI寻路）
            "Niagara",            // Niagara特效系统
            "LevelSequence",      // 关卡序列（过场动画）
        });
    }
}
```

✅ **正确做法**：在Build.cs中声明所有需要的模块依赖
❌ **错误做法**：编译报错找不到头文件，然后忽略，或者用奇怪的include路径绕过
