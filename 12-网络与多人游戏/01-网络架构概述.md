## 为什么需要网络架构

多人游戏的核心挑战在于：**让多个玩家看到同一个游戏世界，并且交互结果一致**。

想象一个场景：玩家A开枪打中了玩家B。如果每个玩家的电脑上分别计算伤害，那么：

- 玩家A的电脑可能认为"我打中了"。
- 玩家B的电脑可能认为"我没被打中"。
- 这就产生了**不一致**，游戏无法正常进行。

解决这个问题的标准方案就是**客户端-服务器模型（Client-Server Model）**。UE5 深度内置了这套模型，你需要理解它的工作原理才能在 UE5 中写出正确的多人游戏代码。

## UE 的客户端-服务器模型

### 核心概念

```
┌─────────────────────────────────────────────────────────┐
│                      服务器 (Server)                     │
│  ┌─────────────────────────────────────────────────────┐│
│  │  拥有游戏世界的"权威版本"                              ││
│  │  - 所有重要逻辑在服务器上执行                          ││
│  │  - 服务器决定发生了什么                               ││
│  │  - 服务器把结果"同步"给所有客户端                      ││
│  └─────────────────────────────────────────────────────┘│
└────────────┬──────────────────────────────┬──────────────┘
             │                              │
    ┌────────▼────────┐            ┌────────▼────────┐
    │   客户端 A       │            │   客户端 B       │
    │   (玩家1的电脑)   │            │   (玩家2的电脑)   │
    │                  │            │                  │
    │  - 拥有游戏世界的 │            │  - 拥有游戏世界的 │
    │    "近似版本"     │            │    "近似版本"     │
    │  - 显示给玩家看   │            │  - 显示给玩家看   │
    │  - 发送输入给服务器│           │  - 发送输入给服务器│
    └──────────────────┘            └──────────────────┘
```

**一句话总结**：服务器是"法官"，客户端是"观众"。法官说了算，观众只能看和请求（比如请求开枪）。

### 关键规则（必须牢记）

| 规则               | 说明                                                          |
| ------------------ | ------------------------------------------------------------- |
| 服务器拥有权威     | 所有的伤害计算、生命值变更、胜负判定都在服务器执行            |
| 客户端只是"表现层" | 客户端负责渲染、播放声音、显示UI                              |
| 客户端可以"请求"   | 客户端通过 RPC 请求服务器执行某操作（如开火），但服务器会验证 |
| 服务器决定同步什么 | 服务器决定哪些数据同步给哪些客户端                            |

## 服务器的三种角色

在 UE5 中，一个游戏实例可以同时担任不同的"网络角色"。理解这三种角色是理解 UE 网络的基础。

### 1. Dedicated Server（专用服务器）

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  客户端 A     │────▶│ Dedicated    │◀────│  客户端 B     │
│  (有画面)     │     │  Server      │     │  (有画面)     │
│              │     │  (无画面!)    │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
```

- **Dedicated Server 没有渲染窗口**。它是一个纯粹的后台进程，只处理游戏逻辑。
- 这是**线上游戏的标准做法**（如 PUBG、Fortnite 的服务器）。
- 优点：性能最好、最安全、没有玩家能直接在服务器上作弊。
- 启动方式：`UE5Editor.exe YourProject.uproject /Game/Maps/BattleMap -server -log`

### 2. Listen Server（监听服务器）

```
┌──────────────────────┐
│   客户端 A (玩家1)    │
│   ┌──────────────┐   │
│   │  Listen      │   │     ┌──────────────┐
│   │  Server      │◀──│────▶│  客户端 B     │
│   │  (同时是服务器)│   │     │  (只有客户端)  │
│   └──────────────┘   │     └──────────────┘
│   有画面 + 有权威     │
└──────────────────────┘
```

- 一个玩家既是客户端（有画面、能操作），又是服务器（有权威）。
- 常用于**局域网联机**或**小规模测试**。
- 缺点：如果这个玩家退出游戏，整个服务器就关掉了。

### 3. Client（客户端）

- 就是普通的玩家端，只有渲染和输入，没有权威。
- 客户端连接到服务器（无论是 Dedicated 还是 Listen Server）。

### 三种角色的关系总结

| 角色             | 有渲染画面 | 有权威 | 典型用途           |
| ---------------- | :--------: | :----: | ------------------ |
| Dedicated Server |     ❌     |   ✅   | 线上游戏的生产环境 |
| Listen Server    |     ✅     |   ✅   | 局域网联机、测试   |
| Client           |     ✅     |   ❌   | 普通玩家           |

## 网络模式详解

UE5 使用 `ENetMode` 枚举来表示当前实例的网络模式。你可以在任何 Actor 上通过 `GetNetMode()` 获取。

### 四种网络模式

```cpp
// 引擎源码中的定义（你不需要写这些，这是帮助你理解的）
UENUM()
enum ENetMode
{
    // 单机模式：没有网络连接，一个人玩
    NM_Standalone,

    // 专用服务器模式：只有逻辑，没有渲染
    NM_DedicatedServer,

    // 监听服务器模式：既是服务器又是客户端（有渲染画面）
    NM_ListenServer,

    // 客户端模式：连接到远程服务器的普通玩家
    NM_Client,

    // 最大枚举值（引擎内部使用，你不需要关心）
    NM_MAX,
};
```

### 如何判断当前网络模式

```cpp
// 在任何 Actor 的成员函数中
void AMyActor::CheckNetMode()
{
    // GetNetMode() 返回当前的网络模式
    ENetMode NetMode = GetNetMode();

    // 判断是否在服务器上（不管是 Dedicated 还是 Listen Server）
    if (GetNetMode() == NM_DedicatedServer || GetNetMode() == NM_ListenServer)
    {
        // 当前实例是服务器，拥有权威
        UE_LOG(LogTemp, Log, TEXT("我是服务器!"));
    }
    else if (GetNetMode() == NM_Client)
    {
        // 当前实例是客户端
        UE_LOG(LogTemp, Log, TEXT("我是客户端!"));
    }
    else // NM_Standalone
    {
        // 单机模式（编辑器中直接点击 Play 就是这个模式）
        UE_LOG(LogTemp, Log, TEXT("我是单机模式!"));
    }
}
```

### 网络模式的一大常见困惑

很多新手会搞混"编辑器里点Play"和"真正的联网运行"。

```cpp
// ✅ 正确理解
// 在编辑器中直接点 Play → NM_Standalone（单机模式）
// 在编辑器中选 "Net Mode: Play As Listen Server" → NM_ListenServer
// 在编辑器中选 "Net Mode: Play As Client" → NM_Client

// ❌ 常见错误
// 错误地以为编辑器里点Play就是"客户端"——不是的，单机模式一切都在本地执行
```

## 网络角色（NetRole）

网络角色描述了一个 Actor 在某个具体客户端或服务器上"是什么身份"。这是 UE 网络编程中**最重要的概念之一**。

### 三种网络角色

```cpp
// 引擎源码的定义（帮助理解的）
UENUM()
enum ENetRole
{
    // 无效角色（内部使用，你不会直接用到）
    ROLE_None,

    // 权威端：意味着这个实例拥有对这个 Actor 的"最终决定权"
    // 服务器上的所有 Actor 都是 ROLE_Authority
    ROLE_Authority,

    // 自主代理：客户端的"我自己的 Pawn"
    // 只有本地玩家控制的 Pawn 才是这个角色
    ROLE_AutonomousProxy,

    // 模拟代理：客户端上"别人的 Pawn"或"其他 Actor"
    // 客户端上不属于自己的联网 Actor 都是这个角色
    ROLE_SimulatedProxy,

    // 最大的枚举值（引擎内部使用）
    ROLE_MAX,
};
```

### 直观理解：三个玩家的视角

假设有三个玩家：小明、小红、小刚。小明的电脑是 Listen Server。

**在小明的电脑上（服务器视角）**：

| Actor       | 网络角色                                    | 解释                                               |
| ----------- | ------------------------------------------- | -------------------------------------------------- |
| 小明的 Pawn | `ROLE_Authority` + `ROLE_AutonomousProxy`\* | 小明是服务器，所以有权威；同时小明自己控制这个Pawn |
| 小红的 Pawn | `ROLE_Authority`                            | 服务器对所有人的 Pawn 都有权威                     |
| 小刚的 Pawn | `ROLE_Authority`                            | 同上                                               |

> \*注：在 Listen Server 上，本地玩家的 Pawn 既是 Authority 又是 AutonomousProxy。这是 Listen Server 的特殊情况。

**在小红的电脑上（客户端视角）**：

| Actor       | 网络角色               | 解释                                 |
| ----------- | ---------------------- | ------------------------------------ |
| 小红的 Pawn | `ROLE_AutonomousProxy` | "这是我自己的角色"                   |
| 小明的 Pawn | `ROLE_SimulatedProxy`  | "这是别人的角色，服务器告诉我它在哪" |
| 小刚的 Pawn | `ROLE_SimulatedProxy`  | "这也是别人的角色"                   |

### 一张图理解网络角色

```
       ┌────────── 服务器 ──────────┐
       │                             │
       │   所有 Actor → Authority    │
       │                             │
       └────────────┬────────────────┘
                    │ 网络同步
       ┌────────────┼────────────────┐
       │            │                │
  ┌────▼─────┐ ┌────▼─────┐  ┌─────▼─────┐
  │ 客户端 A  │ │ 客户端 B  │  │ 客户端 C  │
  │           │ │           │  │           │
  │ 自己的Pawn│ │ 自己的Pawn│  │ 自己的Pawn│
  │  → Auto-  │ │  → Auto-  │  │  → Auto-  │
  │  nomous   │ │  nomous   │  │  nomous   │
  │  Proxy    │ │  Proxy    │  │  Proxy    │
  │           │ │           │  │           │
  │ 别人的Pawn│ │ 别人的Pawn│  │ 别人的Pawn│
  │  → Simu-  │ │  → Simu-  │ │  → Simu-  │
  │  lated    │ │  lated    │ │  lated    │
  │  Proxy    │ │  Proxy    │ │  Proxy    │
  └───────────┘ └───────────┘ └───────────┘
```

## 关键 API：判断"我是谁"

UE5 提供了几个核心函数来帮助你判断当前 Actor 的网络身份。

### 1. GetLocalRole() — 我在本地是什么角色

```cpp
// 返回这个 Actor 在当前机器上的网络角色
ENetRole LocalRole = GetLocalRole();

if (LocalRole == ROLE_Authority)
{
    // 这段代码只在服务器上执行
    // 因为只有服务器上的 Actor 的 LocalRole 才是 Authority
}

if (LocalRole == ROLE_AutonomousProxy)
{
    // 这段代码只在"拥有这个 Actor 的客户端"上执行
    // 典型场景：本地玩家自己的 Pawn
}

if (LocalRole == ROLE_SimulatedProxy)
{
    // 这段代码在"不拥有这个 Actor 的客户端"上执行
    // 典型场景：在别人电脑上看到的其他玩家的 Pawn
}
```

### 2. GetRemoteRole() — 远程端是什么角色

```cpp
// 返回这个 Actor 在"连接的另一端"的网络角色
// 对服务器而言，RemoteRole 是客户端上的角色
// 对客户端而言，RemoteRole 是服务器上的角色（就是 Authority）
ENetRole RemoteRole = GetRemoteRole();
```

### 3. HasAuthority() — 最重要的判断函数

```cpp
// 判断当前 Actor 是否运行在"有权威"的机器上
// 这是你写网络逻辑时最常用的函数！
// 本质上等价于：GetLocalRole() == ROLE_Authority
bool bIsServer = HasAuthority();

if (HasAuthority())
{
    // ✅ 安全的地方：在这里修改血量、计算伤害、判断胜负
    ApplyDamage(50.0f);
}
else
{
    // ❌ 不能在客户端直接修改血量！服务器不会认可
    // 如果客户端改了，会被服务器的值覆盖掉
}
```

### 4. IsLocallyControlled() — 是我在控制吗

```cpp
// 判断这个 Pawn/Character 是否是"本地玩家"在控制
// 对于 AutonomousProxy，返回 true
// 对于 SimulatedProxy 和 Authority（远程控制的），返回 false
if (IsLocallyControlled())
{
    // 这是我的角色！可以处理本地输入、播放本地音效等
    // 注意：这不代表有权威！客户端上的 AutonomousProxy 也是 IsLocallyControlled
}
```

### 5. 完整的使用示例

```cpp
// 在 AMyCharacter 的成员函数中
void AMyCharacter::DoSomething()
{
    // ===== 判断一：我是在服务器上吗？ =====
    if (HasAuthority())
    {
        // 服务器：执行真正的逻辑（修改血量、计算伤害等）
        UE_LOG(LogTemp, Log, TEXT("服务器：执行权威逻辑"));
    }
    else
    {
        // 客户端：不能执行重要逻辑
        UE_LOG(LogTemp, Log, TEXT("客户端：不能执行权威逻辑"));
    }

    // ===== 判断二：这个角色是我在操控吗？ =====
    if (IsLocallyControlled())
    {
        // 是本地玩家操控的角色
        // 可以在这里播放第一人称音效、更新本地UI等
        UE_LOG(LogTemp, Log, TEXT("这是我自己操控的角色"));
    }
    else
    {
        // 不是本地玩家操控的角色（别人的角色，或者服务器上的角色）
        UE_LOG(LogTemp, Log, TEXT("这不是我操控的角色"));
    }

    // ===== 判断三：综合判断（常见组合） =====
    if (HasAuthority() && IsLocallyControlled())
    {
        // 这种情况只出现在 Listen Server 上：
        // 你是服务器，同时你在操控这个角色
    }
    else if (!HasAuthority() && IsLocallyControlled())
    {
        // 最常见的客户端情况：
        // 你是客户端，你在操控自己的角色（AutonomousProxy）
    }
    else if (!HasAuthority() && !IsLocallyControlled())
    {
        // 客户端上别人的角色（SimulatedProxy）
    }
}
```

### 常见陷阱与对比

```cpp
// ❌ 错误：在客户端直接修改血量
void AMyCharacter::TakeDamageFromClient(float Damage)
{
    Health -= Damage;  // 客户端改了，但服务器不知道，会被覆盖！
}

// ✅ 正确：只在服务器修改血量
void AMyCharacter::TakeDamage(float Damage)
{
    if (HasAuthority())  // 确保只有服务器执行
    {
        Health -= Damage;
        UE_LOG(LogTemp, Log, TEXT("服务器：血量从 %f 变为 %f"), Health + Damage, Health);
    }
}

// ❌ 错误：使用 GetLocalRole() 不等于 HasAuthority() 的判断方式
if (GetLocalRole() == ROLE_AutonomousProxy)
{
    // 你以为这就是"客户端上的我自己的角色"——没错
    // 但如果你在 Listen Server 上，本地玩家的 LocalRole 也是 Authority！
    // 所以这个判断在 Listen Server 上会漏掉本地玩家
}

// ✅ 正确：如果需要判断"是否是本地玩家控制"
if (IsLocallyControlled())
{
    // 这个判断在所有网络模式下都正确
    // - 客户端上的 AutonomousProxy → true
    // - Listen Server 上的本地玩家 → true
    // - 其他人的角色 → false
}
```

## 网络连接流程

理解从"启动游戏"到"玩家可以操作"的完整流程，这能帮助你理解 GameMode 和 GameState 的职责。

### 完整流程（以 Listen Server + 1个客户端为例）

```
步骤1：Listen Server 启动
┌─────────────────────────────┐
│  玩家A 启动游戏              │
│  选择"作为 Listen Server"    │
│  加载关卡地图               │
│  → GameMode 初始化           │
│  → GameState 初始化          │
│  → 等待其他玩家连接...       │
└─────────────────────────────┘

步骤2：客户端请求连接
┌──────────────┐          ┌─────────────────────────────┐
│  玩家B 启动   │          │  Listen Server               │
│  输入服务器IP │ ──────▶  │  收到连接请求                 │
│  请求连接     │          │                              │
└──────────────┘          └─────────────────────────────┘

步骤3：服务器验证与接受
┌─────────────────────────────────────────────────────────┐
│  服务器执行：                                             │
│  1. GameMode::PreLogin() → 验证密码/版本/人数上限          │
│     - 如果验证失败 → 返回错误信息，拒绝连接                 │
│     - 如果验证成功 → 继续下一步                            │
│  2. GameMode::Login() → 创建 PlayerController             │
│  3. GameMode::PostLogin() → 玩家成功加入!                   │
│     - 这里可以给玩家初始装备、显示欢迎信息等                  │
└─────────────────────────────────────────────────────────┘

步骤4：生成玩家 Pawn
┌─────────────────────────────────────────────────────────┐
│  服务器执行：                                             │
│  GameMode::RestartPlayer() / HandleStartingNewPlayer()   │
│  → 在场景中创建一个 Pawn（Character）                      │
│  → PlayerController 拥有（Possess）这个 Pawn              │
│  → 服务器把这个 Pawn 的网络角色设为 ROLE_Authority         │
│  → 服务器将这个 Pawn 复制（Replicate）到所有客户端          │
└─────────────────────────────────────────────────────────┘

步骤5：客户端收到 Pawn
┌──────────────┐          ┌──────────────┐
│  客户端B      │          │  客户端A      │
│              │          │              │
│  收到新Pawn  │          │  收到新Pawn  │
│  - 如果是我   │          │  - 这是别人   │
│    的Pawn:   │          │    的Pawn:   │
│    Auto-     │          │    Simu-     │
│    nomous    │          │    lated     │
│    Proxy     │          │    Proxy     │
└──────────────┘          └──────────────┘
```

)
{
    // 检查版本号
    // 检查服务器是否已满
    // 检查是否在黑名单中
    // 如果拒绝连接，将错误信息写入 ErrorMessage
    // ErrorMessage = TEXT("服务器已满，请稍后再试");
}

// 第二步：创建 PlayerController
// 这是引擎自动处理的，你通常不需要覆写 Login

## GameMode 和 GameState 的网络分布

这是一个非常重要的区别，很多新手在这里踩坑。

### GameMode — 只在服务器上存在

```cpp
UCLASS()
class MYGAME_API AMyGameMode : public AGameModeBase
{
    GENERATED_BODY()

public:
    // ⚠️ 这个变量只在服务器上存在！
    // ⚠️ 客户端上 GetGameMode() 返回 nullptr！（如果是远程客户端）
    int32 ServerOnlyData;

    // ✅ 正确的使用方式：只在服务器端访问 GameMode
    void SomeServerFunction()
    {
        if (HasAuthority())
        {
            AMyGameMode* GM = Cast<AMyGameMode>(GetWorld()->GetAuthGameMode());
            if (GM)
            {
                // 在服务器上安全访问
                GM->ServerOnlyData = 42;
            }
        }
    }

    // ❌ 错误的使用方式：在客户端访问 GameMode
    void SomeClientFunction()
    {
        // 客户端上 GetGameMode() 可能返回 nullptr！
        AMyGameMode* GM = Cast<AMyGameMode>(GetWorld()->GetAuthGameMode());
        // GM 在客户端上很可能是 nullptr！
    }
};
```

### GameState — 在所有端上存在

```cpp
UCLASS()
class MYGAME_API AMyGameState : public AGameStateBase
{
    GENERATED_BODY()

public:
    // ✅ 这个变量在所有端上都存在！
    // 服务器修改它，引擎自动同步到所有客户端
    UPROPERTY(Replicated)
    int32 TeamAScore;

    UPROPERTY(Replicated)
    int32 TeamBScore;

    UPROPERTY(Replicated)
    float RemainingTime;
};

// ✅ 正确：在服务器上修改 GameState，自动同步到所有客户端
void AMyGameMode::ScorePointForTeamA()
{
    if (HasAuthority())
    {
        AMyGameState* GS = GetGameState<AMyGameState>();
        if (GS)
        {
            GS->TeamAScore++;  // 服务器修改
            // 引擎自动将这个值同步到所有客户端
            // 客户端上的 TeamAScore 会自动更新
        }
    }
}

// ✅ 正确：在客户端读取 GameState（安全！）
void AMyPlayerController::UpdateScoreUI()
{
    // 客户端可以安全读取 GameState
    AMyGameState* GS = GetWorld()->GetGameState<AMyGameState>();
    if (GS)
    {
        int32 ScoreA = GS->TeamAScore;  // 这个值是服务器同步过来的，可靠的
        // 更新UI...
    }
}
```

### GameMode vs GameState 对比总结

| 特性         | GameMode                        | GameState                    |
| ------------ | ------------------------------- | ---------------------------- |
| 服务器上存在 | ✅                              | ✅                           |
| 客户端上存在 | ❌ (远程客户端为 nullptr)       | ✅ (自动复制)                |
| 用途         | 游戏规则、胜负判定、管理玩家    | 所有玩家都需要知道的游戏状态 |
| 典型变量     | 匹配配置、AI 管理               | 队伍分数、剩余时间、击杀数   |
| 访问方式     | `GetWorld()->GetAuthGameMode()` | `GetWorld()->GetGameState()` |

> **记忆口诀**：GameMode 是裁判（只在服务器），GameState 是记分牌（所有人都能看到）。

## 小结与检查清单

在本章中，你学到了：

1. **客户端-服务器模型**：服务器是权威，客户端是表现层。
2. **三种服务器角色**：Dedicated Server（无画面）、Listen Server（有画面+服务器）、Client（纯客户端）。
3. **四种网络模式**：NM_Standalone、NM_DedicatedServer、NM_ListenServer、NM_Client。
4. **三种网络角色**：ROLE_Authority（服务器上的）、ROLE_AutonomousProxy（客户端上自己控制的）、ROLE_SimulatedProxy（客户端上别人控制的）。
5. **关键判断函数**：`HasAuthority()`（是否服务器）、`IsLocallyControlled()`（是否本地玩家）。
6. **连接流程**：PreLogin → Login → PostLogin → RestartPlayer。
7. **GameMode 只在服务器存在，GameState 在所有端存在。**
